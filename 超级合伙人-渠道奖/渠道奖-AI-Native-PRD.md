# 渠道奖 AI-Native PRD

> 文档类型：产品需求文档 / AI-Native PRD  
> 需求名称：渠道奖  
> 所属系统：分销体系 / 连锁总部后台 / 门店后台  
> 需求性质：基于现有分销数据新增奖项设置与统计能力  
> 是否涉及系统自动发放：否  
> 是否影响其他奖项：否  
> 当前版本：V1.2  
> 最近更新时间：2026-06-12  
> 最近更新人：Codex

## 1. 项目上下文包

### 1.1 业务背景

当前分销体系中已经存在合伙人关系链路、推广业绩、门店归属、连锁总部与门店后台等基础能力。渠道奖是在这些现有能力基础上新增的一种奖项设置与统计能力。

渠道奖的核心不是简单奖励单个合伙人个人业绩，而是通过识别合伙人名下两条核心市场的推广业绩，推动合伙人平衡发展团队。

在实际经营中，部分合伙人可能只重点扶持其中一个市场，导致一边市场发展较好，另一边市场发展较弱。长期来看，这会影响整个分销团队的稳定增长，也会影响团队长之间的协同和收益平衡。

因此，渠道奖希望通过“第一大市场”和“第二大市场”的业绩对比，让合伙人不仅关注自己做得好，也要帮助伙伴、渠道长和团队一起做好。

### 1.2 当前问题

当前业务中存在以下问题：

1. 运营缺少一个专门观察合伙人两大市场发展情况的统计工具。
2. 门店缺少一个清晰的数据页面，用于判断哪些合伙人的两条核心市场发展不均衡。
3. 总部希望能够通过统一规则，引导各门店推动分销团队均衡成长。
4. 历史月份订单可能后续发生退款，如果统计结果不支持重算，会导致渠道奖金额与当前有效订单状态不一致。
5. 渠道奖需要给运营和门店提供发放参考，但当前系统不需要做自动发放。

### 1.3 业务目标

本需求的业务目标是：

- 新增渠道奖设置能力，支持总部和门店配置渠道奖规则。
- 新增渠道奖统计能力，按月查看合伙人第一大市场、第二大市场推广业绩和渠道奖金额。
- 让总部可以查看整个连锁范围内的渠道奖数据。
- 让门店只能查看本门店范围内的渠道奖数据。
- 支持总部将渠道奖规则下发到门店。
- 支持门店在总部下发规则后自行修改本门店规则。
- 支持历史月份按当前最新订单状态和当前最新生效规则重算。
- 支持权限控制，避免无权限人员误改设置或误重算历史数据。
- 为运营和门店提供渠道奖发放参考，但不在系统内自动发放奖金。

### 1.4 用户角色

| 用户角色 | 使用目标 | 关注点 |
|---|---|---|
| 连锁总部运营 | 配置渠道奖规则，查看全连锁渠道奖数据，分析各门店渠道发展情况 | 全局数据、门店对比、规则统一、数据导出 |
| 门店运营 | 查看本门店渠道奖数据，辅助门店进行团队激励和奖励发放 | 本店数据、合伙人表现、两大市场平衡情况 |
| 管理员 | 配置菜单权限、浏览权限、操作权限 | 权限隔离、操作安全、数据范围 |
| 财务/结算人员 | 基于导出结果核对奖励金额，支持后续线下或其他流程发放 | 金额准确性、历史重算、导出结果 |
| 测试人员 | 验证页面、权限、规则、计算、重算、导出等场景 | 业务边界、异常场景、数据口径 |
| 开发人员 | 实现设置、统计、计算、权限、重算、导出能力 | 数据来源、计算规则、权限校验、任务逻辑 |

### 1.5 典型使用场景

#### 场景 1：总部配置并下发渠道奖规则

总部运营在总部后台配置渠道奖开关和佣金比例，并将规则分发到全部门店或指定门店。门店接收规则后，可基于总部规则继续自行修改。

#### 场景 2：门店查看本店渠道奖数据

门店运营进入渠道奖统计页面，按月份查看本门店合伙人的第一大市场、第二大市场推广业绩、佣金基数和渠道奖金额。

#### 场景 3：运营推动两大市场均衡发展

运营通过统计结果发现某合伙人的第一大市场业绩很高，但第二大市场业绩很低。运营可以据此引导合伙人帮助弱市场团队长提升业绩，推动团队平衡成长。

#### 场景 4：历史月份退款后重算

某历史月份的订单在后续发生退款，运营选择该历史月份点击重算。系统按当前最新订单状态和当前最新生效规则重新计算渠道奖，剔除已退款订单影响。

#### 场景 5：奖励发放参考

系统计算并导出渠道奖统计结果。运营或门店基于该结果进行线下奖励发放或进入其他结算流程。本功能本身不做系统内自动发放。

### 1.6 核心业务术语

| 术语 | 定义 |
|---|---|
| 合伙人 | 分销体系中的推广主体，也是渠道奖统计对象 |
| 直推下级 | 合伙人直接邀请或推荐的下一级合伙人 |
| 渠道 | 某个直推下级本人及其下方所有层级成员形成的完整网络 |
| 渠道长 | 某条渠道对应的直推下级本人 |
| 推广业绩 | 系统现有分销体系中已经存在的推广业绩字段 |
| 渠道推广业绩 | 某条渠道内所有成员推广业绩之和 |
| 第一大市场 | 合伙人所有直推渠道中，渠道推广业绩最高的渠道 |
| 第二大市场 | 合伙人所有直推渠道中，渠道推广业绩第二高的渠道 |
| 渠道奖 | 基于第一大市场和第二大市场推广业绩计算出的奖励金额 |
| 重算 | 对历史月份按当前最新订单状态和当前最新生效规则重新计算渠道奖统计结果 |
| 规则分发 | 总部将渠道奖规则下发到门店 |
| 门店自定义规则 | 门店基于总部下发规则或本店需求自行修改后的渠道奖规则 |

### 1.7 系统边界

本需求只做渠道奖的设置、统计、重算、导出和权限控制。

本需求不做以下内容：

- 不修改现有其他奖项的计算逻辑。
- 不影响见单奖、帮扶奖、育才奖、分红奖等其他奖项。
- 不参与系统自动打款。
- 不做系统内自动发放渠道奖。
- 不改变订单主流程。
- 不改变售后退款主流程。
- 不改变推广业绩原始生成逻辑。
- 不修改订单原始数据。
- 不修改其他分销奖励结算流程。

### 1.8 数据边界

本需求读取现有数据用于渠道奖统计，不反向修改原业务数据。

渠道奖统计依赖：

- 现有合伙人关系链路。
- 现有门店/连锁组织关系。
- 现有推广业绩字段。
- 现有订单退款状态。
- 现有权限体系。

渠道奖统计不重新定义推广业绩，不按订单总金额或实收金额重新计算推广业绩。

## 2. 业务结果说明

### 2.1 为什么做渠道奖

渠道奖用于引导合伙人平衡发展两条核心市场。

该奖项不是只奖励某一边市场做得好，而是通过两大市场的业绩关系，让合伙人意识到：如果只发展一边市场，另一边市场过弱，渠道奖佣金基数会受限制；如果两边市场相对均衡，渠道奖结果会更有利。

所以渠道奖的业务价值是：

- 促进合伙人平衡发展团队。
- 推动强市场带动弱市场。
- 帮助运营识别团队结构是否健康。
- 让合伙人不仅关注自己做得好，也关注伙伴和团队长是否一起做好。

### 2.2 期望达成的业务结果

上线后，希望实现以下结果：

1. 总部可以统一配置渠道奖规则，并查看全连锁渠道奖数据。
2. 门店可以查看本店渠道奖统计结果，辅助门店激励合伙人。
3. 运营可以通过第一大市场和第二大市场推广业绩，判断市场是否均衡。
4. 系统可以按月、按 T+1 展示渠道奖统计结果。
5. 历史月份可以按最新订单状态和最新规则重算，保证数据可校正。
6. 渠道奖统计结果可以导出，作为运营核对和线下发放参考。
7. 权限控制清晰，能看和能操作分离，降低误操作风险。

### 2.3 验证方式

可以通过以下方式验证需求是否达成：

- 是否能正确识别合伙人的第一大市场和第二大市场。
- 第一大市场和第二大市场是否取系统已有推广业绩。
- 渠道奖佣金基数是否按规则正确计算。
- 总部是否只能在总部数据范围内查看和操作。
- 门店是否只能查看和操作当前门店数据。
- 总部下发规则后，门店是否可以自行修改。
- 历史月份重算是否按当前最新生效规则执行。
- 历史月份重算是否剔除已退款订单影响。
- 无操作权限用户是否不能保存设置或执行重算。
- 导出结果是否与页面筛选结果一致。

### 2.4 不做该需求的影响

如果不做渠道奖：

- 运营无法系统化观察合伙人两大市场的平衡情况。
- 门店缺少可用于激励团队均衡发展的数据依据。
- 合伙人可能继续只发展强市场，弱市场长期缺少带动。
- 历史月份退款后，渠道奖统计结果无法主动校正。
- 运营需要通过人工方式整理数据，效率低且容易出错。

## 3. 需求分类判断

### 3.1 需求类型

本需求属于：

`存量分销体系下新增奖项设置与统计模块`

它不是完全从 0 到 1 的新系统，而是在现有分销体系、推广业绩、合伙人关系链路和组织架构基础上，新增一种渠道奖的规则配置、数据统计和结果导出能力。

### 3.2 是否为 0-1 新需求

不是独立 0-1 新系统，但属于存量系统中的新增业务模块。

新增内容包括：

- 新增渠道奖设置页面。
- 新增渠道奖统计页面。
- 新增渠道奖权限点。
- 新增渠道奖计算规则。
- 新增历史月份重算能力。
- 新增渠道奖统计导出能力。

### 3.3 是否为耦合需求

本需求不是强耦合改造需求，不影响其他奖励模块。

准确表述为：

`本需求依赖现有分销数据，但不改变现有分销数据的生成逻辑，也不影响其他奖项的计算、统计、结算或发放。`

### 3.4 是否为复杂交互需求

是轻中度复杂后台交互需求。

复杂点主要在：

- 总部和门店两种组织视角。
- 浏览权限和操作权限分离。
- 总部规则下发。
- 门店可自行修改总部下发规则。
- 第一大市场和第二大市场计算。
- 历史月份重算。
- 重算按当前最新生效规则执行。

### 3.5 是否为智能体需求

不是智能体需求。

本需求不涉及 AI 智能体行为规范、Eval 样本、自动决策或自动执行闭环。

## 4. 总体功能范围

### 4.1 功能适用后台

该功能同时适用于：

- 连锁总部后台
- 门店后台

两个后台都需要具备：

- 渠道奖设置
- 渠道奖统计

### 4.2 菜单范围

#### 渠道奖设置

菜单位置：

`奖励设置 > 渠道奖设置`

要求：

- 在 `奖励设置` 下，`育才奖设置` 下面增加 `渠道奖设置`。
- `渠道奖设置` 与 `育才奖设置` 为同级菜单。
- 页面风格沿用现有 `育才奖设置` 页面。

#### 渠道奖统计

菜单位置：

`分红结算 > 渠道奖统计`

要求：

- 在 `分红结算` 下面增加 `渠道奖统计`。
- `渠道奖统计` 与现有分红结算类页面入口逻辑保持一致。
- 页面风格沿用现有 `分红结算` 页面。

### 4.3 页面范围

本需求包含两个页面：

1. 渠道奖设置页面
2. 渠道奖统计页面

### 4.4 能力范围

| 能力 | 是否支持 | 说明 |
|---|---|---|
| 渠道奖开关 | 支持 | 控制是否启用渠道奖统计 |
| 佣金比例设置 | 支持 | 配置渠道奖佣金比例 |
| 总部规则分发 | 支持 | 总部可将规则下发到门店 |
| 门店自行修改 | 支持 | 门店可修改总部下发规则 |
| 月度统计 | 支持 | 按月展示渠道奖结果 |
| T+1 更新 | 支持 | 展示截至昨日的统计数据 |
| 历史月份重算 | 支持 | 按当前最新数据和规则重算 |
| 导出 | 支持 | 导出当前筛选结果 |
| 系统自动发放 | 不支持 | 本功能只输出统计结果，不发放奖金 |

## 5. 组织权限与数据范围

### 5.1 连锁总部后台

总部后台可：

- 进入渠道奖设置页面。
- 进入渠道奖统计页面。
- 查看整个连锁范围内的渠道奖数据。
- 按门店筛选渠道奖数据。
- 配置总部渠道奖规则。
- 将渠道奖规则分发到全部门店或指定门店。
- 查看门店规则来源状态。
- 对历史月份数据执行重算。
- 导出全部门店或指定门店的渠道奖统计结果。

### 5.2 门店后台

门店后台可：

- 进入渠道奖设置页面。
- 进入渠道奖统计页面。
- 只能查看当前门店渠道奖数据。
- 只能查看当前门店范围内合伙人的渠道奖结果。
- 接收总部下发的渠道奖规则。
- 基于总部下发规则自行修改并保存。
- 对当前门店范围内的历史月份数据执行重算。
- 导出当前门店范围内的渠道奖统计结果。

门店后台不可：

- 查看其他门店渠道奖数据。
- 重算其他门店渠道奖数据。
- 导出其他门店渠道奖数据。

### 5.3 数据隔离规则

- 总部视角的数据范围为整个连锁。
- 总部筛选具体门店后，数据范围为被选门店。
- 门店视角的数据范围固定为当前门店。
- 门店不展示可切换门店的数据筛选能力，或门店筛选锁定为当前门店。

## 6. 页面权限控制

渠道奖设置和渠道奖统计两个页面，在总部后台和门店后台都需要设置权限。

每个页面至少拆分两类权限：

- 浏览权限
- 操作权限

### 6.1 渠道奖设置：浏览权限

权限名称建议：

`渠道奖设置-浏览权限`

拥有该权限的用户可以：

- 查看渠道奖设置菜单。
- 进入渠道奖设置页面。
- 查看渠道奖机制开启状态。
- 查看当前佣金比例。
- 查看当前规则来源。
- 查看渠道奖规则说明。
- 查看总部下发状态或门店自定义状态。

没有该权限的用户：

- 不展示渠道奖设置菜单；或
- 进入页面时提示无权限；
- 不允许查看渠道奖规则配置内容。

### 6.2 渠道奖设置：操作权限

权限名称建议：

`渠道奖设置-操作权限`

拥有该权限的用户可以：

- 开启渠道奖机制。
- 关闭渠道奖机制。
- 修改佣金比例。
- 点击保存设置。
- 总部用户可点击分发到门店。
- 门店用户可基于总部下发规则自行修改并保存。

没有该权限的用户：

- 可以查看页面，但不能修改配置。
- 开关、输入框、保存按钮应置灰或隐藏。
- 不允许保存设置。
- 不允许分发规则到门店。
- 不允许修改总部下发规则。

核心控制点：

`保存设置权限`

### 6.3 渠道奖统计：浏览权限

权限名称建议：

`渠道奖统计-浏览权限`

拥有该权限的用户可以：

- 查看渠道奖统计菜单。
- 进入渠道奖统计页面。
- 查看渠道奖统计列表。
- 查看最新统计截止时间。
- 查看最近重算时间。
- 使用月份筛选。
- 总部使用门店筛选。
- 使用合伙人姓名、手机号搜索。
- 查看第一大市场、第二大市场推广业绩。
- 查看佣金基数、佣金比例和佣金金额。

没有该权限的用户：

- 不展示渠道奖统计菜单；或
- 进入页面时提示无权限；
- 不允许查看渠道奖统计数据。

### 6.4 渠道奖统计：操作权限

权限名称建议：

`渠道奖统计-操作权限`

拥有该权限的用户可以：

- 点击历史月份的重算按钮。
- 对权限范围内的数据执行历史月份重算。
- 总部用户可对全部门店或指定门店执行重算。
- 门店用户只能对当前门店执行重算。
- 执行重算后，系统记录操作人和最近重算时间。

没有该权限的用户：

- 可以查看统计数据，但不能执行重算。
- 重算按钮应置灰或隐藏。
- 点击按钮时提示无操作权限。

核心控制点：

`历史月份重算/重刷按钮权限`

按钮文案建议统一为：

`重算`

若现有系统习惯使用“重刷”，也可以使用“重刷”，但业务含义必须明确为：

`按当前最新订单状态和当前最新生效规则，重新计算所选历史月份渠道奖数据。`

## 7. 规则配置与分发

### 7.1 渠道奖设置项

渠道奖设置页需要支持以下配置：

- 渠道奖机制开关
- 渠道奖佣金比例
- 规则来源展示
- 总部分发到门店

默认规则：

- 渠道奖机制默认关闭。
- 开关关闭时，佣金比例不可编辑，且不参与每日渠道奖统计计算。
- 开关开启后，才允许填写和保存佣金比例。
- 佣金比例按百分比数字输入，例如输入 `8` 表示 `8%`。
- 佣金比例为统一比例，不按不同市场、不同合伙人或不同层级单独配置。
- `复制给分店` 仅在连锁总部后台展示，门店后台不展示该操作。
- 门店后台只有 `保存设置` 操作，门店保存后只影响当前门店的渠道奖规则。

### 7.2 总部下发规则

总部可配置渠道奖规则，并将规则下发到：

- 全部门店
- 指定门店

总部下发内容包括：

- 是否开启渠道奖机制
- 渠道奖佣金比例

### 7.3 门店允许自行修改

总部下发规则后，门店允许自行修改。

规则下发后：

- 门店可以看到当前规则来源为总部下发。
- 门店可以基于总部下发规则进行二次修改。
- 门店修改并保存后，当前门店后续渠道奖统计按门店自行修改后的规则执行。
- 总部再次下发规则时，系统应提示可能覆盖门店已有自定义规则。
- 若总部重新下发并覆盖门店规则，则门店后续按总部最新下发规则执行。
- 若门店后续再次修改，则该门店继续按门店自定义规则执行。

### 7.4 规则来源状态

建议增加规则来源状态：

| 状态 | 含义 |
|---|---|
| 总部下发 | 当前门店使用总部下发规则，未自行修改 |
| 门店自定义 | 当前门店自行配置规则 |
| 总部下发后门店已修改 | 门店曾接收总部规则，但已在本店进行修改 |

### 7.5 规则优先级

规则优先级如下：

1. 门店存在自定义规则时，优先使用门店当前最新自定义规则。
2. 门店没有自定义规则时，使用总部下发到该门店的当前最新规则。
3. 总部视角重算多个门店时，各门店分别按自己的当前最新生效规则计算。

## 8. 渠道奖计算规则

### 8.1 统计对象

统计对象为分销体系中的合伙人。

每个合伙人以自己的直推下级作为市场划分起点。

每个直推下级及其下方全部层级成员共同构成一条市场渠道。

### 8.2 推广业绩口径

第一大市场和第二大市场的渠道业绩，统一取系统现有的 `推广业绩`。

也就是说：

- 不需要重新定义新的业绩口径。
- 不取订单总金额。
- 不取普通销售业绩。
- 不取实收总额重新计算。
- 直接沿用当前系统已经存在的推广业绩值。

### 8.3 渠道推广业绩

某个直推下级的渠道推广业绩，等于该直推下级本人推广业绩，加上其下方所有层级成员的推广业绩总和。

即：

`渠道推广业绩 = 渠道长本人推广业绩 + 该渠道下所有成员推广业绩`

渠道范围按无限层级向下递归统计，不只统计一层或两层。

### 8.4 第一大市场和第二大市场

对某个合伙人，系统先统计其所有直推下级对应的渠道推广业绩。

然后按渠道推广业绩从高到低排序：

- 排名第 1 的为第一大市场。
- 排名第 2 的为第二大市场。
- 若直推下级超过 2 个，仅取前 2 名参与渠道奖计算。

判断流程：

```text
读取合伙人所有直推下级
→ 获取每个直推下级对应渠道的推广业绩
→ 按推广业绩从高到低排序
→ 排名第 1 的为第一大市场
→ 排名第 2 的为第二大市场
```

需要注意：

- 排名第 3 及之后的渠道，不参与渠道奖佣金基数计算。
- 排名第 3 及之后的渠道仍属于该合伙人的渠道网络组成部分。
- 第一大市场和第二大市场展示的业绩字段，均展示推广业绩。

### 8.5 触发条件

渠道奖必须同时满足以下条件才可计算：

- 渠道奖机制已开启。
- 合伙人名下至少有 2 个直推下级。
- 第一大市场推广业绩大于 0。
- 第二大市场推广业绩大于 0。
- 当前生效的佣金比例已配置。

若不满足以上条件：

`当前统计结果中的渠道奖佣金 = 0`

### 8.6 计算公式

设：

- `大市场业绩 = 第一大市场推广业绩与第二大市场推广业绩中的较大值`
- `小市场业绩 = 第一大市场推广业绩与第二大市场推广业绩中的较小值`
- `佣金比例 = 当前最新生效的渠道奖佣金比例`

计算逻辑：

```text
如果 大市场业绩 ÷ 小市场业绩 > 2
则：
佣金基数 = 小市场业绩
渠道奖佣金 = 小市场业绩 × 佣金比例

如果 大市场业绩 ÷ 小市场业绩 ≤ 2
则：
佣金基数 = 大市场业绩 ÷ 2
渠道奖佣金 = (大市场业绩 ÷ 2) × 佣金比例
```

### 8.7 规则理解

该规则的核心是：

`按两条市场中相对均衡的那部分推广业绩作为佣金基数。`

具体含义：

- 当两条市场差距过大时，只按小市场推广业绩计算，避免单边市场过强时奖励失真。
- 当两条市场相对均衡时，按大市场推广业绩的一半作为基数，体现双市场平衡发展。

### 8.8 示例说明

#### 示例 1：均衡场景

- 第一大市场推广业绩：18000
- 第二大市场推广业绩：10000
- 比值：1.8
- 命中规则：均衡规则
- 佣金基数：18000 ÷ 2 = 9000

#### 示例 2：悬殊场景

- 第一大市场推广业绩：18000
- 第二大市场推广业绩：3000
- 比值：6
- 命中规则：悬殊规则
- 佣金基数：3000

#### 示例 3：临界场景

- 第一大市场推广业绩：20000
- 第二大市场推广业绩：10000
- 比值：2
- 命中规则：均衡规则
- 佣金基数：20000 ÷ 2 = 10000

## 9. 统计更新与历史重算

### 9.1 T+1 日常统计

渠道奖按 `月` 作为归属维度。

页面筛选维度为 `月份`。

渠道奖机制开启后，渠道推广业绩与佣金结果按 `T+1` 口径更新。

即：

`今天早上可看到截止昨天 23:59:59 的第一大市场、第二大市场推广业绩及当前佣金情况。`

页面必须明确展示：

`最新统计截止时间`

示例：

`当前展示数据截止到：2026-05-22 23:59`

### 9.2 渠道奖机制关闭

当渠道奖机制未开启时：

- 系统不触发每日统计计算。
- 不产生每日累计结果。
- 渠道奖统计页面应提示当前未启用每日统计计算。
- 渠道奖设置页面应提示“未开启时系统不会执行每日渠道奖统计计算”。

### 9.3 历史月份重算入口

在渠道奖统计页面，当用户选择历史月份时，页面提供 `重算` 按钮。

建议展示逻辑：

- 当前月份：默认按 T+1 自动更新，可不展示重算按钮。
- 历史月份：展示重算按钮。
- 总部视角：可对全部门店或指定门店的历史月份数据执行重算。
- 门店视角：只能对当前门店的历史月份数据执行重算。

### 9.4 重算权限

重算按钮受 `渠道奖统计-操作权限` 控制。

有操作权限：

- 可以点击重算按钮。
- 可以执行历史月份数据重算。

无操作权限：

- 可以查看数据，但不能执行重算。
- 重算按钮应置灰或隐藏。
- 点击时提示无权限。

### 9.5 重算规则

历史月份执行重算时，系统按 `当前最新生效规则` 重新计算，不按当月历史规则计算。

重算时需要：

- 读取所选月份当前最新订单状态。
- 剔除已全额退款订单。
- 按现有退款规则处理部分退款订单。
- 重新统计该月份推广业绩。
- 重新识别第一大市场和第二大市场。
- 按当前最新生效的渠道奖规则计算佣金基数和佣金金额。
- 覆盖更新该月份原统计结果。
- 记录最近一次重算时间和操作人。

### 9.6 重算规则优先级

重算时规则优先级：

- 若门店当前存在自定义规则，则按门店当前最新自定义规则重算。
- 若门店未自定义，当前使用总部下发规则，则按总部当前下发到该门店的最新规则重算。
- 总部视角重算多个门店时，各门店分别按自己当前最新生效规则计算。

### 9.7 重算结果

重算完成后：

- 历史月份数据更新为基于最新有效订单状态计算后的结果。
- 已退款订单不再计入对应渠道推广业绩。
- 第一大市场、第二大市场排序可能发生变化。
- 佣金基数和佣金金额可能发生变化。
- 页面刷新展示重算后的最新结果。
- 页面记录并展示最近一次重算时间。
- 页面记录并展示最近一次重算操作人。

### 9.8 重算确认提示

点击重算前，系统需要二次确认。

确认文案建议：

```text
确认要重算当前月份的渠道奖数据吗？

系统将按当前最新订单状态和当前最新生效规则，重新计算该月份渠道推广业绩和渠道奖佣金。
已退款订单将从统计结果中剔除，重算后会覆盖当前月份原有统计结果。
```

页面提示建议：

```text
历史月份重算将按当前最新生效规则重新计算，不再按该月份原历史规则计算。
重算后会覆盖当前月份原统计结果，请确认后操作。
```

### 9.9 重算状态

重算过程需要有明确状态反馈：

- 重算中
- 重算成功
- 重算失败

若重算失败，需要提示失败原因，并保留原统计结果。

建议失败原因包括：

- 当前月份订单数据读取失败。
- 推广关系链路数据异常。
- 渠道奖规则未配置。
- 门店权限异常。
- 系统计算异常。

│ 第二大市场信息                                                               │
│ 第二大市场推广业绩                                                           │
│ 当前佣金基数                                                                 │
│ 佣金比例                                                                     │
│ 佣金金额                                                                     │
└──────────────────────────────────────────────────────────────────────────────┘

### 筛选条件

总部后台支持：

- 月份

- 合伙人手机号

门店后台支持：

- 月份
- 合伙人姓名
- 合伙人手机号

门店后台的门店范围固定为当前门店，不允许切换。

### 页面交互要求

- 顶部必须有月份选择器，且只支持按月选择。
- 顶部必须展示最新统计截止时间。
- 历史月份需要支持重算。
- 重算后需要展示最近重算时间和最近重算操作人。
- 总部视角必须支持门店筛选。
- 门店视角门店默认锁定为当前门店。
- 渠道奖机制未开启时，页面应提示当前未启用每日统计计算。
- 支持点击导出，导出当前筛选结果。
- 列表默认按佣金金额倒序排列。
- 点击重算必须校验用户是否拥有统计页操作权限。
- 无操作权限时，不允许执行重算。

## 12. 数据依赖与影响边界说明

### 12.1 本需求新增内容

本需求新增：

- 渠道奖设置菜单。
- 渠道奖统计菜单。
- 渠道奖设置页面。
- 渠道奖统计页面。
- 渠道奖规则配置。
- 总部规则分发。
- 门店规则自定义。
- 渠道奖统计计算。
- 历史月份重算。
- 渠道奖导出。
- 渠道奖相关权限点。

### 12.2 本需求依赖的数据

本需求依赖读取：

- 合伙人关系链路。
- 合伙人等级。
- 合伙人手机号和姓名。
- 门店归属关系。
- 连锁总部与门店关系。
- 系统已有推广业绩。
- 订单退款状态。
- 权限配置。

### 12.3 本需求不影响的范围

本需求不影响：

- 其他奖励设置。
- 其他奖励统计。
- 其他分销奖项计算。
- 分红结算主流程。
- 见单奖、帮扶奖、育才奖、分红奖等已有奖项。
- 订单主流程。
- 售后退款主流程。
- 订单原始数据。
- 推广业绩原始生成逻辑。
- 系统自动发放流程。

### 12.4 与其他奖项的关系

渠道奖与其他奖项是并列关系。

渠道奖不参与其他奖项计算，也不影响其他奖项结果。

其他奖项也不影响渠道奖计算，渠道奖只读取现有推广业绩和合伙人关系链路。

### 12.5 与订单/售后/结算的关系

本需求与订单、售后、结算模块没有流程改造关系，只有数据读取关系。

- 订单模块：提供推广业绩和订单状态数据。
- 售后模块：提供退款完成状态，用于历史月份重算时剔除或冲减。
- 结算模块：本需求不自动生成结算单，不自动发放奖金。

### 12.6 与奖励发放的关系

渠道奖统计结果仅作为运营和门店发放奖励的参考依据。

系统不自动发放渠道奖。

后续如需要发放，可由运营或门店基于导出结果在线下或其他结算流程中处理。

## 14. 业务流程

### 14.1 主流程

```text
商户在奖励设置中完成渠道奖规则配置
→ 开启渠道奖机制
→ 如为总部后台，可通过复制给分店将规则同步到门店
→ 门店可基于总部下发规则自行修改
→ 系统按当前生效佣金比例应用规则
→ 系统每日按 T+1 更新渠道奖统计数据
→ 今日早上生成截至昨日的最新市场推广业绩与佣金结果
→ 运营进入渠道奖统计页面按月份查看结果
→ 总部可按门店筛选，门店仅查看当前门店
→ 运营可按姓名、手机号查询目标合伙人
→ 系统按佣金金额从高到低展示统计列表
→ 运营可导出当月渠道奖统计结果
→ 运营或门店基于统计结果进行线下奖励发放参考
```

### 14.2 设置流程

```text
用户进入渠道奖设置页面
→ 系统校验是否有浏览权限
→ 有浏览权限则展示当前规则
→ 默认状态下渠道奖机制开关关闭
→ 开关关闭时不展示佣金比例输入区
→ 用户开启渠道奖机制后，系统展示佣金比例输入区
→ 用户填写或修改佣金比例
→ 点击保存设置
→ 系统校验是否有操作权限
→ 有操作权限则保存成功
→ 无操作权限则提示无权限
```

### 14.3 总部分发流程

```text
总部用户进入渠道奖设置页面
→ 系统校验浏览权限
→ 总部开启渠道奖机制并配置佣金比例
→ 点击复制给分店
→ 系统校验操作权限
→ 选择全部门店或指定门店
→ 系统提示可能覆盖门店已有规则
→ 用户确认
→ 总部当前开关状态和佣金比例复制到门店
→ 门店后续按复制后的规则执行
```

### 14.4 门店自定义流程

```text
门店用户进入渠道奖设置页面
→ 页面不展示复制给分店
→ 查看总部复制/下发到当前门店的规则
→ 门店用户可修改开关或佣金比例
→ 点击保存设置
→ 系统校验操作权限
→ 保存成功后，规则来源变为总部下发后门店已修改
→ 当前门店后续按门店自定义规则执行
```

### 14.5 计算流程

```text
读取某合伙人全部直推下级
→ 获取每个直推下级对应渠道的推广业绩
→ 按推广业绩排序
→ 取前 2 名作为第一大市场和第二大市场
→ 判断是否满足渠道奖计算条件
→ 判断大市场与小市场比值
→ 按命中规则计算佣金基数
→ 读取当前生效佣金比例
→ 计算渠道奖佣金金额
→ 写入当前月份截至昨日的最新统计结果
```

### 14.6 历史月份重算流程

```text
运营选择历史月份
→ 点击重算
→ 系统校验是否有统计页操作权限
→ 有权限则弹出二次确认
→ 用户确认重算
→ 系统读取该月份当前最新订单状态
→ 剔除已全额退款订单
→ 按退款规则处理部分退款订单
→ 重新统计渠道推广业绩
→ 重新识别第一大市场和第二大市场
→ 按当前最新生效规则重新计算佣金基数和佣金金额
→ 覆盖更新该月份统计结果
→ 记录最近重算时间和操作人
→ 页面刷新展示最新结果
```

## 15. 异常流程

### 15.1 无浏览权限

```text
用户访问渠道奖设置或渠道奖统计页面
→ 系统校验无浏览权限
→ 不展示对应菜单
或
→ 页面提示无权限访问
```

### 15.2 无设置操作权限

```text
用户进入渠道奖设置页面
→ 用户修改配置并点击保存
→ 系统校验无操作权限
→ 阻止保存
→ 提示无权限操作
```

### 15.3 无统计操作权限

```text
用户进入渠道奖统计页面
→ 用户点击重算按钮
→ 系统校验无操作权限
→ 阻止重算
→ 提示无权限操作
```

### 15.4 直推市场不足

```text
系统统计某合伙人直推市场
→ 发现有效市场不足 2 条
→ 不触发渠道奖计算
→ 该合伙人当前佣金金额为 0
```

### 15.5 第二大市场推广业绩为 0

```text
系统识别第一大市场和第二大市场
→ 第二大市场推广业绩为 0
→ 不触发正常比值计算
→ 该合伙人当前佣金金额为 0
```

### 15.6 佣金比例未配置

```text
系统完成市场排名与佣金基数判断
→ 发现当前统一佣金比例未配置
→ 阻止生成正常佣金结果
→ 页面提示规则待配置
```

### 15.7 重算失败

```text
用户点击历史月份重算
→ 系统执行重算
→ 重算过程中发生异常
→ 系统提示重算失败原因
→ 保留原统计结果
→ 用户可稍后重新发起重算
```

## 17. 测试用例

### 17.1 菜单与页面测试

1. 验证 `奖励设置` 下存在 `渠道奖设置` 菜单，且与 `育才奖设置` 同级。
2. 验证 `分红结算` 下存在 `渠道奖统计` 菜单。
3. 验证渠道奖功能总部后台可进入。
4. 验证渠道奖功能门店后台可进入。
5. 验证渠道奖设置页面风格与现有 `育才奖设置` 页面一致。
6. 验证渠道奖统计页面风格与现有 `分红结算` 页面一致。

### 17.2 权限测试

1. 验证无 `渠道奖设置-浏览权限` 时，不展示渠道奖设置菜单或提示无权限。
2. 验证有 `渠道奖设置-浏览权限` 时，可查看渠道奖设置页面。
3. 验证无 `渠道奖设置-操作权限` 时，不可保存设置。
4. 验证无 `渠道奖设置-操作权限` 时，不可修改开关和佣金比例。
5. 验证有 `渠道奖设置-操作权限` 时，可点击保存设置。
6. 验证总部无设置操作权限时，不可分发规则到门店。
7. 验证总部有设置操作权限时，可分发规则到门店。
8. 验证门店无设置操作权限时，不可修改总部下发规则。
9. 验证门店有设置操作权限时，可修改总部下发规则。
10. 验证无 `渠道奖统计-浏览权限` 时，不展示渠道奖统计菜单或提示无权限。
11. 验证有 `渠道奖统计-浏览权限` 时，可查看统计页面。
12. 验证无 `渠道奖统计-操作权限` 时，不可点击重算按钮。
13. 验证有 `渠道奖统计-操作权限` 时，可执行历史月份重算。
14. 验证总部只能在权限范围内重算全部门店或指定门店数据。
15. 验证门店只能重算当前门店数据。

### 17.3 组织数据范围测试

1. 验证总部支持查看整个连锁范围内渠道奖数据。
2. 验证总部支持按门店筛选渠道奖数据。
3. 验证门店只能查看当前门店渠道奖数据。
4. 验证门店不能查看其他门店渠道奖数据。
5. 验证门店后台门店筛选锁定为当前门店。

### 17.4 规则分发测试

1. 验证连锁总部后台展示 `复制给分店` 操作。
2. 验证门店后台不展示 `复制给分店` 操作。
3. 验证总部点击 `复制给分店` 后，支持选择全部门店或指定门店。
4. 验证总部可将当前渠道奖机制开关状态和佣金比例复制给门店。
5. 验证门店侧可看到总部下发规则来源。
6. 验证总部下发规则后，门店允许自行修改。
7. 验证门店修改后，规则状态变为 `总部下发后门店已修改`。
8. 验证总部再次下发规则时，系统提示可能覆盖门店已有自定义规则。
9. 验证总部覆盖下发后，门店规则按总部最新下发规则执行。
10. 验证门店再次修改后，后续按门店自定义规则执行。

### 17.5 渠道奖设置测试

1. 验证渠道奖机制开关默认关闭。
2. 验证渠道奖设置页不展示单独的大标题说明块。
3. 验证渠道奖设置页说明内容为小字浅色辅助信息。
4. 验证渠道奖设置页关闭时不展示佣金比例输入区和规则说明区。
5. 验证渠道奖设置页开启后才展示佣金比例输入区和规则说明区。
6. 验证渠道奖设置页开启后才允许填写佣金比例。
7. 验证佣金比例输入 `8` 时按 `8%` 理解。
8. 验证渠道奖机制未开启时不执行每日统计计算。
9. 验证渠道奖机制开启后才按 T+1 执行每日统计计算。
10. 验证总部后台底部展示 `复制给分店` 和 `保存设置`。
11. 验证门店后台底部只展示 `保存设置`。
12. 验证门店保存设置后，只影响当前门店渠道奖规则。
13. 验证无操作权限时，开关、佣金比例输入框、`复制给分店`、`保存设置` 不可操作。

### 17.6 渠道奖统计测试

1. 验证渠道奖统计页仅支持按月筛选。
2. 验证渠道奖统计页展示最新统计截止时间。
3. 验证总部视角支持门店筛选。
4. 验证分店视角门店范围锁定为当前门店。
5. 验证系统按 T+1 更新统计结果，今天早上可看到截止昨天的数据。
6. 验证列表默认按佣金金额从高到低排序。
7. 验证支持按门店、合伙人姓名和手机号搜索。
8. 验证支持导出当前筛选结果。
9. 验证历史月份展示重算按钮。
10. 验证点击重算前有二次确认。
11. 验证重算后页面展示最近重算时间。
12. 验证重算后页面展示最近重算操作人。

### 17.7 计算逻辑测试

1. 验证系统可正确识别第一大市场和第二大市场。
2. 验证第一大市场业绩取系统已有推广业绩。
3. 验证第二大市场业绩取系统已有推广业绩。
4. 验证当直推市场超过 2 条时，仅取前 2 名参与渠道奖计算。
5. 验证当有效市场不足 2 条时，渠道奖结果为 0。
6. 验证当第二大市场推广业绩为 0 时，渠道奖结果为 0。
7. 验证当大市场业绩 ÷ 小市场业绩 > 2 时，佣金基数取小市场业绩。
8. 验证当大市场业绩 ÷ 小市场业绩 ≤ 2 时，佣金基数取大市场业绩的一半。
9. 验证系统按当前生效佣金比例计算结果。
10. 验证列表中第一大市场推广业绩和第二大市场推广业绩单独成列显示。

### 17.8 重算测试

1. 验证历史月份可执行重算。
2. 验证当前月份默认按 T+1 自动更新。
3. 验证重算时按当前最新订单状态计算。
4. 验证重算后会剔除已全额退款订单。
5. 验证重算后会按规则处理部分退款订单。
6. 验证重算后会重新识别第一大市场和第二大市场。
7. 验证重算后会按当前最新生效规则计算。
8. 验证重算不按当月历史规则计算。
9. 验证总部重算多个门店时，各门店分别按自己的当前最新生效规则计算。
10. 验证重算失败时保留原统计结果并提示失败原因。

### 17.9 影响边界测试

1. 验证渠道奖不影响其他奖项设置。
2. 验证渠道奖不影响其他奖项统计。
3. 验证渠道奖不影响现有分红结算流程。
4. 验证渠道奖不自动生成发放记录。
5. 验证渠道奖不自动打款。
6. 验证渠道奖只读取推广业绩，不修改推广业绩原始数据。
7. 验证渠道奖重算不修改订单原始数据。

## 18. 场景测试

### 场景 1：均衡场景

- 第一大市场推广业绩：18000
- 第二大市场推广业绩：10000
- 比值：1.8
- 命中规则：均衡规则
- 佣金基数：18000 ÷ 2 = 9000

### 场景 2：悬殊场景

- 第一大市场推广业绩：18000
- 第二大市场推广业绩：3000
- 比值：6
- 命中规则：悬殊规则
- 佣金基数：3000

### 场景 3：临界场景

- 第一大市场推广业绩：20000
- 第二大市场推广业绩：10000
- 比值：2
- 命中规则：均衡规则
- 佣金基数：20000 ÷ 2 = 10000

### 场景 4：单市场场景

- 只有 1 条有效直推市场
- 不触发渠道奖计算
- 佣金金额为 0

### 场景 5：历史月份退款后重算

原始统计：

- 统计月份：2026年05月
- 第一大市场推广业绩：30000
- 第二大市场推广业绩：12000
- 佣金基数：12000

后续情况：

- 第一大市场中有一笔 10000 元订单在 2026年06月发生全额退款

重算后：

- 第一大市场推广业绩变为：20000
- 第二大市场推广业绩仍为：12000
- 比值：1.67
- 命中规则：均衡规则
- 新佣金基数：20000 ÷ 2 = 10000

### 场景 6：历史月份退款导致市场排名变化

原始统计：

- 渠道 A 推广业绩：30000
- 渠道 B 推广业绩：20000
- 渠道 C 推广业绩：15000
- 第一大市场：渠道 A
- 第二大市场：渠道 B

后续情况：

- 渠道 A 中有 20000 元订单发生全额退款

重算后：

- 渠道 A 推广业绩：10000
- 渠道 B 推广业绩：20000
- 渠道 C 推广业绩：15000
- 第一大市场变为：渠道 B
- 第二大市场变为：渠道 C
- 系统按新的第一大市场、第二大市场重新计算渠道奖

### 场景 7：总部下发后门店修改规则

原始规则：

- 总部下发佣金比例：8%
- 门店 A 接收总部规则

门店操作：

- 门店 A 将佣金比例修改为 10%
- 保存成功

后续计算：

- 门店 A 后续渠道奖按 10% 计算
- 规则状态展示为：总部下发后门店已修改

### 场景 8：无操作权限不可保存设置

用户权限：

- 有渠道奖设置浏览权限
- 无渠道奖设置操作权限

用户操作：

- 进入渠道奖设置页面
- 尝试修改佣金比例并保存

系统结果：

- 阻止保存
- 提示无权限操作
- 原规则不变

### 场景 9：无操作权限不可重算

用户权限：

- 有渠道奖统计浏览权限
- 无渠道奖统计操作权限

用户操作：

- 进入渠道奖统计页面
- 选择历史月份
- 点击重算

系统结果：

- 阻止重算
- 提示无权限操作
- 原统计结果不变

### 场景 10：渠道奖不影响其他奖项

前置条件：

- 系统已有见单奖、帮扶奖、育才奖、分红奖等其他奖项

用户操作：

- 开启渠道奖
- 配置渠道奖佣金比例
- 查看渠道奖统计

系统结果：

- 其他奖项设置不变
- 其他奖项统计不变
- 其他奖项计算结果不变
- 渠道奖作为独立统计结果展示

## 19. 验收标准

### 19.1 功能验收

- 总部和门店后台都能按权限进入渠道奖设置和渠道奖统计页面。
- 渠道奖设置支持开关、佣金比例、规则来源展示、保存设置。
- 总部支持将规则下发到全部门店或指定门店。
- 门店支持修改总部下发规则。
- 渠道奖统计支持按月、门店、合伙人姓名、手机号筛选。
- 渠道奖统计支持导出。
- 历史月份支持重算。

### 19.2 数据验收

- 第一大市场和第二大市场取系统已有推广业绩。
- 第一大市场和第二大市场排序正确。
- 佣金基数计算正确。
- 佣金金额计算正确。
- T+1 统计截止时间展示正确。
- 历史重算按当前最新订单状态和当前最新生效规则执行。
- 重算失败不覆盖原数据。

### 19.3 权限验收

- 无浏览权限不可查看菜单或页面。
- 有浏览权限但无操作权限，只能查看，不能保存或重算。
- 设置页保存受设置操作权限控制。
- 统计页重算受统计操作权限控制。
- 总部和门店数据范围隔离正确。

### 19.4 边界验收

- 渠道奖不影响其他奖项。
- 渠道奖不自动发放奖金。
- 渠道奖不修改订单原始数据。
- 渠道奖不修改推广业绩原始数据。
- 渠道奖统计结果仅作为运营和门店发放参考。
