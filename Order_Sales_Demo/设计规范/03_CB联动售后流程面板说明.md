# C/B 联动售后流程场景面板说明

> 用途：本文按业务场景输出 C 端消费者页面链路、B 端商家处理链路和 C/B 状态联动，帮助设计、开发、测试按场景理解商家自营售后流程。
>
> 依据：`PRD/商家自营订单售后C-B联动完整PRD.md`、`PRD/商家自营C端-B端功能状态页面ASCII草图.md`、`设计规范/01_C端消费者端设计规范.md`。
>
> 输出原则：每个业务场景一个面板；C 端和 B 端分开表达；不做三栏联动看板；不把开发说明、测试用例、字段字典做成产品页面。

---

## 0. 场景面板统一结构

每个场景按以下结构整理：

```text
场景名称与适用订单
1. C 端消费者页面链路
2. B 端商家处理链路
3. C/B 状态联动表
4. 设计 / 开发 / 测试关注点
```

页面节点统一用短卡片表达：

```text
页面名称｜当前状态｜主按钮 / 动作｜点击后 / 处理后
```

关键口径：

1. 待发货商品物流订单点击 `申请退款` 后，直接进入 `仅退款申请页`，不走普通确认弹窗，不进入售后类型页。
2. 非物流退款型订单点击 `申请退款` 后，只弹普通退款确认弹窗；确认后直接进入 B 端 `待退款`。
3. 已发货商品订单点击 `申请售后` 后，只展示 `退货退款（已收到货）` 和 `仅退款（未收货仅退款）` 两个入口。
4. 补寄不是 C 端主动申请入口，只作为 B 端审核后的处理结果。
5. C 端不展示分账、责任归因、合伙人佣金、钱包扣减、云仓内部处理等后台概念。

## 1. 非物流订单仅退款场景

适用订单：商品自提待自提、套餐/次卡未核销、开卡充值资产满足回退、预定金权益未使用。

### 1.1 C 端消费者页面链路

```text
订单列表页｜状态：待自提 / 未核销 / 资产可退 / 权益未使用｜按钮：申请退款｜点击后：进入订单详情或直接触发退款入口
  ↓
订单详情页｜状态：可申请退款｜底部按钮：申请退款｜点击后：打开普通退款确认弹窗
  ↓
退款确认弹窗｜标题：退款申请｜按钮：取消 / 申请退款｜确认后：创建售后单
  ↓
售后详情页｜状态：待退款｜展示：等待商家处理退款、申请退款金额、预计退回方式｜按钮：联系卖家
  ↓
售后详情页｜状态：退款中｜展示：商家已发起退款、实际退回方式｜按钮：联系卖家
  ↓
售后详情页｜状态：退款成功｜展示：退款金额、退款方式、退款成功时间｜按钮：查看结果
```

### 1.2 B 端商家处理链路

```text
售后管理列表｜状态 Tab：待退款｜新增售后单：非物流仅退款｜操作：查看详情 / 操作退款
  ↓
售后详情窗口｜状态：待退款｜展示：订单业务形态、用户信息、退款金额、资产回退摘要｜主操作：操作退款
  ↓
操作退款弹窗｜确认：实际退款金额、退款方式明细、退款说明｜提交后：进入退款中
  ↓
售后详情窗口｜状态：退款中｜展示：退款单号、渠道状态、发起时间｜等待：支付渠道回执
  ↓
售后详情窗口｜状态：已完成｜展示：退款成功、退款水单、资产回退结果、操作日志
```

### 1.3 C/B 状态联动表

| C 端动作 / 页面 | B 端状态 | B 端处理结果 | C 端回显 |
| --- | --- | --- | --- |
| 用户确认退款申请 | 待退款 | 生成非物流退款型售后单 | 售后详情显示 `待退款` |
| 财务点击操作退款 | 退款中 | 发起退款，等待渠道回执 | 售后详情显示 `退款中` |
| 渠道退款成功 | 已完成 | 生成退款单 / 退款水单，回退资产 | 售后详情显示 `退款成功` |
| 渠道退款失败 | 退款失败 | 等待财务重新操作退款 | 售后详情显示 `退款失败` |

### 1.4 设计 / 开发 / 测试关注点

- 非物流订单不进入申请页，不展示退款原因、申请说明、图片凭证、商品/SKU/数量选择。
- 创建售后单时系统默认申请理由为 `NO_WANT / 不想要了`，C 端不展示该默认理由选择过程。
- 该场景不进入 `待审核 / 待寄回 / 待收货 / 补寄`。
- 自提、套餐/次卡、开卡充值、预定金都只支持整单全额退款。
- 退款成功后需要生成退款水单，并回退对应资产或权益。

## 2. 发货商品待发货仅退款场景

适用订单：商品物流发货订单，已付款未发货。覆盖整单退款、部分商品退款、同 SKU 部分数量退款。

### 2.1 C 端消费者页面链路

```text
订单列表页｜状态：待发货｜按钮：申请退款｜点击后：进入订单详情或直接进入仅退款申请页
  ↓
订单详情页｜状态：待发货｜底部按钮：申请退款 / 提醒发货｜点击申请退款：进入仅退款申请页
  ↓
仅退款申请页｜申请类型：仅退款｜选择：退款商品、SKU、数量｜填写：申请原因、申请退款金额、说明、联系电话｜提交后：创建售后单
  ↓
售后详情页｜状态：待审核｜展示：申请金额、预计退回方式、申请信息｜按钮：撤销申请 / 联系卖家
  ↓
售后详情页｜状态：待退款｜展示：最终审批退款金额、实际退回方式｜按钮：联系卖家
  ↓
售后详情页｜状态：退款成功｜展示：退款金额、退款方式、处理结果卡片｜按钮：查看结果
```

### 2.2 B 端商家处理链路

```text
售后管理列表｜状态 Tab：待审核｜售后类型：仅退款｜操作：审核
  ↓
售后详情窗口｜状态：待审核｜展示：退款商品、申请金额、最多可退金额、支付构成、申请原因｜操作：同意申请 / 拒绝申请 / 转补寄
  ↓
同意申请弹窗｜处理结果：同意仅退款｜填写：审批退款金额、原单运费退款金额、审核备注｜确认后：进入待退款
  ↓
售后详情窗口｜状态：待退款｜展示：审批金额、商品退款、原单运费退款、支付退回明细｜操作：操作退款
  ↓
操作退款弹窗｜确认：实际退款金额、退款方式明细｜提交后：退款中
  ↓
售后详情窗口｜状态：已完成 / 退款失败｜退款成功则完成，退款失败则等待重新操作
```

### 2.3 C/B 状态联动表

| C 端动作 / 页面 | B 端状态 | B 端处理结果 | C 端回显 |
| --- | --- | --- | --- |
| 提交待发货仅退款申请 | 待审核 | 生成仅退款售后单 | 售后详情显示 `待审核` |
| B 端同意仅退款 | 待退款 | 锁定审批退款金额 | 售后详情显示 `待退款` |
| B 端操作退款 | 退款中 | 发起退款 | 售后详情显示 `退款中` |
| 退款成功 | 已完成 | 生成退款记录和退款水单 | 售后详情显示 `退款成功` |
| B 端拒绝申请 | 已关闭 | 关闭类型：已拒绝 | 售后详情显示 `已关闭`，展示拒绝原因 |

### 2.4 设计 / 开发 / 测试关注点

- 待发货物流订单不弹普通退款确认弹窗，不进入售后类型页。
- 申请页必须支持商品、SKU、数量选择。
- 选择当前剩余全部商品和数量时，判定为整单 / 全额仅退款，最多可退金额默认包含原订单运费。
- 只选择部分商品、SKU 或数量时，判定为部分仅退款；若订单仍有商品需要发货，原订单运费默认不退。
- 申请理由展示待发货理由枚举，例如不想要了、拍错/多拍、地址错误、商家迟迟未发货。
- 部分退款成功后，原订单继续保持待发货，追加 `部分退款` 或 `部分售后中` 标识，剩余商品继续履约。

## 3. 发货商品已发货仅退款场景

适用订单：商品物流发货订单，已发货 / 待收货，消费者未收到货或发生物流异常，不寄回商品直接申请退款。

### 3.1 C 端消费者页面链路

```text
订单列表页｜状态：已发货 / 待收货｜按钮：申请售后｜点击后：进入订单详情或售后类型页
  ↓
订单详情页｜状态：已发货 / 待收货｜底部按钮：申请售后 / 查看物流｜点击申请售后：进入售后类型页
  ↓
售后类型页｜入口：仅退款（未收货仅退款） / 退货退款（已收到货）｜选择仅退款：进入仅退款申请页
  ↓
仅退款申请页｜申请类型：仅退款（未收货仅退款）｜选择：商品、SKU、数量｜填写：原因、金额、说明、凭证｜提交后：创建售后单
  ↓
售后详情页｜状态：待审核｜展示：申请金额、预计退回方式、原订单运费默认不可退提示｜按钮：撤销申请 / 联系卖家
  ↓
售后详情页｜状态：待退款 / 退款中 / 退款成功｜按 B 端退款进度回显
```

### 3.2 B 端商家处理链路

```text
售后管理列表｜状态 Tab：待审核｜售后类型：仅退款｜操作：审核
  ↓
售后详情窗口｜状态：待审核｜展示：未收货原因、物流信息、申请金额、支付构成、凭证｜操作：同意申请 / 拒绝申请 / 转补寄
  ↓
同意申请弹窗｜处理结果：同意仅退款｜填写：审批退款金额、原单运费退款金额、审核备注｜确认后：待退款
  ↓
售后详情窗口｜状态：待退款｜展示：最终审批金额、调额说明、支付退回明细｜操作：操作退款
  ↓
操作退款弹窗｜确认退款｜提交后：退款中
  ↓
售后详情窗口｜状态：已完成 / 退款失败｜按渠道回执更新
```

### 3.3 C/B 状态联动表

| C 端动作 / 页面 | B 端状态 | B 端处理结果 | C 端回显 |
| --- | --- | --- | --- |
| 提交已发货仅退款申请 | 待审核 | 生成仅退款售后单 | `待审核`，可撤销 |
| B 端同意仅退款 | 待退款 | 审批退款金额生效 | `待退款`，展示最终审批金额 |
| B 端调低金额 | 待退款 | 记录调额原因 | 展示调额说明 |
| B 端操作退款 | 退款中 | 发起退款 | `退款中` |
| 退款成功 | 已完成 | 售后完成 | `退款成功` |
| B 端转补寄 | 待补寄出库 | 进入补寄链路 | `补寄处理中` |

### 3.4 设计 / 开发 / 测试关注点

- 已发货售后类型页只展示两项：`仅退款（未收货仅退款）`、`退货退款（已收到货）`。
- 已发货仅退款申请页必须选择商品、SKU、数量，并填写申请退款金额。
- 已发货默认不退原订单运费；金额区需要提示 `商品已发货，原订单运费默认不可退`。
- 申请原因展示未收货/物流异常类，例如一直未收到货、物流长时间未更新、包裹丢失、显示签收但未收到、少件/漏发。
- 质量、少件、拒收、包裹异常等原因可引导上传凭证。

## 4. 发货商品退货退款场景

适用订单：商品物流发货订单，消费者已收到货，需要退回商品后退款。

### 4.1 C 端消费者页面链路

```text
订单列表页｜状态：已发货 / 待收货 / 已完成且售后期内｜按钮：申请售后｜点击后：进入订单详情或售后类型页
  ↓
订单详情页｜状态：已发货 / 已完成｜底部按钮：申请售后 / 查看物流｜点击申请售后：进入售后类型页
  ↓
售后类型页｜入口：退货退款（已收到货） / 仅退款（未收货仅退款）｜选择退货退款：进入退货退款申请页
  ↓
退货退款申请页｜选择：退货商品、SKU、数量｜填写：原因、申请退款金额、说明、凭证、联系电话｜提交后：创建售后单
  ↓
售后详情页｜状态：待审核｜展示：申请信息、预计退回方式｜按钮：撤销申请 / 联系卖家
  ↓
售后详情页｜状态：待寄回｜展示：退货地址、寄回截止时间、寄回提示｜按钮：填写物流 / 撤销申请
  ↓
填写退货物流页｜填写：物流公司、物流单号、联系电话、备注｜提交后：进入待收货
  ↓
售后详情页｜状态：待收货｜展示：买家退货物流、等待商家确认收货｜按钮：查看退货物流 / 联系卖家
  ↓
售后详情页｜状态：待退款 / 退款中｜展示：质检结果、最终审批金额、实际退回方式｜按钮：联系卖家
  ↓
售后详情页｜状态：退款成功｜展示：退款金额、处理结果卡片、售后信息｜按钮：查看结果
```

### 4.2 B 端商家处理链路

```text
售后管理列表｜状态 Tab：待审核｜售后类型：退货退款｜操作：审核
  ↓
售后详情窗口｜状态：待审核｜展示：申请商品、原因、凭证、申请金额、支付构成｜操作：同意申请 / 拒绝申请 / 转补寄
  ↓
同意申请弹窗｜处理结果：同意退货退款｜填写：审批金额、退货地址、寄回截止时间、退货责任方、寄回邮费责任方｜确认后：待寄回
  ↓
售后详情窗口｜状态：待寄回｜展示：审核结果、退货地址、寄回截止时间｜等待：买家填写物流
  ↓
售后详情窗口｜状态：待收货｜展示：买家退货物流｜操作：签收并录入质检
  ↓
签收质检弹窗｜填写：签收数量、完好入库数量、破损数量、缺失数量、污染/影响二销数量、异常说明、图片凭证｜确认后：待退款
  ↓
售后详情窗口｜状态：待退款｜展示：质检结果、退货入库 / 异常记录、最终审批金额｜操作：操作退款
  ↓
操作退款弹窗｜确认退款金额和方式｜提交后：退款中
  ↓
售后详情窗口｜状态：已完成 / 退款失败｜按渠道回执更新
```

### 4.3 C/B 状态联动表

| C 端动作 / 页面 | B 端状态 | B 端处理结果 | C 端回显 |
| --- | --- | --- | --- |
| 提交退货退款申请 | 待审核 | 生成退货退款售后单 | `待审核` |
| B 端同意退货退款 | 待寄回 | 生成退货地址和寄回截止时间 | `待寄回`，可填写物流 |
| 用户填写退货物流 | 待收货 | B 端看到买家退货物流 | `待收货` |
| B 端签收质检 | 待退款 | 完好入库 / 异常记录完成 | `待退款`，展示质检与调额结果 |
| B 端操作退款 | 退款中 | 发起退款 | `退款中` |
| 退款成功 | 已完成 | 售后完成 | `退款成功` |

### 4.4 设计 / 开发 / 测试关注点

- 退货退款申请页必须选择商品、SKU、数量；全额或部分由选择范围自动判定。
- 已发货退货退款默认不退原订单运费，申请页必须展示提示。
- B 端审核时决定退货地址、寄回截止时间、责任方、寄回邮费责任方。
- 待寄回时 C 端展示退货地址和截止时间；待收货时 C 端展示买家退货物流。
- 质检后才能进入待退款；不可在买家填写物流后直接退款完成。
- 完好可二销数量生成退货入库并回补库存；破损、缺失、污染、空包等只记录异常，不回补库存。
- C 端只展示消费者可理解的质检结论和调额说明，不展示内部责任归因。

## 5. 商家拒绝申请场景

适用订单：发货商品仅退款、退货退款在 `待审核` 阶段被 B 端拒绝。

### 5.1 C 端消费者页面链路

```text
售后详情页｜状态：待审核｜按钮：撤销申请 / 联系卖家｜等待：商家审核
  ↓
售后详情页｜状态：已关闭｜关闭类型：已拒绝｜展示：拒绝原因、拒绝说明、关闭时间｜按钮：重新申请 / 联系卖家
```

### 5.2 B 端商家处理链路

```text
售后管理列表｜状态 Tab：待审核｜操作：审核
  ↓
售后详情窗口｜状态：待审核｜操作：拒绝申请
  ↓
拒绝申请弹窗｜填写：拒绝原因、拒绝说明（C 端可见）｜确认后：已关闭
  ↓
售后详情窗口｜状态：已关闭｜关闭类型：已拒绝｜展示：关闭信息、操作日志
```

### 5.3 C/B 状态联动表

| C 端动作 / 页面 | B 端状态 | B 端处理结果 | C 端回显 |
| --- | --- | --- | --- |
| 用户提交售后申请 | 待审核 | 等待商家审核 | `待审核` |
| B 端拒绝申请 | 已关闭 | 关闭类型：已拒绝 | `已关闭`，展示拒绝原因 |
| 用户重新申请 | 新售后单待审核 | 生成新的售后单号 | 新售后详情显示 `待审核` |

### 5.4 设计 / 开发 / 测试关注点

- `已拒绝` 是关闭类型，不是售后主状态；售后主状态为 `已关闭`。
- C 端必须展示拒绝原因和拒绝说明。
- 如果仍在售后期且仍有可售后范围，C 端展示 `重新申请`；超过售后期则隐藏。
- 重新申请必须生成新的售后单号，不复用已关闭售后单。

## 6. 用户撤销与超时关闭场景

适用状态：待审核、待寄回。待收货后原则上不允许消费者自行撤销。

### 6.1 C 端消费者页面链路

```text
售后详情页｜状态：待审核｜按钮：撤销申请 / 联系卖家｜点击撤销：打开撤销确认弹窗
  ↓
撤销确认弹窗｜按钮：取消 / 确认撤销｜确认后：关闭售后
  ↓
售后详情页｜状态：已关闭｜关闭类型：已撤销｜展示：撤销时间、关闭来源｜按钮：重新申请
```

```text
售后详情页｜状态：待寄回｜展示：寄回截止时间｜超时未填写物流：系统关闭
  ↓
售后详情页｜状态：已关闭｜关闭类型：超时关闭｜展示：关闭原因、截止时间｜按钮：重新申请 / 联系卖家
```

### 6.2 B 端商家处理链路

```text
售后管理列表｜状态：待审核 / 待寄回｜等待：用户动作或系统超时
  ↓
用户撤销 / 系统超时关闭
  ↓
售后详情窗口｜状态：已关闭｜关闭类型：已撤销 / 超时关闭｜展示：关闭来源、关闭时间、操作日志
```

### 6.3 C/B 状态联动表

| 触发动作 | B 端状态 | B 端处理结果 | C 端回显 |
| --- | --- | --- | --- |
| 待审核用户撤销 | 已关闭 | 关闭类型：已撤销 | `已关闭`，展示用户主动撤销 |
| 待寄回用户撤销 | 已关闭 | 关闭类型：已撤销 | `已关闭`，展示用户主动撤销 |
| 待寄回超时未填写物流 | 已关闭 | 关闭类型：超时关闭 | `已关闭`，展示超时原因 |
| 待收货后用户尝试撤销 | 不允许自行撤销 | 需商家异常处理 | C 端不展示撤销入口 |

### 6.4 设计 / 开发 / 测试关注点

- 待审核、待寄回可撤销；待收货、待退款、退款中、退款失败不可由消费者自行撤销。
- 超时关闭必须记录寄回截止时间和关闭原因。
- 已撤销或超时关闭后，若仍满足售后期和可售后范围，可重新申请。
- 已关闭主状态下通过关闭类型区分 `已撤销`、`超时关闭`、`已拒绝`、`人工关闭`。

## 7. 退款失败与重新退款场景

适用订单：任意进入退款执行的售后单，包括非物流退款、仅退款、退货退款。

### 7.1 C 端消费者页面链路

```text
售后详情页｜状态：退款中｜展示：商家已发起退款、实际退回方式｜按钮：联系卖家
  ↓
售后详情页｜状态：退款失败｜展示：失败原因、失败时间、等待商家重新处理｜按钮：联系卖家
  ↓
售后详情页｜状态：退款中｜B 端重新操作退款后回到处理中｜按钮：联系卖家
  ↓
售后详情页｜状态：退款成功｜退款成功后展示最终结果｜按钮：查看结果
```

### 7.2 B 端商家处理链路

```text
售后详情窗口｜状态：退款中｜等待：渠道回执
  ↓
售后详情窗口｜状态：退款失败｜展示：失败原因、渠道返回码、失败时间｜操作：重新操作退款
  ↓
重新操作退款弹窗｜展示：上次失败原因、退款金额｜填写：重试说明｜确认后：退款中
  ↓
售后详情窗口｜状态：已完成｜渠道退款成功后完成
```

### 7.3 C/B 状态联动表

| 触发动作 | B 端状态 | B 端处理结果 | C 端回显 |
| --- | --- | --- | --- |
| 财务发起退款 | 退款中 | 等待渠道回执 | `退款中` |
| 渠道返回失败 | 退款失败 | 记录失败原因和返回码 | `退款失败` |
| 财务重新操作退款 | 退款中 | 重新发起退款 | `退款中` |
| 重新退款成功 | 已完成 | 售后完成 | `退款成功` |

### 7.4 设计 / 开发 / 测试关注点

- 退款失败不等于拒绝申请，不进入 `已关闭 / 已拒绝`。
- 账户余额不足等失败原因由 B 端财务处理，C 端只展示消费者可理解的失败说明。
- 退款失败状态下 C 端不展示撤销入口。
- 重新操作退款必须记录重试说明和操作日志。

## 8. 商家转补寄场景

适用订单：发货商品仅退款或退货退款在 B 端审核时判断为少件、漏发、错发等适合补寄的场景。

### 8.1 C 端消费者页面链路

```text
售后详情页｜状态：待审核｜展示：申请已提交，等待商家处理｜按钮：撤销申请 / 联系卖家
  ↓
售后详情页｜状态：补寄处理中｜展示：商家已为您转补寄、补寄商品、补寄原因、等待仓库出库｜按钮：联系卖家
  ↓
售后详情页｜状态：待买家收货｜展示：补寄商品已发出、快递公司、物流单号、物流轨迹入口｜按钮：查看补寄物流 / 确认收货
  ↓
售后详情页｜状态：售后完成｜展示：补寄完成、签收时间、补寄商品｜按钮：查看结果
```

### 8.2 B 端商家处理链路

```text
售后管理列表｜状态 Tab：待审核｜操作：审核
  ↓
售后详情窗口｜状态：待审核｜操作：转补寄处理
  ↓
转补寄处理弹窗｜填写：补寄原因、补寄责任方、补寄商品、数量、补寄说明｜确认后：待补寄出库
  ↓
售后详情窗口｜状态：待补寄出库｜展示：补寄处理结果、出库状态｜操作：创建补寄出库申请
  ↓
创建补寄出库申请弹窗｜填写：出库仓 / 云仓、补寄商品、数量、补寄运费承担方、给云仓说明｜确认后：等待云仓发货
  ↓
售后详情窗口｜状态：待买家收货｜云仓回传物流后展示：快递公司、物流单号、发货时间
  ↓
售后详情窗口｜状态：已完成｜买家确认收货或物流签收后完成
```

### 8.3 C/B 状态联动表

| C 端动作 / 页面 | B 端状态 | B 端处理结果 | C 端回显 |
| --- | --- | --- | --- |
| 用户提交售后申请 | 待审核 | 等待商家审核 | `待审核` |
| B 端转补寄 | 待补寄出库 | 记录补寄原因和补寄商品 | `补寄处理中` |
| B 端创建补寄出库申请 | 待补寄出库 | 推送云仓，等待发货 | 仍显示 `补寄处理中`，不展示物流单号 |
| 云仓发货回传物流 | 待买家收货 | 记录快递公司、物流单号、发货时间 | `待买家收货`，展示补寄物流 |
| 买家确认收货 / 物流签收 | 已完成 | 补寄完成 | `售后完成` |

### 8.4 设计 / 开发 / 测试关注点

- C 端不展示 `申请补寄` 入口，补寄只来自 B 端处理结果。
- 云仓未发货前，C/B 端都不展示快递公司、物流单号、物流轨迹。
- 创建补寄出库申请时不填写快递公司和物流单号，只填写出库仓、补寄商品、数量、运费承担方和给云仓说明。
- 补寄通常不退款，但 B 端需保留补寄成本、补寄出库申请、云仓发货结果和操作日志。
- C 端只展示补寄商品、补寄进度和补寄物流，不展示补寄责任方、补寄运费承担方等后台字段。

## 10. 后续 HTML 面板产出建议

后续产出 HTML 时，建议按本文场景顺序生成页面：

1. 每个业务场景一个大面板。
2. 大面板内分为 `C 端消费者流程`、`B 端商家处理流程`、`C/B 状态联动` 三个区块。
3. C 端流程使用手机页面卡片；B 端流程使用后台页面 / 弹窗卡片。
4. 每张页面卡片只展示真实产品内容，不放开发说明、测试说明或 PRD 摘要。
5. 页面补充说明放在 HTML 页面外的设计交付说明中，不进入真实产品界面。
