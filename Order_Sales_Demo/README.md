# 商家自营售后系统

商家自营售后系统是面向美业商家自营订单的售后业务前端工程规划，覆盖 C 端消费者售后申请与 B 端商家售后处理两条链路。项目目标是把订单售后入口、售后类型选择、退款/退货申请、商家审核、退货收货、退款执行、补寄处理、状态回显和售后结果展示串联成一套可演示、可评审、可继续研发落地的完整系统。

当前仓库主要包含产品需求文档、设计规范和 C/B 端参考图。后续前端工程可基于 Vue 3、Vite、TypeScript、Element Plus 和 Pinia 搭建，并可预留 `server/` Express 服务骨架，用于本地联调或扩展后端接口。

## 技术栈

- Vue 3 + TypeScript + Vite
- Element Plus + Tailwind CSS
- Pinia + Vue Router
- Axios 请求层
- Storybook 组件调试
- GitLab CI + Docker + Serverless Devs 部署流程

## 业务范围

系统围绕商家自营订单售后闭环设计，重点覆盖以下场景：

- C 端消费者从订单列表、订单详情进入售后申请。
- C 端按订单状态和订单类型选择正确的售后类型。
- C 端提交仅退款、退货退款等售后申请，并查看售后进度。
- B 端商家查看售后单、审核售后、拒绝或同意申请。
- B 端处理退货收货、质检、退款、补寄和异常记录。
- 系统同步退款金额、优惠资产回退、分账回退、责任归因和状态流转。
- C 端根据 B 端处理结果展示待审核、待寄回、待退款、退款成功、申请驳回等状态。

## Demo 输出边界

第一版可点击 Demo 只输出真实产品中的 C 端消费者页面和 B 端商家后台页面，不输出开发、测试、产品评审专用的说明型页面。

C 端只展示消费者在真实使用中需要看到和操作的内容，例如我的订单、订单详情、售后类型选择、售后申请、售后详情、退款进度、退货物流填写、退款成功、申请驳回和售后关闭等。

B 端只展示商家后台真实处理售后需要看到和操作的内容，例如售后列表、售后详情、订单与商品信息、用户申请原因和凭证、审核处理、退款金额调整、责任归因、退货收货/质检、补寄处理、退款处理和操作日志等。

Demo 不应出现以下内容：

- 开发说明页
- 测试用例页
- PRD 摘要页
- 状态机说明图
- 字段字典页
- 技术架构页
- Demo 使用说明页
- 面向内部评审人员的流程解释卡片

开发和测试需要理解的规则，应通过真实页面状态、按钮显隐、弹窗字段、金额拆分、处理结果和 C/B 端状态联动自然体现，而不是以额外说明页的方式呈现。


## 本地开发

安装依赖：

```bash
npm install
```

启动开发环境：

```bash
npm run dev
```

启动 Storybook：

```bash
npm run storybook
```

构建生产包：

```bash
npm run build
```

本地预览构建结果：

```bash
npm run preview
```

## 核心模块规划

| 模块 | 说明 |
| --- | --- |
| C 端订单售后入口 | 在订单列表、订单详情中展示申请退款、申请售后、售后详情等入口 |
| C 端售后申请 | 根据订单类型、商品状态、履约状态展示可申请的售后类型 |
| C 端售后详情 | 展示审核进度、退款金额、寄回物流、处理结果和失败原因 |
| B 端售后列表 | 支持按状态、订单号、售后单号、申请时间、售后类型筛选售后单 |
| B 端售后详情 | 展示订单、商品、金额、凭证、责任归因、退款与分账信息 |
| 审核处理 | 支持同意、拒绝、调额、转补寄、确认收货、退款等操作 |
| 退款与资产回退 | 处理实付金额、优惠券、积分、余额、赠品、分账等回退逻辑 |
| 状态流转 | 维护 C 端与 B 端一致的售后状态机和操作按钮 |

## 设计规范

项目视觉规范参考 `设计规范/设计规范.md`：

- B 端后台强调高效、清晰、稳定和高密度信息处理。
- C 端商城强调精致、温和、可信和流畅消费体验。
- 双端需要保持业务状态、权益口径、时间口径和金额口径一致。

## PRD 文档

核心需求文档位于：

该文档覆盖 C 端消费者申请售后、B 端商家处理售后、仓库验收、退款执行、补寄、财务处理和 C 端结果回显等端到端流程。

## 部署流程规划

项目可按以下流程部署：

1. GitLab CI 执行代码检查、类型检查和生产构建。
2. Docker 构建前端镜像或静态资源包。
3. Serverless Devs 发布到目标环境。
4. 发布后通过预览地址验证 C 端与 B 端核心流程。
