## 1. 项目上下文包

### 1.1 业务背景

商家自营订单售后目前存在C端申请、B端审核处理、仓库验收、退款执行、资产回退、分账冲正和C端结果展示割裂的问题。原有能力更像若干退款能力拼接，无法稳定支撑发货商品、非物流退款型订单、分账订单、部分售后、补寄、退货入库、异常记录、财务对账等完整售后场景。

本需求不是新增一个孤立退款功能，而是将商家自营订单已有的普通退款、局部退货退款能力，升级为一套覆盖C端消费者真实操作路径和B端商家处理路径的标准售后体系。

当前不做该需求的影响：

1. 产品、研发、测试需要继续跨多份文档拼接流程，容易产生状态和金额口径偏差。
2. 非物流订单可能被误套进退货寄回链路。
3. 发货商品订单的仅退款、退货退款、部分售后、补寄、仓库验收难以形成统一状态机。
4. 商品退款金额、原单运费、寄回邮费补贴、资产退回、分账回退可能混算，导致财务对账困难。
5. C端用户无法清晰理解售后进度，客服咨询量上升。

### 1.2 用户角色

| 角色 | 目标 | 典型操作 | 关注点 |
| --- | --- | --- | --- |
| C端消费者 | 基于订单真实状态发起退款/售后并查看进度 | 订单列表/详情发起申请，填写金额/原因/物流，查看退款或补寄结果 | 哪些订单可申请、能退多少钱、进度到哪、为什么被调额或拒绝 |
| 商家客服 | 审核消费者售后诉求并选择退款、拒绝或补寄 | 查看售后详情、同意申请、拒绝、转补寄、调额、调责 | 商品、金额、责任、凭证、分账状态、处理留痕 |
| 仓库/门店/云仓 | 对退货或补寄履约进行实物处理 | 签收质检、退货入库、异常记录、补寄出库、回传物流 | 可二销数量、异常数量、库存回补、补寄物流 |
| 财务/测算 | 执行退款并沉淀财务、资产、分账数据 | 操作退款、重试退款、查看分账回退、记录失败明细 | 退款金额、资产回退、水单、分账冲正、合伙人佣金 |
| 运营/管理者 | 评估售后效率、责任成本和经营结果 | 查看统计、筛选门店、导出或分析售后数据 | 责任归因、退款原因、门店表现、报表口径 |
| 研发 | 实现页面、接口、状态、字段、权限和日志 | 拆分前后端任务、设计状态机和数据模型 | 边界完整、状态明确、接口可落地、幂等和异常可处理 |
| 测试 | 生成主流程、异常流、边界流和回归用例 | 编写用例、准备Mock数据、验证端到端流程 | 覆盖订单类型、状态、金额、权限、异常、数据一致性 |

### 1.3 核心流程

入口：

1. C端订单列表。
2. C端订单详情。
3. B端商家后台售后管理列表。

主路径：

```text
C端订单列表/订单详情
  -> 判断订单业务形态和订单状态
    -> 发货商品待发货：进入仅退款申请页
    -> 发货商品已发货/已收货且售后期内：进入售后类型页
    -> 非物流退款型订单：弹普通退款确认弹窗
      -> 创建售后单
        -> B端售后列表/详情处理
          -> 审核/质检/退款/补寄/关闭
            -> C端售后详情按状态和结果卡片回显
```

关键节点：

1. 订单入口准入判断。
2. C端售后类型、申请商品/SKU/数量、申请退款金额、原因和凭证。
3. B端待审核、待寄回、待收货、待退款、退款失败、待补寄出库等状态处理。
4. 仓库签收质检、退货入库、异常记录。
5. 财务退款、资产回退、水单生成、分账回退记录。
6. C端结果卡片、按钮和金额/物流/关闭原因回显。

结束状态：

1. 已完成：退款完成或补寄完成。
2. 已关闭：已拒绝、已撤销、超时关闭、人工关闭。

### 1.4 业务术语

| 术语 | 定义 | 备注 |
| --- | --- | --- |
| 商家自营订单 | 商家自主履约、售后由商家侧承接的线上订单 | 与供应链/平台后台联动订单区分 |
| 发货商品订单 | 需要物流履约的商品订单 | 支持仅退款、退货退款、补寄 |
| 非物流退款型订单 | 自提、套餐/次卡、开卡充值、预定金等不需要退货寄回的订单 | 只支持整单全额退款，进入待退款 |
| 仅退款 | 不要求消费者寄回商品的退款诉求 | 待发货和已发货未收货均可存在 |
| 退款退货 | 消费者寄回商品，仓库/门店验收后退款 | 仅适用于发货商品订单 |
| 补寄 | B端审核后选择以补发商品替代退款或退货 | C端不主动申请 |
| 售后主状态 | 售后单进度状态，如待审核、待寄回、待退款、退款失败等 | 不覆盖原订单履约状态 |
| 订单售后标识 | 原订单上追加的逆向进度标签，如售后中、部分退款、补寄中 | 用于展示和防重复申请 |
| 商品行级售后状态 | 商品/SKU/数量维度的售后锁定和完成状态 | 用于多次售后和部分售后 |
| 原单运费退款金额 | 正向订单运费是否退回及退回多少 | 不与商品退款混算 |
| 寄回邮费补贴/赔付 | 消费者寄回商品产生的邮费补贴 | 不并入商品退款 |
| 退款水单 | 退款成功后新增的业务水单 | 不删除或覆盖原购买水单 |
| 分账回退失败记录 | 分账冲正失败后沉淀的数据库明细 | 本期不做前台追回处理 |

### 1.5 系统边界

本需求包含：

1. C端订单列表、订单详情、售后类型页、申请页、填写物流页、售后详情页、结果页。
2. B端售后列表、状态筛选、售后详情、中央弹窗、处理结果卡片、退款清分账单、日志。
3. 售后状态机、订单售后标识、商品行级售后状态。
4. 退款金额、原单运费、寄回邮费补贴、资产退回、优惠券状态、满赠赠品、预定金权益回退。
5. 仓库签收质检、退货入库、异常不入库、补寄出库和云仓物流回传。
6. 分账回退成功/失败数据库记录，合伙人佣金回退和钱包负余额扣减记录。
7. 权限、风控校验、操作日志、Mock数据和测试验收。

本需求不包含：

1. 国际物流、跨境税费逆向处理。
2. 自动向物流或外部供应商追偿。
3. 复杂残次品定价、维修、报废审批流。
4. 残次库存和破损出库单。
5. 分账回退失败前台报表、待追回列表、导出入口、处理人分配、追回处理功能。

依赖系统 / 页面 / 接口：

1. 订单系统：订单状态、订单商品行、支付快照、售后标识。
2. C端商城/小程序：订单列表、订单详情、售后申请、售后详情。
3. 商家后台：售后列表、售后详情、中央弹窗、权限、日志。
4. 支付退款系统：退款发起、退款回调、退款失败重试。
5. 会员资产系统：卡金、赠送金、积分、套餐/次数、储值资产回退。
6. 优惠券和活动系统：优惠券退回、满赠赠品回收、预定金权益回退。
7. 库存/仓库/云仓系统：退货入库、异常记录、补寄出库、物流回传。
8. 分账/财务系统：分账冲正、回退失败记录、合伙人佣金处理。

复用约束：

1. C端必须复用现有订单列表和订单详情真实入口，不新建脱离订单的售后入口。
2. 非物流退款型订单复用当前线上普通退款确认弹窗。
3. B端复杂操作统一使用中央弹窗，不使用底部抽屉承接处理。
4. 原订单履约主状态保留，不被售后主状态直接覆盖。

## 3. 业务结果说明

### 3.1 为什么做

本需求需要将商家自营订单售后从分散能力统一为端到端标准流程，使消费者、客服、仓库、财务、研发和测试使用同一套流程、状态和金额语言。尤其要解决发货商品完整售后、非物流退款型订单轻量退款、分账订单售后、仓库验收入库、补寄、资产水单、分账回退记录等关键缺口。

### 3.2 希望改变的指标或结果

| 指标 / 结果 | 当前状态 | 目标状态 | 验证方式 |
| --- | --- | --- | --- |
| 售后处理时长 | 申请、审核、退款、仓库处理割裂 | 缩短从申请到审核、退款、完成的整体时间 | 售后状态耗时统计 |
| 退款准确率 | 商品退款、运费、资产、分账口径易混 | 商品退款、原单运费、寄回邮费、资产退回分开记录 | 财务对账、测试金额用例 |
| 客服咨询量 | C端进度和金额解释不足 | C端通过状态和结果卡片看懂进度与结果 | 客服咨询原因统计 |
| 可解释性 | 拒绝、调额、关闭原因不完整 | 每次审核、调额、拒绝、关闭均有原因和展示 | 操作日志和C端回显检查 |
| 财务对账准确率 | 分账和资产回退明细不足 | 退款、水单、分账回退成功/失败可追溯 | 财务抽样对账 |
| 仓储处理准确率 | 退货是否可二销、是否入库不标准 | 完好可二销入库，异常只记录不入库 | 仓库质检/库存记录核对 |
| 分账回退失败记录完整率 | 失败明细沉淀不足 | 普通失败只入库记录，合伙人佣金按特殊规则处理 | 数据库明细检查 |
| 测试覆盖完整度 | 跨文档拆用例困难 | 单文档可拆主流程、异常流、边界流 | 测试用例评审 |

### 3.3 当前问题

1. C/B端流程割裂，状态和回显口径不一致。
2. 订单业务形态边界不清，非物流订单容易误进寄回链路。
3. 已分账物流发货订单缺少完整售后处理。
4. 旧开关语义过窄，不适合本期标准售后能力。
5. 补寄、责任归因、运费承担、调额、仓库验收和库存回补未标准化。
6. 分账回退失败和合伙人佣金回退缺少完整数据沉淀。
7. C端售后详情只展示当前状态时，消费者难以理解已发生处理结果。

### 3.4 如何验证

业务验证：

1. 产品评审能从订单列表一路讲到退款/补寄完成。
2. 运营、客服、仓库、财务能确认各自处理节点和异常边界。

用户验证：

1. C端可明确知道是否可申请、能退多少、当前状态、下一步动作和结果原因。
2. C端不暴露分账、云仓内部字段、合伙人佣金等后台概念。

数据验证：

1. 退款金额、资产退回、水单、分账回退、质检入库记录可追溯。
2. 分账回退失败只做数据库记录，不影响C端售后完成。

流程验证：

1. 测试能覆盖仅退款、退货退款、补寄、退款失败、拒绝、撤销、超时关闭、质检异常。
2. AI只读本文可拆出C端、B端、后端、数据、测试任务。

## 4. 需求分类判断

### 4.1 分类结论

- [x] 0-1 / 复杂交互需求
- [x] 有耦合需求
- [ ] 智能体需求
- [x] 合规 / 风险敏感需求
- [x] 数据 / 分析需求

### 4.2 判断依据

1. 复杂交互：涉及C端多入口、多订单形态、多售后类型、多状态详情和B端标准工作台。
2. 有耦合：影响订单、退款、资产、优惠券、库存、云仓、分账、财务、权限、日志和报表口径。
3. 非智能体：本需求不涉及AI自动决策、模型输出或自动代理操作。
4. 合规/风险敏感：涉及支付退款、用户资产、分账、库存、审计留痕。
5. 数据/分析：涉及退款指标、责任统计、仓库统计、分账回退失败、合伙人佣金和水单分析。

## 5. AI-Native PRD 主体

### 5.1 需求目标

用户目标：

1. C端消费者可以从真实订单路径发起正确的退款/售后申请。
2. 消费者可以看到申请金额、实际退款金额、退款方式、物流、补寄、拒绝或关闭原因。
3. 消费者不需要理解后台分账、云仓、合伙人佣金等内部概念。

业务目标：

1. 建立商家自营订单统一售后体系。
2. 按订单业务形态区分发货商品完整售后和非物流轻量退款。
3. 沉淀退款、资产、分账、仓库、补寄、责任和日志数据。

产品目标：

1. C端入口和详情清晰。
2. B端从退款处理升级为标准售后工作台。
3. 售后主状态、订单售后标识、商品行级状态统一。

技术目标：

1. 支撑多次售后、部分售后、商品/SKU/数量维度锁定。
2. 支撑退款、资产、优惠券、库存、分账、云仓等系统幂等联动。
3. 支撑状态机、日志、权限和异常补偿。

### 5.2 范围边界

| 类型 | 内容 |
| --- | --- |
| 本期做 | 发货商品订单仅退款/退货退款/补寄；非物流退款型订单轻量退款；C端入口和回显；B端售后工作台；金额/资产/水单；仓库质检入库；分账回退明细；合伙人佣金特殊回退；权限日志；Mock与验收 |
| 本期不做 | 分账回退失败前台报表、待追回列表、失败导出、处理人分配、自动重试、线下扣减、人工收款、坏账标记；国际物流；跨境税费；复杂残次品定价/维修/报废审批 |
| 后续可能做 | 分账失败追回处理、失败报表、导出、自动化追偿、残次库存、破损出库单、更细的财务分析 |
| 明确排除 | 用旧开关继续限制商品物流发货订单售后；C端主动申请补寄；非物流订单进入寄回/质检/补寄链路 |

### 5.3 业务场景

| 场景编号 | 场景名称 | 触发条件 | 参与角色 | 期望结果 |
| --- | --- | --- | --- | --- |
| S1 | 发货商品待发货整单仅退款 | 已付款未发货，选择全部剩余商品和数量 | C端消费者、商家客服/财务 | 可退商品金额和原订单运费，进入待审核或待退款后退款完成 |
| S2 | 发货商品待发货部分仅退款 | 已付款未发货，只选择部分商品/SKU/数量 | C端消费者、商家客服/财务 | 只退对应商品分摊金额，剩余商品继续履约 |
| S3 | 发货商品已发货仅退款 | 已发货/待收货，消费者未收到货 | C端消费者、商家客服、财务 | C端填写金额/原因/凭证，B端审核后退款或拒绝/补寄 |
| S4 | 发货商品退货退款 | 已收到货且售后期内 | C端消费者、商家客服、仓库、财务 | 审核通过、寄回、质检、入库/异常、退款完成 |
| S5 | 少件/漏发/错发转补寄 | B端审核认为适合补寄 | 商家客服、仓库/云仓、C端消费者 | 创建补寄出库申请，云仓发货后C端查看补寄物流，最终完成 |
| S6 | 非物流退款型订单 | 自提未自提、套餐未核销、资产满足回退、预定金未使用 | C端消费者、财务 | C端确认弹窗提交，直接进入B端待退款，不填写原因和凭证 |
| S7 | 退款失败重试 | 支付渠道或商家资金异常导致退款失败 | 财务、系统、C端消费者 | 状态进入退款失败，财务排除异常后重新退款 |
| S8 | 售后关闭 | 商家拒绝、消费者撤销、超时未寄回、人工关闭 | C端消费者、商家客服、管理员、系统 | 记录关闭类型和原因，C端展示可理解说明 |

### 5.4 主流程

| 步骤 | 用户 / 系统动作 | 页面 / 位置 | 输入 | 系统处理 | 输出 / 状态 |
| --- | --- | --- | --- | --- | --- |
| 1 | 系统判断订单是否可售后 | C端订单列表/详情 | 订单业务形态、订单状态、售后期、资产状态 | 计算入口按钮 | 申请退款/申请售后/售后详情/不可申请 |
| 2 | 用户发起申请 | C端入口 | 点击按钮 | 按订单形态路由 | 进入申请页、售后类型页或确认弹窗 |
| 3 | 用户提交售后 | C端申请页/弹窗 | 商品、SKU、数量、金额、原因、凭证或确认动作 | 创建售后单，锁定商品行可售后范围 | 待审核或待退款 |
| 4 | B端处理申请 | 售后列表/详情 | 售后单、申请信息、金额、凭证 | 同意、拒绝、转补寄、调额、调责 | 待寄回、待退款、已关闭、待补寄出库 |
| 5 | 仓库处理退货 | B端待收货/质检弹窗 | 退货物流、签收数量、质检结果 | 生成退货入库或异常记录 | 待退款或人工复核 |
| 6 | 财务操作退款 | B端待退款/退款失败 | 审批金额、退款方式、资产明细 | 发起退款，处理回调 | 退款中、已完成、退款失败 |
| 7 | 系统处理资产和分账 | 后端任务/回调 | 退款成功结果 | 生成退款水单、回退资产、触发分账回退 | 明细入库，售后完成 |
| 8 | C端查看进度 | 售后详情 | 售后单状态和结果卡片 | 展示当前状态、已发生结果、按钮 | 用户理解下一步或最终结果 |

### 5.5 分支流程

| 分支编号 | 触发条件 | 分支路径 | 回归主流程方式 | 结果状态 |
| --- | --- | --- | --- | --- |
| B1 | 待发货整单仅退款 | 选择全部剩余商品和数量，最多可退金额包含原订单运费 | 审核或自动进入待退款 | 已完成/退款失败 |
| B2 | 待发货部分仅退款 | 选择部分商品/SKU/数量，原订单运费默认不退 | 剩余商品继续待发货履约 | 部分退款 |
| B3 | 已发货仅退款 | 选择仅退款（未收货仅退款），不寄回 | B端审核后退款、拒绝或转补寄 | 待退款/已关闭/待补寄出库 |
| B4 | 退货退款 | 选择退货退款，审核通过后寄回 | 仓库质检完成后进入待退款 | 已完成/退款失败 |
| B5 | 转补寄 | B端判断少件、漏发、错发适合补寄 | 创建补寄出库申请，云仓发货回传 | 待买家收货/已完成 |
| B6 | 非物流退款 | C端确认弹窗提交 | 系统默认原因不想要了，直接待退款 | 已完成/退款失败 |
| B7 | 消费者撤销 | 待审核或待寄回允许撤销 | 关闭售后，释放可售后范围 | 已关闭 |
| B8 | 超时未寄回 | 待寄回超过截止时间未填物流 | 系统自动关闭 | 已关闭 |
| B9 | 退款失败 | 退款调用失败或商家账户余额不足 | 财务重试或管理员人工关闭 | 退款中/已关闭 |
| B10 | 分账回退失败 | 消费者退款成功后普通分账方回退失败 | 只生成数据库明细，不影响C端完成 | 售后已完成，分账失败已记录 |

### 5.6 状态规则

| 状态 | 进入条件 | 可执行动作 | 下一状态 | 退出条件 | 备注 |
| --- | --- | --- | --- | --- | --- |
| 待审核 | 发货商品订单创建售后单 | 同意仅退款、同意退货退款、拒绝、转补寄、消费者撤销 | 待退款、待寄回、已关闭、待补寄出库 | B端处理完成或消费者撤销 | 非物流订单不进入待审核 |
| 待寄回 | B端同意退货退款 | C端填写物流、撤销、系统超时关闭 | 待收货、已关闭 | 买家填物流或超时/撤销 | 展示退货地址和截止时间 |
| 待收货 | 买家填写退货物流 | 仓库/门店签收质检 | 待退款、人工复核 | 质检完成 | 消费者原则上不可自行撤销 |
| 待退款 | 审核同意、质检完成、非物流订单提交 | 财务操作退款 | 退款中 | 发起退款 | 可保留调额能力，必须留痕 |
| 退款中 | 财务发起退款 | 等待渠道回调 | 已完成、退款失败 | 支付回调 | 不允许消费者撤销 |
| 退款失败 | 退款渠道失败或资金不足 | 重新操作退款、人工关闭 | 退款中、已关闭 | 重试或关闭 | 账户余额不足不应改为已拒绝 |
| 待补寄出库 | B端转补寄处理 | 创建/编辑补寄出库申请、推送云仓 | 待补寄出库、待买家收货 | 云仓发货回传 | 未发货前不展示物流单号 |
| 待买家收货 | 云仓发货回传物流 | 查看补寄物流、确认收货、系统签收 | 已完成 | 买家确认或物流签收 | C端展示补寄物流 |
| 已完成 | 退款成功或补寄完成 | 查看结果 | 终态 | 无 | 不可重复退款 |
| 已关闭 | 拒绝、撤销、超时、人工关闭 | 重新申请、查看原因 | 终态或新售后单 | 满足条件重新申请需新单号 | 关闭类型不是主状态 |

关闭类型：

| 关闭类型 | 触发来源 | C端展示重点 |
| --- | --- | --- |
| 已拒绝 | 商家拒绝申请 | 拒绝原因、拒绝说明、是否可重新申请 |
| 已撤销 | 消费者主动撤销 | 撤销时间、撤销来源、是否可重新申请 |
| 超时关闭 | 待寄回超时未填写物流 | 截止时间、关闭原因 |
| 人工关闭 | 财务/管理员关闭异常售后 | 关闭原因、审批说明 |

原订单联动：

1. 售后状态不直接覆盖原订单履约状态。
2. 创建售后单后，原订单增加售后中或部分售后中标识。
3. 售后涉及商品、SKU、数量或金额需要锁定。
4. 售后拒绝或撤销后，如无其他进行中售后，原订单恢复原履约状态并保留售后记录。
5. 售后完成后，按全额/部分、退款/退货/补寄结果更新订单售后标识和商品行状态。

### 5.8 数据规则

| 数据项 | 来源 | 生成 / 更新时机 | 口径 | 是否必填 | 异常处理 |
| --- | --- | --- | --- | --- | --- |
| 售后单号 | 售后系统 | 创建售后单 | 每次售后独立生成，不复用关闭单号 | 是 | 重复提交需幂等 |
| 订单业务形态 | 订单系统 | 入口判断和售后创建 | 商品物流、商品自提、套餐/次卡、开卡充值、预定金 | 是 | 不明确时禁止提交 |
| 售后类型 | C端选择或B端处理 | 创建/转补寄 | 仅退款、退货退款、补寄 | 是 | C端不允许主动选择补寄 |
| 售后范围 | 系统计算 | 提交申请 | 全额、部分、同SKU部分数量 | 是 | 超过剩余范围拦截 |
| 售后主状态 | 售后状态机 | 每次动作后更新 | 待审核、待寄回、待收货、待退款、退款中、退款失败、待补寄出库、待买家收货、已完成、已关闭 | 是 | 非法流转拦截 |
| 关闭类型 | 操作来源 | 进入已关闭时 | 已拒绝、已撤销、超时关闭、人工关闭 | 条件必填 | 关闭原因缺失拦截 |
| 商品退款金额 | 支付快照/商品行 | 申请、审核、退款 | 本次涉及商品/SKU/数量分摊金额 | 是 | 超最大可退拦截 |
| 原单运费退款金额 | 订单运费 | 审核/退款 | 与商品退款分开记录 | 是，可为0 | 特殊补退需留痕 |
| 寄回邮费补贴/赔付金额 | B端审核/协商 | 审核或退款 | 不并入商品退款 | 是，可为0 | 调整需原因 |
| 实际退款金额 | 财务退款 | 发起退款 | 商品退款+原单运费退款+寄回邮费补贴等合计 | 是 | 与分账拆分不一致时拦截 |
| 资产退回明细 | 支付/会员资产快照 | 退款成功 | 现金、卡金、赠送金、积分、优惠券、套餐次数、储值资产、预定金权益 | 条件必填 | 回退失败需记录 |
| 优惠券退回状态 | 优惠券系统 | 退款成功后 | 关联商品全部退款才按规则退回 | 条件必填 | 过期/不可退需记录 |
| 质检结果 | 仓库/门店 | 待收货签收质检 | 完好、破损、缺失、污染、影响二销、空包、错件等 | 退货退款必填 | 数量超限拦截 |
| 退货入库记录 | 库存系统 | 质检确认 | 仅完好可二销数量生成退货入库 | 条件必填 | 异常数量不入库 |
| 补寄出库申请 | B端/仓库/云仓 | 转补寄后创建 | 出库仓/云仓、补寄商品、数量、运费承担方、给云仓说明 | 补寄必填 | 未发货前不展示物流 |
| 分账回退记录 | 分账系统 | 消费者退款成功后 | 成功和失败均形成明细 | 已分账订单必填 | 普通失败只入库记录 |
| 合伙人佣金记录 | 分账/钱包 | 退款后计算佣金回退 | 可正常回退则回退，否则钱包可扣为负数 | 条件必填 | C端不展示 |
| 退款水单 | 财务/水单系统 | 退款成功后 | 新增退款水单，不删除原购买水单 | 是 | 生成失败需告警和补偿 |
| 操作日志 | 各处理动作 | 每次审核、调额、质检、退款、补寄、关闭 | 操作人、时间、动作、前后值、原因 | 是 | 缺日志视为异常 |

### 5.9 异常边界

| 异常场景 | 触发条件 | 系统表现 | 用户提示 | 恢复 / 回滚方式 | 是否阻断 |
| --- | --- | --- | --- | --- | --- |
| 重复申请超数量 | 商品行售后中或已退款数量超出可售后范围 | 拦截提交 | 当前商品不可重复申请或数量超出 | 重新选择剩余可售后数量 | 是 |
| 申请金额超最大可退 | C端填写金额超过系统计算 | 拦截提交 | 申请退款金额不能超过最多可退金额 | 修改金额 | 是 |
| 待寄回超时 | 超过寄回截止时间未填物流 | 自动关闭 | 售后因超时未寄回关闭 | 售后期内重新申请需新单 | 否 |
| 待收货未质检直接退款 | 退货退款单未录入质检 | 拦截退款 | 内部提示先完成质检 | 仓库质检后再退款 | 是 |
| 质检数量超限 | 完好+破损+缺失+污染数量超过申请退货数量 | 拦截提交 | 内部提示数量不一致 | 修正质检数量 | 是 |
| 退款失败 | 渠道失败、商家余额不足、账户异常 | 状态进入退款失败 | 退款失败，等待商家重新处理 | 财务补足资金或排除异常后重试 | 否 |
| 分账回退失败 | 普通分账方已提现、余额不足、账户冻结、接口失败 | 记录失败明细 | C端不展示 | 后续基于明细扩展追回能力 | 否 |
| 合伙人佣金无法回退 | 佣金账户无法正常扣回 | 扣减合伙人钱包，可为负数 | C端不展示 | 形成负余额扣减明细 | 否 |
| 资产不足不可退 | 非物流订单资产已使用或不足回退 | 不展示入口或置灰 | 展示不可退款原因 | 无，除非资产状态恢复 | 是 |
| 旧开关误限制 | 旧配置仍被售后准入引用 | 应移除引用 | 不向用户展示旧开关影响 | 代码移除或标记旧配置不生效 | 是 |

### 5.10 页面与交互要求

C端页面 / 组件：

1. 订单列表：展示申请退款、申请售后、售后详情或不可申请。
2. 订单详情：展示订单状态、商品、数量、实付、运费、支付方式、售后期和底部主按钮。
3. 售后类型页：已发货订单只展示退货退款（已收到货）和仅退款（未收货仅退款）。
4. 仅退款/退货退款申请页：选择商品、SKU、数量，填写申请退款金额、原因、说明、凭证。
5. 普通退款确认弹窗：用于非物流退款型订单，不进入申请页。
6. 填写退货物流页：填写物流公司、物流单号。
7. 售后详情页：当前状态 + 已发生处理结果卡片 + 底部按钮。

B端页面 / 组件：

1. 售后列表：状态Tab、关闭类型筛选、申请时间、售后类型、订单业务形态、营销类型、交易门店、分账状态、责任方、关键词。
2. 售后详情：顶部状态提示、用户信息、退款项目明细、申请信息、处理结果卡片、退款清分账单、分账回退区、日志。
3. 中央弹窗：同意申请、拒绝申请、转补寄、签收质检、操作退款、重新退款、创建/编辑补寄出库申请。

入口与跳转：

1. 待发货商品物流订单点击申请退款，直接进入仅退款申请页。
2. 已发货商品物流订单点击申请售后，进入售后类型页。
3. 非物流退款型订单点击申请退款，只弹普通退款确认弹窗。
4. 任意订单已有售后进行中时，按钮为售后详情。

表单与校验：

1. 申请退款金额不得超过最多可退金额。
2. 商品/SKU/数量不得超过剩余可售后范围。
3. 调额、调责、调运费承担方必须填写原因。
4. 质检数量必须满足数量校验。
5. 分账拆分明细与审批退款金额不一致时不允许执行退款。

操作反馈：

1. C端申请后展示售后详情和当前状态。
2. 调额后展示申请退款金额、最终审批金额、实际退款金额和调额说明。
3. 云仓未发货前不展示补寄物流，发货回传后展示查看补寄物流。
4. 关闭后展示关闭类型和关闭原因。

需要沿用的现有结构：

1. C端订单列表和订单详情。
2. 当前线上普通退款确认弹窗。
3. B端售后管理基础列表和详情入口。

## 6. 按需模块

### 6.1 0-1 / 复杂交互需求

原型 / Demo 目标：

1. 可演示C端从订单列表/详情发起售后到查看进度的完整路径。
2. 可演示B端按状态处理售后单，包括审核、质检、退款、补寄、失败重试。
3. 可演示C/B端状态和结果卡片联动。

提前验证流程：

1. 先按业务场景面板走查入口规则、状态流和结果回显。
2. 再基于Mock覆盖所有关键订单形态和售后状态。
3. 研发评审前确认本期范围和不做边界。

用户理解风险：

1. C端用户可能不理解为什么已发货默认不退原单运费。
2. 调额后若不展示调额原因，会增加客服咨询。
3. 补寄未发货前不展示物流，需要用清晰文案解释等待出库。
4. 非物流订单不进入申请页，需要保持普通退款弹窗语义一致。

业务反馈机制：

1. 产品评审收集客服、仓库、财务反馈。
2. Mock演示收集研发和测试对字段、状态、接口的遗漏反馈。
3. 上线后通过售后处理时长、退款失败、客服咨询原因、对账差异进行复盘。

不确定点：

1. 非物流退款型订单是否完全不需要业务审核节点，需业务最终确认。
2. 待发货仅退款是否支持自动审核或统一人工审核，需结合配置确认。
3. B端一期是否覆盖完整工作台，还是按高频状态分阶段落地，需研发排期确认。

## 7. 测试与验收

### 7.1 测试覆盖点

主流程：

1. 发货商品待发货整单退款。
2. 发货商品待发货部分仅退款。
3. 发货商品已发货仅退款。
4. 发货商品退货退款。
5. B端转补寄。
6. 非物流退款型订单确认退款。
7. 退款成功后的资产、水单、分账记录。

分支流程：

1. 同一SKU多件只退部分数量。
2. 已发货申请金额低于/等于/高于最多可退金额。
3. 退货全部完好、部分破损、部分缺失、严重污染不可二销。
4. 优惠券关联商品全部退款和部分退款。
5. 预定金权益未使用可退和已使用不可退。

异常流程：

1. 商家拒绝申请。
2. 消费者撤销申请。
3. 待寄回超时关闭。
4. 退款调用失败和重新操作退款。
5. 分账回退失败只记录数据库明细。
6. 合伙人佣金无法正常回退时钱包负数扣减。

角色权限：

1. 客服可审核、拒绝、转补寄、按配置调额/调责。
2. 仓库可质检和录入入库/异常结果。
3. 财务可退款和重试退款。
4. 管理员可人工关闭和导出。
5. 总部可看全部门店，分店默认本店。

状态流转：

1. 覆盖待审核、待寄回、待收货、待退款、退款中、退款失败、待补寄出库、待买家收货、已完成、已关闭。
2. 覆盖关闭类型已拒绝、已撤销、超时关闭、人工关闭。
3. 验证订单主状态不被售后状态覆盖。

数据口径：

1. 商品退款金额、原单运费退款金额、寄回邮费补贴/赔付金额分开。
2. 现金、卡金、赠送金、积分、优惠券、套餐次数、储值资产、预定金权益回退分开。
3. 购买水单保留，退款成功新增退款水单。
4. 分账回退成功/失败明细完整。

回归范围：

1. 原订单列表、详情和支付退款能力。
2. 原有非物流普通退款链路。
3. 订单发货、确认收货、售后期判断。
4. 会员资产、优惠券、库存、云仓、分账相关接口。

| 待发货整单退款 | 商品物流订单已付款未发货 | 多商品订单，选择全部剩余商品 | 最多可退金额含商品和原订单运费 |
| 待发货部分退款 | 商品物流订单已付款未发货 | 只退一个SKU或同SKU 1件 | 原订单运费不退，剩余商品继续履约 |
| 已发货仅退款 | 商品物流订单已发货未收货 | 选择仅退款，填写物流异常原因 | 进入待审核，审核后待退款 |
| 退货退款全部完好 | 已收货且售后期内 | 买家寄回，仓库质检全部完好 | 生成退货入库并回补库存 |
| 退货退款部分破损 | 已收货且售后期内 | 2件退货，1件完好，1件破损 | 1件入库，1件异常不入库 |
| 补寄 | 少件/漏发 | B端转补寄，云仓后续发货 | 未发货前无物流，发货后展示补寄物流 |
| 非物流退款 | 套餐未核销或自提未自提 | C端点击申请退款确认 | 直接进入待退款，不展示原因和凭证 |
| 预定金可退 | 权益未使用且在可退款期 | C端确认退款 | 创建售后单并待退款 |
| 退款失败 | 商家账户余额不足或渠道异常 | 财务发起退款失败 | 状态退款失败，支持重试 |
| 分账失败 | 消费者退款成功后普通分账方无法回退 | 分账方已提现或余额不足 | 售后完成不受影响，失败明细入库 |
| 合伙人负余额 | 合伙人佣金无法正常回退 | 钱包余额不足 | 钱包扣为负数并记录前后余额 |

### 7.3 验收标准

1. 产品只读本文可从订单列表讲通售后申请、B端处理、C端结果。
2. 研发只读本文可拆出C端、B端、后端、数据、状态机、权限、日志、集成任务。
3. 测试只读本文可写主流程、异常流、边界流、权限流、金额流和回归用例。
4. 财务只读本文可看清退款、资产、水单、分账、合伙人佣金和报表口径。
5. C端不展示后台内部财务、分账、云仓和合伙人概念。
6. B端所有关键处理动作都有中央弹窗、结果卡片和操作日志。
7. 旧开关不再限制商品物流发货订单售后能力。
