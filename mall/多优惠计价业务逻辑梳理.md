# 多优惠计价业务逻辑梳理

> 版本：v1.9 | 状态：正式版 | 适用业务：微商城、秒杀、拼团、积分订单 | 最后更新：2026-05-21

---

## 一、优惠类型总览

### 1.1 折扣类

| 类型    | 作用对象 | 计算方式     | 备注          |
| :---- | :--- | :------- | :---------- |
| 会员卡折扣 | 所有品  | 原价 × 折扣率 | 改变计价基准，最先执行 |

### 1.2 优惠券类（3种）

| 类型    | 作用对象        | 门槛         | 计算方式            | 分摊方式         | 门槛计算规则              |
| :---- | :---------- | :--------- | :-------------- | :----------- | :------------------ |
| 项目抵用券 | 绑定指定品/SKU   | 无门槛，品存在即可用 | 直接从该品扣（全额或固定金额） | 不分摊，直接作用于绑定品 | 无门槛                 |
| 现金券   | 指定品类+指定订单类型 | 满X元 or 满X件 | 满足门槛后减固定金额      | 按适用品当前价比例分摊  | **按该券适用品券前价快照计算门槛** |
| 折扣券   | 指定品类+指定订单类型 | 满X元 or 满X件 | 满足门槛后打折         | 按适用品当前价比例分摊  | **按该券适用品券前价快照计算门槛** |

> **重要说明：券门槛计算规则**\
> 所有优惠券（现金券、折扣券）的门槛判断统一按照「券前价快照」计算。\
> 「券前价快照」不是整单金额，而是每张券独立按自己的适用品范围计算：会员折扣、预定金抵扣完成后，且在任意优惠券核销前，冻结该券适用品的金额/件数。\
> 也就是说，一张只适用于水漾系列的现金券，只统计水漾系列商品；不适用的商品即使在同一订单里，也不计入该券门槛。项目抵用券、现金券、折扣券在真正核销时会影响应付金额，但不会回刷后续券的门槛判断，避免"前一张券把后一张券门槛打没了"。

### 1.3 活动类

| 类型          | 性质         | 来源               | 使用限制                       |
| :---------- | :--------- | :--------------- | :------------------------- |
| 预定金         | 活动预付资金     | 用户参与预定金活动时预付     | 仅限预定金活动商品使用，有活动期限，可全额或部分抵扣 |
| **预定金随机立减** | **活动额外优惠** | **预定金活动配置的随机立减** | **仅在预定金活动开启随机立减时生效，最后执行**  |

> **预定金活动说明**

```
- **活动机制**：用户在活动期间预先支付定金，尾款期可享受预定金抵扣优惠
- **适用活动类型**：仅限秒杀活动和拼团活动，普通商品和限时折扣不支持预定金
- **活动配置**：后台可设置哪些具体的秒杀/拼团活动启用预定金功能
- **抵扣方式**： 
    - 全额抵扣：预定金完全抵扣商品价格，商品变为0元（退出后续分摊池） 
    - 部分抵扣：预定金按设定金额抵扣，剩余部分继续参与后续优惠计算

- **计价位置**：在会员折扣之后、券类优惠之前执行，确保预定金优先级
- **叠加规则**：与其他优惠可叠加使用，但受商品级「仅限现金支付」限制
- **🎲 随机立减机制**： 
    - **触发条件**：预定金活动开启随机立减功能 + 用户参与该预定金活动
    - **执行时机**：在支付流程中实时计算，基于最终现金支付金额进行随机立减
    - **立减范围**：后台可配置立减金额区间（如1-50元） 
    - **立减对象**：仅对预定金活动商品的最终现金支付金额生效
    - **计算方式**：支付流程中系统随机生成立减金额，从该商品最终现金支付金额中扣除
```

### 1.4 虚拟资产类

| 类型  | 性质     | 来源        | 使用限制                          |
| :-- | :----- | :-------- | :---------------------------- |
| 积分  | 平台虚拟积分 | 消费累积      | 有余额门槛、单笔抵扣上限                  |
| 赠送金 | 平台赠送资产 | 平台赠送/活动奖励 | 有有效期、适用项目限制、订单类型限制、是否叠加会员折扣开关 |

### 1.5 支付类

| 类型       | 性质     | 说明                                                   |
| :------- | :----- | :--------------------------------------------------- |
| 会员卡卡金    | 用户真实储值 | 用户充值的真实资金；可配置适用品类/项目范围；最后消耗；若商品标记为「仅限现金支付」则该商品不可使用卡金 |
| 现金/第三方支付 | 真实资金   | 微信/支付宝/银行卡，兜底支付；亦为「仅限现金支付」商品的唯一可用支付方式                |

## 二（附）、商品级「仅限现金支付」最高优先级配置

### 核心定义

> **「仅限现金支付」= 仅限制支付方式，不限制折扣计价。**  该标记只禁止使用卡金、赠送金、积分、券等非现金支付手段；会员卡折扣属于「改变计价基准」的折扣行为，不受此限制，正常执行。 **预定金随机立减属于活动优惠，不受此限制，正常执行。**

### 品级隔离原则

```plaintext
「仅限现金支付」配置不扩散至整笔订单：
  - 仅影响被标记商品自身的支付方式
  - 同订单内其他未标记商品的优惠计算不受影响
  - 品级隔离，互不干扰
  - 预定金随机立减不受此限制影响
```

### 后台配置项

```plaintext
商品级「仅限现金支付」配置 = {
  开关：是否标记为「仅限现金支付」[ON / OFF]
  └ OFF：该商品正常参与所有优惠计算与支付方式
  └ ON：继续以下约束

  【折扣计价（不受限）】
  STEP1 会员卡折扣：✅ 正常执行（若在折扣适用品范围内）

  【被禁止的支付手段（全部跳过）】
  STEP2 预定金抵扣：❌ 跳过
  STEP3 项目抵用券：❌ 跳过
  STEP4 现金券：    ❌ 跳过
  STEP5 折扣券：    ❌ 跳过
  STEP6 积分抵扣：  ❌ 跳过
  STEP7 赠送金：    ❌ 跳过
  STEP8 卡金：      ❌ 跳过

  【最终结算】
  STEP9 现金支付：支付 STEP1 折后价（或原价，取决于折扣是否适用）
  STEP10 预定金随机立减：🎲 支付流程中实时计算（若该商品参与预定金活动且开启随机立减）

  ⚠️ 备注：该标记在后台商品详情页或批量管理页独立配置
}
```

## 二、订单类型 × 优惠支持矩阵

> 所有开关均可在后台按订单类型独立配置，以下为系统默认值。

| 优惠类型        | 普通微商城     | 秒杀订单      | 拼团订单      | 积分订单                        |
| :---------- | :-------- | :-------- | :-------- | :-------------------------- |
| 会员卡折扣       | ✅ 可配置     | ✅ 可配置     | ✅ 可配置     | 🔸 电商模式：仅现金部分可享受📱 标准模式：不适用 |
| 预定金抵扣       | ❌ 不适用     | ✅ 可配置     | ✅ 可配置     | ❌ 不适用                       |
| **预定金随机立减** | **❌ 不适用** | **✅ 可配置** | **✅ 可配置** | **❌ 不适用**                   |
| 项目抵用券       | ✅ 可配置     | ✅ 可配置     | ✅ 可配置     | ❌ 不适用                       |
| 现金券         | ✅ 可配置     | ✅ 可配置     | ✅ 可配置     | ❌ 不适用                       |
| 折扣券         | ✅ 可配置     | ✅ 可配置     | ✅ 可配置     | ❌ 不适用                       |
| 积分抵扣        | ✅ 可配置     | ✅ 可配置     | ✅ 可配置     | ❌ 不适用（已用积分兑换）               |
| 赠送金         | ✅ 可配置     | ✅ 可配置     | ✅ 可配置     | 🔸 电商模式：仅现金部分可抵扣📱 标准模式：可配置 |
| 会员卡卡金       | ✅ 可配置     | ✅ 可配置     | ✅ 可配置     | 🔸 电商模式：仅现金部分可支付📱 标准模式：可配置 |

> **积分订单说明**：积分兑换本身即为积分消费，不叠加任何折扣类优惠及券类优惠；赠送金可按业务需要配置开放；卡金支付可按业务需要开放。

### 2.5 积分商城混合支付规则（仅电商模式）

> **模式说明**：此规则仅适用于电商模式，标准模式下积分订单保持现有逻辑（纯积分兑换，不叠加其他优惠）

**积分商城商品构成**：

- 积分商城商品通常采用"积分+现金"混合支付模式
- 商品价格 = 固定积分 + 现金部分

**优惠适用规则**：

1. **积分部分**：
   - 积分数量固定，不参与任何优惠计算
   - 不能享受会员折扣、优惠券、赠送金等优惠
   - 只能使用用户积分余额支付

2. **现金部分**：
   - 可以参与其他优惠的计算和抵扣
   - 可享受会员折扣（如果积分订单配置允许）
   - 可使用赠送金抵扣（如果配置开放）
   - 可使用卡金支付（如果配置开放）
   - 最终剩余部分用现金支付

**计价示例**：

```plaintext
原商品：8000积分 + 20元现金
├ 会员9折（如开放）：8000积分 + 18元现金  
├ 赠送金抵扣5元：8000积分 + 13元现金
├ 卡金抵扣10元：8000积分 + 3元现金
└ 最终支付：8000积分 + 3元现金
```

**门槛计算**：

- 优惠券门槛判断时，只计算现金部分金额
- 积分部分不计入门槛计算

### 2.6 邮费卡金支付配置

**配置说明**：

- 后台可独立配置邮费是否允许使用卡金支付
- 默认设置：允许卡金支付邮费

**规则逻辑**：

```plaintext
邮费卡金支付配置 = {
  开关：邮费是否可用卡金支付 [ON / OFF]
  └ ON：邮费可以使用卡金支付（默认）
  └ OFF：邮费只能现金支付，卡金跳过邮费部分
}
```

### 2.7 赠送金与会员折扣互斥配置

**配置说明**：

- 后台可配置赠送金是否与会员折扣互斥使用
- 默认设置：可共用（先享受会员折扣，再用赠送金抵扣）

**规则逻辑**：

```plaintext
赠送金折扣互斥配置 = {
  开关：赠送金与会员折扣是否互斥 [共用 / 互斥]
  
  └ 共用（默认）：
    STEP1: 先执行会员折扣
    STEP7: 再执行赠送金抵扣
    
  └ 互斥：
    用户需选择其一：
    - 选择会员折扣：正常享受折扣，赠送金不可用
    - 选择赠送金：按原价计算，跳过会员折扣
}
```

**计价示例**：

```plaintext
原商品100元，会员9折，有20元赠送金

共用模式：
100元 → 会员9折90元 → 赠送金抵扣20元 → 实付70元

互斥模式：
方案A: 100元 → 会员9折90元 → 实付90元（赠送金不可用）
方案B: 100元 → 赠送金抵扣20元 → 实付80元（不享受折扣）
用户可选择方案A（更优惠）
```

### 2.8 预定金随机立减配置

**配置说明**：

- 预定金活动可独立配置是否开启随机立减功能
- 默认设置：关闭随机立减

**规则逻辑**：

```plaintext
预定金随机立减配置 = {
  开关：是否开启随机立减 [ON / OFF]
  └ OFF：不启用随机立减（默认）
  └ ON：启用随机立减，继续以下配置

  【立减金额配置】
  立减金额区间：最小金额 ~ 最大金额（如：1元 ~ 50元）
  立减算法：随机算法类型（均匀分布/加权分布等）
  
  【立减条件】
  最低订单金额：订单现金支付部分需满足X元才可享受立减
  每用户限制：每个用户在该活动期间最多享受X次立减
  
  【立减时机】
  执行时点：支付流程中实时计算，对最终现金支付金额执行立减
  作用范围：仅对参与该预定金活动的商品生效
  分摊方式：若订单内有多个预定金活动商品，按最终现金支付金额比例分摊立减额
}
```

**计价示例**：

```plaintext
预定金活动商品原价100元，预定金20元，随机立减区间1-10元

STEP1-8: 正常优惠计算后，现金支付金额为30元
STEP9: 支付流程中随机立减5元（实时计算）
STEP10: 最终现金支付25元
最终支付：25元现金 + 20元预定金 = 45元总成本（原价100元）
```

## 三、完整计价链（核心）

```plaintext
【订单内商品分类】
将订单内所有商品拆分为两个子集：
  子集A：普通商品（无「仅限现金支付」标记）
  子集B：仅限现金支付商品（有标记）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
子集A（普通商品）完整计价链
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
STEP1  会员折扣
  ├ 判断：订单类型是否开启会员折扣 + 商品是否在折扣适用品范围
  ├ 判断：订单是否使用赠送金 + 赠送金与折扣是否配置为「互斥」
  │   → 若互斥且已勾选赠送金：子集A所有商品折扣率强制 = 1（按原价）
  │   → 否则：各品按折扣率正常计算
  └ 输出：各品折后价（= 原价 × 折扣率，或原价）

STEP2  预定金抵扣（若有预定金活动商品）
  ├ 判断：商品是否为预定金活动商品 + 用户是否已支付预定金
  ├ 抵扣方式：全额抵扣或部分抵扣（按活动配置）
  ├ 全额抵扣：该品 current_price = 0，退出分摊池
  ├ 部分抵扣：该品 current_price -= 预定金金额（最低 = 0）
  └ 检查：current_price = 0 的品移出分摊池

STEP2.5  冻结优惠券门槛快照
  ├ 生成时点：STEP1会员折扣、STEP2预定金抵扣完成后
  ├ 快照粒度：每张现金券/折扣券独立生成一份快照
  ├ 快照内容：该券「适用品范围内」的金额合计、件数合计
  ├ 不计入范围：不在该券适用品范围内的商品、仅限现金支付商品、邮费
  ├ 预定金影响：预定金先按商品分摊，券快照读取预定金分摊后的商品金额
  └ 目的：确保每张券只按自己的可用品项验门槛，且多张券依次核销时门槛稳定可预期

STEP3  项目抵用券（若有）
  ├ 每张券直接作用于其绑定品/SKU
  ├ 全额抵用：该品 current_price = 0，退出分摊池
  ├ 固定金额：该品 current_price -= 抵用金额（最低 = 0）
  └ 检查：current_price = 0 的品移出分摊池（但不回刷已冻结的券门槛快照）

STEP4  现金券（若有，按「满金额门槛从高到低」排序执行）
  ├ **验证门槛（基于该现金券自己的STEP2.5券前价快照）**
  ├ 按「适用品当前价比例」分摊减免额到各品
  └ 更新各品 current_price

STEP5  折扣券（若有，按「满金额门槛从高到低」排序执行）
  ├ **验证门槛（基于该折扣券自己的STEP2.5券前价快照）**
  ├ 按「适用品当前价比例」分摊折扣减免额到各品
  └ 更新各品 current_price

STEP6  积分抵扣（若开启且用户选择使用）
  ├ 验证：用户积分余额 ≥ 最低余额门槛
  ├ 计算可抵金额：min(用户积分/换算比例, 单笔金额上限, 订单应付额×比例上限)
  ├ 按「分摊池内各品当前价比例」分摊积分减免额
  └ 更新各品 current_price

STEP7  赠送金（若有且用户选择使用）
  ├ 按赠送金适用品范围过滤（全品/白名单/黑名单）
  ├ 计算可用金额：赠送金余额（不超过适用品当前价总和）
  ├ 按「赠送金适用品当前价比例」分摊减免额
  └ 更新各品 current_price

STEP8  卡金支付（若开启且用户选择使用）
  ├ 按卡金适用品范围过滤（全品/白名单/黑名单）
  ├ 计算可用金额：min(卡金余额, 适用品当前价总和)
  ├ 按「卡金适用品当前价比例」分摊卡金支付额
  └ 更新各品 current_price

**STEP9  预定金随机立减（若有预定金活动且开启随机立减）**
  ├ 判断：商品是否为预定金活动商品 + 该活动是否开启随机立减
  ├ 验证：订单现金支付总额是否满足立减最低门槛
  ├ 生成：系统随机生成立减金额（在配置区间内）
  ├ 分摊：若有多个预定金活动商品，按各品当前价比例分摊立减额
  ├ 限制：检查用户立减次数限制，超限则跳过
  └ 更新：各预定金活动商品 current_price -= 立减分摊额

STEP10 现金支付
  └ 各品 cash_amount = current_price（STEP9后剩余值）

STEP1  会员折扣：✅ 正常执行（若在折扣适用品范围内）
STEP2  预定金抵扣：❌ 跳过
```

## 四、分摊尾差处理（最大余数法）

```plaintext
所有按比例分摊的计算，统一使用最大余数法处理精度误差：

Step 1：每品按比例计算，向下截断到分（保留2位小数，截断不四舍五入）
Step 2：计算所有品截断后总和 vs 应分摊总额的差值（尾差）
Step 3：将尾差按"小数部分从大到小"逐一补0.01给各品，直到补完
Step 4：验证：所有品分摊额之和 = 应分摊总额 ✅

目的：保证分摊无损（所有品之和 = 订单总额），不多一分不少一分。

⚠️ 随机立减分摊同样使用最大余数法处理精度误差。
```

## 七、品级支付分摊 · 完整规则

### 7.1 分摊池管理

```plaintext
分摊池规则：
  初始化：所有订单内的品
  移出条件：品的 current_price = 0（已免单）
  移出时机：项目抵用券执行后立即检查

  分摊池一旦品被移出，后续所有步骤该品不参与任何分摊
  被移出品的积分/赠送金/卡金/现金分摊额均记为 0
  
  ⚠️ 随机立减独立计算，不依赖分摊池状态
```

### 7.2 每品的完整价格链

```plaintext
品i 的价格链：
  原价
  - 会员折扣减免额    = 原价 × (1 - 折扣率)
                        【互斥重算：若赠送金与折扣配置为「互斥」，
                        且用户已勾选赠送金，则此项 = 0，该品按原价进入后续计算】
  - 项目抵用券减免额  = 直接抵用（若有绑定券）
  - 现金券减免分摊额  = 按适用品当前价比例分摊（若适用）
  - 折扣券减免分摊额  = 按适用品当前价比例分摊（若适用）
  - 积分抵扣分摊额    = 按分摊池当前价比例分摊
  - 赠送金抵扣分摊额  = 按赠送金适用品当前价比例分摊
  - 卡金支付分摊额    = 按卡金适用品当前价比例分摊
  **- 预定金随机立减分摊额 = 按预定金活动商品当前价比例分摊（若适用）**
  = **现金支付额**        = **上述计算后的剩余值**

  验证：以上所有项之和 = 该品原价 ✅
```

### 7.3 多张券的品级分摊记录

```plaintext
每品 × 每张券 = 一条分摊记录
记录内容：
  - 该品是否在该券适用范围内
  - 该券计算时该品的当前价（审计用）
  - 该券分摊到该品的减免额
  - 分摊前/后的价格快照

每品 × 预定金随机立减 = 一条立减记录
记录内容：
  - 该品是否为预定金活动商品
  - 立减计算时该品的当前价（STEP8卡金支付后）
  - 立减分摊到该品的减免额
  - 立减前/后的现金支付金额
```

## 九、退款规则

### 9.1 退款基本原则

```plaintext
① 退款严格按品级支付快照执行，不重新计算
② 原路逆向退回：现金→卡金→赠送金→积分→券→预定金随机立减
③ 部分退款（退某品）：只退该品的快照数据，其他品不受影响
④ 支付快照一旦落库即为不可变数据，任何退款操作不修改快照
```

### 9.2 各优惠的退款规则

| 优惠类型        | 退款方式                        | 特殊规则                                          |
| :---------- | :-------------------------- | :-------------------------------------------- |
| 现金支付        | 原路退回现金                      | 无                                             |
| 卡金支付        | 退回卡金余额                      | 无                                             |
| 预定金抵扣       | 部分退款：不退预定金；全额退款：原路退回        | **部分退款**：预定金不退回，保持已使用状态**全额退款**：预定金原路退回预定金账户  |
| **预定金随机立减** | **部分退款：不退立减金额；全额退款：不退立减金额** | **立减金额属于活动优惠，一旦享受不可退回\*\*\*\*用户实际少支付的金额不予退还** |
| 赠送金         | 原路退回赠送金余额                   | 只管原路退回，不判断是否过期                                |
| 积分          | 原路返还积分                      | 只管原路退回；若积分已被消费，依商家配置方案处理（详见9.4场景③）            |
| 现金券/折扣券     | 全额退款→退回券；部分退款→券不退           | 全额退款：券恢复可用。部分退款：券不退，保持已核销状态，不追差价              |
| 项目抵用券       | 视情况处理                       | 未发货：退回凭证或现金补偿；已发货：不退                          |
| 会员折扣        | 无需处理                        | 折扣不涉及资产退还                                     |

### 9.3 退款逆向顺序

```plaintext
退款执行顺序（支付顺序的逆向）：
① 退现金（微信/支付宝退款接口）
② **预定金随机立减：不退回（活动优惠，一旦享受不可逆）**
③ 退卡金（余额账户加回）
④ 退预定金（预定金账户加回或现金补偿）
⑤ 退赠送金（视规则加回或现金补偿）
⑥ 返还积分（积分账户返还）
⑦ 处理券（退回/作废）
   会员折扣不涉及退还
```

### 9.4 退款特殊场景

```plaintext
场景①：退掉"已满足券门槛"的品后，券门槛不再满足
  → 推荐方案：按快照执行，不追溯其他品的券减免（体验优先）

场景②：退款时赠送金已过期
  → 原路退回赠送金账户，不管是否过期

场景③：退款时积分已被消费（账户积分余额不足以回退）
  【方案A：严格回退模式（默认）】
  → 余额不足：退款不可执行，提示"积分余额不足以回退，无法退款"
  【方案B：宽松退款模式】
  → 余额不足：将积分余额扣减至0，不足部分不追偿，退款正常执行

场景④：优惠券退款规则
  → 全额退款：券原路退回，恢复可用状态
  → 部分退款：券不退，只要有任意一个关联商品未退款，该券均不退

场景⑤：全额退款（整单取消）
  → 按所有品快照逐品逆向退回
  → 所有券恢复未使用状态（或视规则作废）
  → **预定金随机立减不退回**

**场景⑥：预定金随机立减退款处理**
  → **部分退款**：立减金额不退回，用户实际享受的优惠保持不变
  → **全额退款**：立减金额不退回，属于活动优惠，一旦享受不可逆
  → **业务逻辑**：用户通过预定金活动获得的随机立减属于活动奖励，不纳入退款范围
  → **财务处理**：立减金额由活动成本承担，不影响用户退款金额计算
```

## 十一、设计核心原则总览

```plaintext
【最高优先级规则】
①（最高） 商品级「仅限现金支付」标识高于一切订单级、活动级配置，
   该商品不得参与支付类优惠（卡金/赠送金/积分/券），但可享受会员卡折扣和预定金随机立减

【计价顺序原则】
② 先折价定基准 → 冻结券门槛快照 → 再核销优惠 → **随机立减** → 最后现金支付
③ 会员折扣、预定金先执行，用于生成每张现金券/折扣券自己的券前价快照
④ 现金券、折扣券按各自适用品范围快照校验门槛，后续核销不回刷门槛
⑤ 项目抵用券直接作用于绑定品，影响实付金额，但不改变已冻结的券门槛
⑥ 满金额门槛券先于满件数门槛券；同门槛下按优惠力度从大到小
⑦ 虚拟资产先于真实储值：积分→赠送金→卡金（消耗平台成本资产在前）
⑧ **预定金随机立减在现金支付前执行，作为最后一道优惠，基于卡金支付后的剩余金额计算**

【分摊原则】
⑨ 项目抵用券直接作用于绑定品，不分摊给其他品
⑩ 免单品退出分摊池，不参与后续任何计算
⑪ 「仅限现金支付」品退出支付类分摊池（STEP2~STEP8），但可享受会员折扣和随机立减
⑫ 每种优惠只在"适用范围内的品"中分摊
⑬ 统一使用最大余数法处理分摊尾差，保证分摊无损
⑭ **随机立减按预定金活动商品当前价（卡金支付后）比例分摊，在现金支付前执行**

【数据原则】
⑮ 每品保存完整的支付明细快照，退款直接读快照，不重算
⑯ 预定金活动仅适用于秒杀和拼团，可配置具体哪些活动启用预定金功能
⑰ 分账逻辑：仅微信和支付宝等线上支付方式触发分账，卡金不参与分账
⑱ 所有操作写流水记录，支持审计和对账
⑲ 退款按品级快照原路逆向，不影响其他品
⑳ **随机立减金额一旦生成不可退回，属于活动优惠性质**
```

## 十二、预定金随机立减技术实现要点

### 12.0 支付时机与页面展示逻辑

#### 立减计算时机

- **确认订单页面**：不计算具体立减金额，只显示活动提示
- **支付流程中**：实时计算随机立减金额，基于最终现金支付金额执行立减
- **支付完成后**：将立减金额存储到订单数据中，用于后续展示

#### 页面展示规则

1. **确认订单页**：「该商品参与随机立减活动，支付时将随机减免1-50元」
2. **支付成功页**：「恭喜！您获得随机立减XX元」
3. **订单详情页**：「随机立减：-XX元」

#### 技术实现关键点

- 避免在确认订单阶段预先计算立减金额（防止用户刷新页面刷立减）
- 支付接口调用时实时生成立减金额
- 立减金额一旦确定即存储，不可更改

### 12.1 随机算法配置

```plaintext
随机立减算法配置 = {
  算法类型：均匀分布 / 加权分布 / 阶梯概率
  
  【均匀分布】
  立减区间：[最小值, 最大值]
  生成方式：Math.random() * (max - min) + min
  
  【加权分布】
  权重配置：
    - 小额立减（1-10元）：权重70%
    - 中额立减（11-30元）：权重25%
    - 大额立减（31-50元）：权重5%
  
  【阶梯概率】
  概率阶梯：
    - 1-5元：50%概率
    - 6-15元：30%概率
    - 16-30元：15%概率
    - 31-50元：5%概率
}
```

### 12.2 立减时机控制

```plaintext
立减执行时机控制 = {
  触发时点：用户点击"立即支付"按钮时
  生成逻辑：
    ① 验证用户立减次数限制
    ② 验证订单现金支付金额门槛
    ③ 筛选预定金活动商品
    ④ 按配置算法生成立减金额
    ⑤ 按商品现金支付比例分摊立减额
    ⑥ 更新订单最终支付金额
    ⑦ 记录立减流水日志
  
  幂等性保证：
    - 同一订单号只能生成一次立减
    - 支付失败重试时复用已生成的立减金额
    - 立减金额与订单绑定，不重新计算
}
```
