# AI-Native PRD 输出规范

## 核心原则

AI-Native PRD = 人能看 + AI能读 + 研发能拆 + 测试能测，统一、完整、信息闭合。

完整要求：业务结果 + 范围边界 + 场景路径 + 状态规则 + 权限规则 + 异常边界

## 四维标准

### 人能看
业务方、产品、研发、测试都能理解业务目标、用户场景和方案边界。
- 业务背景和价值判断清晰
- 用户角色和目标明确
- 核心流程可从头讲到尾

### AI能读
结构清晰、规则明确，业务场景/状态/权限/异常/范围边界可被AI提取和复用。
- 使用结构化表格描述规则
- 状态机、枚举值、字段定义标准化
- 场景路径用伪代码或流程图表达

### 研发能拆
可识别页面、流程、数据、权限、状态流转和实现边界。
- 页面和组件清单明确
- 接口依赖和字段定义完整
- 状态流转表可直接转为代码逻辑

### 测试能测
可拆出主流程、分支流程、异常流程、角色权限、边界场景、回归范围。
- 主流程步骤化描述
- 异常场景和恢复方式明确
- 测试数据和验收标准可直接生成用例

## PRD 文档结构

### 0. 文档协作信息
- 基本信息表（需求名称、版本、状态、负责人、协同方、更新时间）
- 变更记录表（时间、变更人、内容、影响范围、原因）
- 评审与确认状态表

### 1. 项目上下文包
- 业务背景（为什么做、不做的影响）
- 用户角色表（角色、目标、典型操作、关注点）
- 核心流程（入口、主路径、关键节点、结束状态）
- 业务术语表
- 系统边界（包含、不包含、依赖、复用约束）

### 2. 范围说明
- 影响范围表（影响类别、对象、说明、风险等级、负责人）
- 必须覆盖范围表（功能、场景、数据、权限、状态、集成、验收）

### 3. 业务结果说明
- 为什么做
- 目标指标表（指标、当前状态、目标状态、验证方式）
- 当前问题列表
- 验证方式（业务验证、用户验证、数据验证、流程验证）

### 4. 需求分类判断
- 分类结论（复杂交互 / 有耦合 / 智能体 / 合规风险 / 数据分析）
- 判断依据
- 自动判断矩阵表

### 5. PRD 主体
- **需求目标**：用户目标、业务目标、产品目标、技术目标
- **范围边界**：本期做、本期不做、后续可能做、明确排除
- **业务场景表**：场景编号、名称、触发条件、参与角色、期望结果
- **主流程表**：步骤、动作、页面、输入、系统处理、输出/状态
- **分支流程表**：编号、触发条件、路径、回归方式、结果状态
- **状态规则表**：状态、进入条件、可执行动作、下一状态、退出条件、备注
- **权限规则表**：角色、可见、可操作、禁止、数据范围
- **数据规则表**：数据项、来源、时机、口径、是否必填、异常处理
- **异常边界表**：场景、触发条件、系统表现、用户提示、恢复方式、是否阻断
- **页面与交互要求**：C端页面、B端页面、入口跳转、表单校验、操作反馈
- **接口与系统依赖表**：依赖对象、用途、输入、输出、失败处理、负责人

### 6. 按需模块（按分类结论决定是否需要）
- 复杂交互模块：原型目标、提前验证、用户理解风险、不确定点
- 耦合需求模块：影响对象表、回归范围、兼容策略
- 智能体模块：Eval样本、模型输出、智能体行为（不适用时标记）
- 合规/风险模块：风险项表、审核人、处理方式
- 数据/分析模块：指标表、埋点、分析口径、使用方

### 7. 测试与验收
- 测试覆盖点（主流程、分支、异常、权限、状态、数据口径、回归）
- 测试数据表（场景、前置条件、数据、期望结果）
- 验收标准

### 8. 研发任务拆解建议
- 按模块拆分（C端前端、B端前端、后端状态机、退款资产、仓库库存、分账财务、权限日志、Mock测试）

### 9. 开放问题
- 编号、优先级、问题、建议处理

## 写作规则

1. 每个规则必须有明确的触发条件和预期结果
2. 状态流转必须用表格描述，包含进入条件、可执行动作、下一状态
3. 金额、权限、异常必须独立成表，不混入流程描述
4. 使用枚举值而非自然语言描述选项（如 reasonCode: NO_WANT / WRONG_ORDER）
5. 每个页面/弹窗必须说明触发条件、展示内容、操作按钮和操作结果
6. 分支流程必须说明如何回归主流程
7. 异常场景必须说明是否阻断、用户提示和恢复方式
8. 不同角色的可见信息和可操作范围必须独立描述
9. 本期做和不做必须明确列出，不留模糊地带
10. 所有调整类操作（调额、调责、调运费）必须填写原因并留痕
