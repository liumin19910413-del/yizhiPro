980团购套餐卡资金拆解与核销结算需求PRD

一、需求背景

1.1 现状说明（线上已上线版本）

当前线上系统中，商城推广服务费10%的资金已按直推（共建）30%、间推（销售归属门店）30%进行分佣处理。但其中40%的平台营销费部分，目前并未作为独立分佣账户处理：

- 若用户使用微信/支付宝等现金支付，这40%直接进入收款方账户，未单独拆分；
- 若用户使用积分支付，收款方侧没有对应的分账记录，这部分钱的归属不清晰。

原来的处理逻辑是将这40%视为商家收入的一部分整体处理，未做独立记账。

⸻

1.2 改造目标（本次需求为目标态）

随着业务发展，商家收益需要拆分得更加清晰，每一笔收入都需要有明确的名目和归属。因此本次需求的目标是：

将商城推广服务费10%中的40%平台营销费从收款方收入中拆分出来，作为独立的系统虚拟账户（平台营销费账户）单独记账，不再混入收款方账户。

本文档描述的是改造后的目标态，即未来希望达到的状态。

⸻

1.3 业务背景

当前商城存在一类团购套餐卡，例如 980团购套餐卡。该卡本质上是基于系统已有套餐进行商城售卖，用户支付后获得对应套餐权益，并可在后续到门店进行一次或多次核销。

与普通商品或纯销售卡不同，980团购套餐卡存在以下特点：

1. 套餐本身已在系统中存在，不需要重新创建套餐。
2. 商城售卖时，需要在已有套餐上额外绑定一套资金拆解与分账规则。
3. 用户购买后，支付成功阶段需要先扣除平台技术服务费、商城推广服务费、总部交付成本等固定项。
4. 剩余金额形成项目结算金额，后续按用户实际核销进行释放。
5. 用户核销时，销售门店和服务门店可能是同一家，也可能不是同一家。
6. 销售侧收益与履约侧收益必须按不同业务名目分别入账。
7. 服务门店本身不单独维护复杂入驻流程，只需要关联已有系统门店账号，并配置其履约服务费实际收款主体。
8. 服务门店的履约服务费可以结算到门店账户，也可以结算到指定合伙人账户，但收入名目必须清晰区分。

因此，本需求需要新增一套 团购套餐结算规则配置 + 服务门店关联配置 + 核销后资金清分能力。

⸻

二、需求目标

2.1 业务目标

通过980团购套餐卡的结算规则配置，实现：

1. 支持已有套餐在商城售卖后，自动套用团购套餐结算规则。
2. 支持支付成功阶段资金拆解。
3. 支持核销阶段按规则释放项目结算金额。
4. 支持跨店核销场景下，销售服务费和履约服务费分别结算。
5. 支持服务门店的履约服务费结算到指定门店账户或合伙人账户。
6. 通过要求服务门店具备系统门店账号，形成平台商业化入口。
7. 保障后续财务对账、结算、开票、收入统计时名目清晰。

2.2 产品目标

本期需要完成以下能力：

1. 新增「团购套餐结算规则」配置。
2. 支持选择已有套餐并绑定结算规则。
3. 新增「服务门店关联配置」。
4. 支持服务门店关联系统门店账号与履约服务费收款主体。
5. 支持980卡支付成功后生成资金拆解快照。
6. 支持核销成功后生成履约服务费、销售服务费两类结算记录。
7. 支持同一门店既是销售门店又是服务门店时，仍按不同名目生成两条记录。
8. 支持服务门店履约服务费入账到指定合伙人名下。
9. 支持历史订单规则快照，避免后续配置变更影响历史结算。

⸻

三、适用范围

3.1 本期适用业务

本期以 980团购套餐卡 为首个落地场景。

后续该能力可复用到：

1. 其他团购套餐卡；
2. 单次体验卡；
3. 多次套餐卡；
4. 组合项目卡；
5. 跨店核销套餐；
6. 已有套餐在商城售卖后的结算规则扩展。

3.2 不适用范围

本需求不处理以下内容：

1. 不重新创建套餐。
2. 不重构现有套餐体系。
3. 不新增复杂的服务门店入驻流程。
4. 不维护服务门店详细资料。
5. 不单独配置核销权限、服务区域、可核销套餐范围等复杂信息。
6. 不在本期处理退款后资金回退规则。
7. 不处理拉卡拉预留10%的规则，因为该规则属于拉卡拉体系，不属于本系统资金清分账本必需内容。

四、核心概念定义

4.1 已有套餐

指系统中已经存在的套餐权益，例如980团购套餐卡对应的套餐。
本需求中不新建套餐，只选择已有套餐并绑定结算规则。

4.2 团购套餐结算规则

指某个已有套餐在商城售卖后，适用的资金拆解和核销结算规则。

该规则决定：

1. 支付成功后怎么扣除固定费用；
2. 商城推广服务费怎么分；
3. 剩余项目结算金额如何进入待核销释放；
4. 每次核销释放金额如何计算；
5. 履约服务费和销售服务费如何分配。

4.3 销售归属门店

指该订单销售关系归属的门店。

判断规则：

销售来源	销售归属门店
店家自己销售	当前销售门店
共建销售	共建所属上级店家
门店员工销售	员工所属门店
平台销售	平台默认销售归属方
无法识别	进入异常或按默认配置处理

销售归属门店用于接收：

1. 商城推广服务费中的店家份额；
2. 核销阶段产生的销售服务费。

4.4 服务门店

指用户实际核销套餐权益的系统门店账号。

服务门店用于识别：

1. 本次核销发生在哪个系统门店；
2. 本次履约服务费应该根据哪个服务门店关联配置找到收款主体。

4.5 服务门店关联配置

指系统门店账号与履约服务费收款主体之间的关联关系。

例如：

系统门店账号	履约收款主体类型	履约收款主体
广州天河店	合伙人	合伙人A
深圳南山店	门店账户	深圳南山店账户

该配置只决定：

该服务门店核销后产生的履约服务费，实际入账到哪个账户。

4.6 履约服务费

指用户核销套餐权益时，实际完成服务履约的一方应获得的收入名目。

履约服务费金额由团购套餐结算规则决定。
履约服务费收款主体由服务门店关联配置决定。

4.7 销售服务费

指用户核销套餐权益时，基于订单销售归属关系，销售归属门店应获得的收入名目。

销售服务费金额由团购套餐结算规则决定。
销售服务费收款方由订单支付时固化的销售归属门店决定。

4.8 项目结算金额

指用户支付成功后，扣除平台技术服务费、商城推广服务费、总部交付成本后，剩余用于后续核销释放的金额。

公式：

项目结算金额 = 用户实付金额 - 平台技术服务费 - 商城推广服务费 - 总部交付成本

五、980团购套餐卡资金拆解规则

5.1 基础数据

项目	数值
用户实付金额	980元
平台技术服务费比例	1.5%
拉卡拉支付通道费率	0.38%
商城推广服务费比例	10%
总部交付成本	120元
套餐权益次数	10次
项目结算金额	747.3元
单次核销释放金额	74.73元

⸻

5.2 平台技术服务费

平台技术服务费按用户实付金额的1.5%收取。

平台技术服务费 = 980 × 1.5% = 14.7元

其中拉卡拉支付通道成本为：

拉卡拉支付手续费 = 980 × 0.38% = 3.72元

平台技术服务补差为：

平台技术服务补差 = 14.7 - 3.72 = 10.98元

系统规则：

1. 系统按1.5%计提平台技术服务费。
2. 拉卡拉0.38%仅作为支付通道成本记录。
3. 不需要单独为拉卡拉支付手续费建立分账账本。
4. 如支付通道不走拉卡拉，则不产生拉卡拉手续费，但平台技术服务费仍按1.5%收满。

⸻

5.3 商城推广服务费

商城推广服务费按用户实付金额的10%计提。

商城推广服务费 = 980 × 10% = 98元

商城推广服务费分配规则：

商城推广服务费通过现有分佣规则配置，按佣金池比例分配给三个角色：

分佣角色	比例	金额	说明
直推（共建）	30%	29.4元	真实收益账户
间推（销售归属门店）	30%	29.4元	真实收益账户
平台营销费	40%	39.2元	系统虚拟账户（系统级固定接收方）
合计	100%	98元	-

按销售场景的实际分配：

销售场景	直推（共建）	间推（销售归属门店）	平台营销费账户
共建销售	30%（29.4元）	30%（29.4元）	40%（39.2元）
店家直接销售	—（不存在）	30%（29.4元）	70%（68.6元）
无销售人	—	—	100%（98元）

核心规则：当某个分佣角色（直推/间推）在订单中不存在时，该角色对应份额自动归入平台营销费账户。

特殊规则：

1. 有共建归属时，共建30%结算给对应共建。
2. 无共建归属时，共建30%归入平台营销费账户。
3. 销售归属门店30%根据订单销售归属关系确定。
4. 无销售归属门店时，该30%归入平台营销费账户。
5. 平台营销费40%为基础份额，始终归入平台营销费账户。

⸻

5.4 总部交付成本

980卡总部交付成本固定为120元。

总部交付成本 = 120元

系统规则：

1. 支付成功后立即计提总部交付成本。
2. 总部交付成本不参与后续核销结算。
3. 总部交付成本的收款账户可配置为总部账户或平台指定账户。

⸻

5.5 项目结算金额

项目结算金额 = 980 - 14.7 - 98 - 120 = 747.3元

该金额进入待核销释放池。

980卡为10次套餐，因此：

单次核销释放金额 = 747.3 ÷ 10 = 74.73元

六、980卡核销阶段结算规则

6.1 核销结算比例

每次核销释放74.73元，并拆分为：

结算对象	收入名目	比例	单次金额
服务门店对应收款主体	履约服务费	65%	48.57元
销售归属门店	销售服务费	35%	26.16元
合计	单次项目结算金额	100%	74.73元

计算公式：

履约服务费 = 74.73 × 65% = 48.5745 ≈ 48.57元
销售服务费 = 74.73 × 35% = 26.1555 ≈ 26.16元

尾差处理：

1. 系统底层建议按分为单位计算。
2. 前N-1次按规则四舍五入。
3. 最后一次核销自动补齐尾差。
4. 确保10次核销完成后，累计释放金额严格等于747.3元。

八、功能模块二：服务门店关联配置

8.1 功能定位

新增「服务门店关联配置」，用于维护系统门店账号与履约服务费收款主体之间的关系。

8.5 关联规则

1. 服务门店必须选择已有系统门店账号。
2. 同一个系统门店账号，同一时间只能存在一条启用中的服务门店关联配置。
3. 服务门店关联配置启用前，必须配置履约收款主体和收款钱包。
4. 履约收款主体可以是：
    * 门店账户；
    * 合伙人账户；
    * 平台指定账户。
5. 若履约收款主体选择合伙人账户，则该服务门店核销产生的履约服务费入账到对应合伙人名下。
6. 即使收入入账到合伙人名下，收入名目仍必须记录为「履约服务费」。
7. 服务门店关联配置只决定履约服务费收款主体，不影响销售服务费收款方。

十、核销成功处理逻辑

10.1 触发条件

用户在系统门店账号下核销已绑定团购套餐结算规则的套餐权益。

10.2 系统处理流程

用户核销套餐
→ 获取本次核销的系统门店账号
→ 判断该套餐订单是否存在待核销释放池
→ 查询服务门店关联配置
→ 获取履约服务费收款主体
→ 计算本次释放金额
→ 计算履约服务费
→ 计算销售服务费
→ 生成履约服务费结算记录
→ 生成销售服务费结算记录
→ 更新待核销释放池

⸻

10.3 本次释放金额计算

980卡默认按次数均摊：

本次释放金额 = 项目结算金额 ÷ 总权益次数

即：

747.3 ÷ 10 = 74.73元

最后一次核销需做尾差兜底：

最后一次释放金额 = 项目结算金额 - 已释放金额

⸻

10.4 履约服务费生成规则

履约服务费 = 本次释放金额 × 履约服务费比例

980卡：

履约服务费 = 74.73 × 65% = 48.57元

收款方：

履约服务费收款方 = 本次核销门店对应的服务门店关联配置中的收款主体

⸻

10.5 销售服务费生成规则

销售服务费 = 本次释放金额 × 销售服务费比例

980卡：

销售服务费 = 74.73 × 35% = 26.16元

收款方：

销售服务费收款方 = 订单支付成功时固化的销售归属门店

⸻

十一、异常处理规则

11.1 套餐未绑定结算规则

场景：

用户购买的套餐未绑定启用中的团购套餐结算规则。

处理：

1. 不执行本需求资金拆解逻辑。
2. 按原有套餐购买逻辑处理。
3. 不生成项目结算金额、待核销释放池、履约服务费、销售服务费。

⸻

11.2 核销门店未配置服务门店关联关系

场景：

用户在某系统门店核销，但该门店未配置启用中的服务门店关联配置。

建议处理：

不允许核销，并提示：当前门店未配置履约服务费收款账户，暂不可核销该团购套餐。

原因：

避免核销成功但履约服务费无法找到收款主体，导致后续人工补账。

⸻

11.3 服务门店关联配置已停用

处理：

不允许核销，并提示：当前门店履约收款配置已停用，暂不可核销该团购套餐。

⸻

11.4 履约收款主体不可用

场景：

收款主体被冻结、停用、钱包不可用。

处理：

不允许核销，并提示：当前门店履约收款账户不可用，请联系管理员处理。

⸻

11.5 销售归属门店缺失

场景：

订单支付时未能识别销售归属门店。

处理建议：

1. 支付成功时进入销售归属异常记录；
2. 核销时如销售归属仍缺失，则销售服务费进入异常待处理；
3. 或按系统默认平台账户承接销售服务费。

本期建议：

支付成功时必须固化销售归属门店；无法固化时，进入异常待处理。

⸻

十六、验收标准

1. 可以新增团购套餐结算规则。
2. 可以选择已有套餐进行绑定。
3. 同一套餐同一时间只能启用一套结算规则。
4. 平台技术服务费、商城推广服务费、总部交付成本可配置。

6. 履约服务费与销售服务费比例合计必须等于100%。
7. 规则启用后，新支付订单可正常套用规则。

十六、余额待分账池

980团购卡所有业务的各方分账出款（平台技术服务费、商城推广服务费、总部交付成本、核销阶段履约服务费、销售服务费），统一通过平台余额待分账池处理。详见《余额待分账池》。

核心机制：

1. 用户下单支付后，拉卡拉自动扣除通道手续费，剩余金额全部进入余额待分账池；
2. 系统账本按订单原始金额足额记账，各方收益金额不受支付方式影响；
3. 实际打款来源均由余额待分账池承接，跨单滚动使用，不依赖单笔订单的拉卡拉可分账额度；
4. 积分、卡金、混合支付场景下的分账出款，同样由余额待分账池兜底。

十八、风险与注意事项

18.1 收入名目混淆风险

履约服务费和销售服务费虽然都可能入账到同一个主体名下，但必须分开记录。

原因：

1. 业务来源不同；
2. 财务核算不同；
3. 后续开票口径不同；
4. 统计分析不同；
5. 退款回退规则不同。

⸻

18.2 历史规则变更风险

结算规则后续可能调整，因此订单支付成功时必须保存规则快照。

不得在核销时重新读取最新规则覆盖历史订单。

⸻

18.3 服务门店收款主体变更风险

服务门店关联配置后续可能修改，因此核销结算记录中必须保存收款主体快照。

历史核销记录不得因配置变更而改变收款主体。

⸻

18.4 跨店核销对账风险

跨店核销会导致销售门店和服务门店不同，因此对账列表必须支持按以下维度筛选：

1. 销售归属门店；
2. 服务门店；
3. 收款主体；
4. 收入名目；
5. 核销记录；
6. 订单记录。

⸻

二十、最终标准口径

980团购套餐卡不是一张独立特殊卡，而是：

在已有套餐基础上，绑定一套团购套餐结算规则。

用户支付成功后：

980元
- 平台技术服务费14.7元
- 商城推广服务费98元
- 总部交付成本120元
= 项目结算金额747.3元

支付成功阶段：

商城推广服务费98元
= 共建29.4元
+ 销售归属门店29.4元
+ 平台营销39.2元

每次核销阶段：

项目结算金额747.3元 ÷ 10次 = 单次释放金额74.73元

每次核销拆分为：

履约服务费 = 74.73 × 65% = 48.57元
销售服务费 = 74.73 × 35% = 26.16元

收款规则：

履约服务费 → 本次服务门店关联配置中的收款主体
销售服务费 → 订单支付时固化的销售归属门店

即使收款主体相同，也必须按不同收入名目分别生成结算记录。
