# 统一分账框架说明

## 一、文档定位

本文档用于定义恒奕分账合规业务中的统一分账框架。

当前不同业务中已经存在多种分钱场景，例如：

1. 1万共建、30万联盟店身份办理业务；
2. 399控制糖卡销售业务；
3. 980团购套餐卡核销业务；
4. 后续其他商城商品、团购套餐、身份权益、服务项目业务。

这些业务表面上规则不同，但底层都可以抽象为：

```text
客户实付金额
→ 按业务规则拆出不同资金模块
→ 每个资金模块内部定义收益名目、接收方、比例、缺位托底
→ 生成真实收益账户、系统虚拟账户、中间待释放池或专项池流水
→ 后续真实出款统一通过余额待分账池或既有提现流程处理
```

本文档只定义统一框架和标准口径，不替换现有 PRD。现有业务 PRD 后续可引用本文档，并在各自业务中补充具体配置。

---

## 二、核心原则

### 1. 先定义钱是什么，再定义钱给谁

分账设计不能只从“给谁分钱”开始，而应先定义每一部分钱的业务含义。

例如，同样是共建拿到的钱，可能来自：

1. 商城推广费中的直推推广佣金；
2. 销售提成中的共建销售提成；
3. 奖金池线下发放；
4. 团购业务中的履约服务费。

这些钱虽然可能进入同一个人的收益账户，但业务名目、财务口径、计算规则、退款冲正规则都不同，必须分开记录。

### 2. 分账计算基于规则池，而不是直接混在整单金额中

每类资金模块都有自己的计算基数。

例如商城推广费不是直接把客户实付金额全部参与分佣，而是先按分销毛利率形成商城推广费池：

```text
商城推广费池 = 客户实付金额 × 分销毛利率
```

然后再在商城推广费池内部拆分给直推、间推和平台推广营销费。

### 3. 业务收益和真实出款分离

系统账本记录的是“应得收益、成本归集、内部对账和待释放金额”，不等于立即真实出款。

真实出款仍由余额待分账池、提现、线下发放或既有财务流程处理。

### 4. 现有分销能力优先复用

商城推广费中的直推、间推能力，应优先沿用现有分销逻辑。

本框架重点补充：

1. 平台推广营销费；
2. 直推、间推缺位后的托底规则；
3. 不同资金模块的统一名目和账本口径。

### 5. 不同业务只启用部分资金模块

不是所有业务都需要全部资金模块。

例如：

1. 399控制糖卡通常包含商城推广费、销售提成、总部成本承担；
2. 980团购套餐卡包含商城推广费、总部成本承担、团购待核销释放金额；
3. 1万共建和30万联盟店包含奖金池、总部成本承担、销售收益、收款商户留存；
4. 某些新业务可能只包含其中两类资金模块。

## 三、整体模型

统一分账框架由六层组成：

```text
第一层：业务事件
第二层：资金模块
第三层：资金模块内部收益名目
第四层：接收方识别规则
第五层：缺位托底规则
第六层：账本账户与真实出款
```

### 1. 业务事件

业务事件是触发分账计算的动作。

| 业务事件 | 说明 | 示例 |
| --- | --- | --- |
| 支付成功 | 客户完成订单支付后触发 | 399控制糖卡、980团购套餐卡 |
| 办理完成 | 员工完成身份业务办理后触发 | 1万共建、30万联盟店 |
| 核销成功 | 用户到门店核销权益后触发 | 980团购套餐卡 |
| 发放确认 | 后台确认奖金发放批次后触发 | 共建奖金池、联盟店奖金池 |
| 退款完成 | 订单发生退款或售后后触发 | 真实收益冲正、系统账户冲正 |

不同业务可有一个或多个业务事件。

980团购套餐卡属于典型的两段式业务：

```text
支付成功阶段：生成商城推广费、总部成本承担、团购待核销释放池
核销成功阶段：从待核销释放池释放履约服务费和履约销售费
```

### 2. 资金模块

资金模块用于定义“这一部分钱是什么”。

| 资金模块 | 定义 | 常见业务 |
| --- | --- | --- |
| 商城推广费 | 基于分销毛利率形成的推广分佣资金池 | 399、980、后续商城商品 |
| 销售提成 | 基于销售成交关系形成的销售收益池 | 399、身份办理类业务 |
| 总部成本承担 | 总部用于技术、商户预留、交付、供应链、运营等成本归集的钱 | 399、980、身份办理类业务 |
| 团购待核销释放金额 | 团购类业务支付后暂未最终分账、等待核销后释放的钱 | 980、其他团购套餐 |
| 奖金池 | 专项奖励池，用于后续周期性或线下发放 | 1万共建、30万联盟店 |
| 收款商户留存 | 线下 POS 或聚合码收款场景中留在收款商户侧的钱 | 1万共建、30万联盟店 |

### 3. 收益名目

收益名目用于定义账本里展示和对账时“这笔钱叫什么”。

同一个接收方可以获得多个收益名目，但不能合并成一条综合收入。

例如共建可能同时获得：

1. 直推推广佣金；
2. 共建销售提成；
3. 奖金池线下奖金。

账本中必须按不同收益名目分别记录。

## 四、资金模块一：商城推广费

### 1. 模块定义

商城推广费是基于分销毛利率从客户实付金额中提取出来的推广分佣资金池。

它用于承接现有分销体系中的直推、间推收益，并新增平台推广营销费口径。

### 2. 计算公式

```text
商城推广费池 = 客户实付金额 × 分销毛利率
```

示例：

```text
客户实付金额 = 100元
分销毛利率 = 10%
商城推广费池 = 100 × 10% = 10元
```

这里的“分销毛利率”可以理解为当前分销体系中用于计算分佣的毛利比例。

### 3. 内部分配

商城推广费池内部通常拆为：

| 收益名目 | 接收方 | 示例比例 | 账户类型 |
| --- | --- | --- | --- |
| 直推推广佣金 | 直接推广人、直接销售的共建 | 30% | 真实收益账户 |
| 间推推广佣金 | 直推上级店家、销售归属店家 | 30% | 真实收益账户 |
| 平台推广营销费 | 平台或总部营销账户 | 40% | 系统虚拟账户 |

示例：

```text
商城推广费池 = 10元
直推推广佣金 = 10 × 30% = 3元
间推推广佣金 = 10 × 30% = 3元
平台推广营销费 = 10 × 40% = 4元
```

### 4. 与现有分销逻辑的关系

商城推广费中的直推和间推，应沿用现有分销能力。

现有分销已经具备：

1. 配置分销毛利率；
2. 配置直推比例；
3. 配置间推比例；
4. 识别直推关系；
5. 识别间推或上级关系；
6. 按规则给直推、间推生成分佣。

因此，统一分账业务中不重复配置分销毛利率、直推比例和间推比例。

统一分账业务只负责引用现有分销规则，并在现有分销基础上补充：

1. 平台推广营销费这一固定系统接收方；
2. 直推、间推缺失时的自动托底规则；
3. 平台推广营销费比例或平台托底规则；
4. 平台推广营销费入账账户；
5. 商城推广费池内部最终归属结果记录。

### 5. 缺位托底规则

当直推或间推不存在时，对应份额进入平台推广营销费。

| 场景 | 直推推广佣金 | 间推推广佣金 | 平台推广营销费 |
| --- | --- | --- | --- |
| 直推、间推都存在 | 30% | 30% | 40% |
| 无直推，有间推 | 0% | 30% | 70% |
| 有直推，无间推 | 30% | 0% | 70% |
| 无直推、无间推 | 0% | 0% | 100% |

托底规则说明：

1. 平台推广营销费基础份额固定存在；
2. 缺失的直推份额归入平台推广营销费；
3. 缺失的间推份额归入平台推广营销费；
4. 商城推广费池内部必须全部分配完，不允许出现未归属金额。

### 6. 配置字段建议

| 字段 | 说明 |
| --- | --- |
| 关联分销规则 | 引用分销模块中已经配置好的毛利率、直推比例、间推比例 |
| 平台推广营销费比例 | 商城推广费池中进入平台推广营销费账户的比例，可在本规则中配置或从分销规则扩展读取 |
| 平台推广营销费账户 | 平台推广营销费对应的系统虚拟账户 |
| 缺失直推托底账户 | 默认平台推广营销费账户 |
| 缺失间推托底账户 | 默认平台推广营销费账户 |
| 比例合计校验 | 读取分销规则中的直推比例、间推比例后，与平台推广营销费比例合计必须等于100% |

配置边界：

1. 分销毛利率不在统一分账业务中配置；
2. 直推比例、间推比例不在统一分账业务中配置；
3. 统一分账业务只负责选择关联分销规则、配置平台推广营销费、配置缺位托底和记录入账结果；
4. 如果后续分销模块支持平台推广营销费配置，也可以由统一分账业务只读取分销规则，不再重复维护比例。

## 五、资金模块二：销售提成

### 1. 模块定义

销售提成是基于销售成交关系形成的销售收益池。

它和商城推广费不同：

1. 商城推广费解决“推广链路怎么分”；
2. 销售提成解决“成交销售收益怎么分”。

同一个人可能同时获得商城推广费和销售提成，但账本名目必须分开。

### 2. 计算公式

```text
销售提成池 = 客户实付金额 × 销售提成比例
```

不同业务可以配置不同销售提成比例。

### 3. 常见分配规则

| 销售场景 | 收益名目 | 接收方 | 分配方式 |
| --- | --- | --- | --- |
| 店家销售 | 店家销售提成 | 店家 | 销售提成池 × 100% |
| 共建销售 | 共建销售提成 | 共建 | 销售提成池 × 配置比例 |
| 共建销售 | 上级店家销售提成 | 共建上级店家 | 销售提成池 × 配置比例 |
| 无销售人 | 销售提成兜底 | 总部兜底账户或异常池 | 按业务配置处理 |

示例：

```text
客户实付金额 = 399元
销售提成比例 = 40%
销售提成池 = 399 × 40% = 159.60元

如果共建销售：
共建销售提成 = 159.60 × 50% = 79.80元
上级店家销售提成 = 159.60 × 50% = 79.80元
```

### 4. 与商城推广费的区别

| 对比项 | 商城推广费 | 销售提成 |
| --- | --- | --- |
| 核心含义 | 推广链路收益 | 销售成交收益 |
| 计算基数 | 客户实付金额 × 分销毛利率形成的推广费池 | 客户实付金额 × 销售提成比例 |
| 常见接收方 | 直推、间推、平台推广营销费 | 共建、上级店家、店家、总部兜底 |
| 是否复用现有分销 | 是 | 可复用销售关系和上级关系 |
| 是否可与商城推广费同时存在 | 可以 | 可以 |

### 5. 缺位处理

销售提成缺位时，不一定全部进入平台推广营销费。

可按业务配置选择：

| 缺位情况 | 处理方式 |
| --- | --- |
| 无销售人 | 进入总部兜底账户，或进入异常待处理 |
| 共建无上级店家 | 共建部分正常入账，上级店家部分进入总部兜底或异常待处理 |
| 销售人身份异常 | 进入异常待处理 |
| 关系失效 | 以订单或办理时固化的关系快照为准 |

## 六、资金模块三：总部成本承担

### 1. 模块定义

总部成本承担是总部用于承接技术服务、商户主体预留、交付履约、供应链、售后、运营等成本的钱。

它不是单一收益名目，而是一个成本承担池。

### 2. 计算方式

总部成本承担可以按比例或固定金额计算。

```text
总部成本承担 = 客户实付金额 × 总部成本承担比例
```

或：

```text
总部成本承担 = 固定金额
```

不同商品、品牌和业务可以配置不同规则。

### 3. 内部拆分

总部成本承担内部可拆为：

| 子项 | 定义 | 账户或处理方式 |
| --- | --- | --- |
| 技术服务费 | 平台或技术公司针对商户、品牌或交易收取的技术服务费用 | 技术服务费账户 |
| 商户主体预留 | 为满足收款商户或拉卡拉分账限制预留的钱 | 资金预留或分账额度占用 |
| 总部实际交付成本 | 总部履约、供应链、商品、物流、运营、售后等实际成本 | 总部交付成本账户 |

公式：

```text
总部成本承担
= 技术服务费
+ 商户主体预留
+ 总部实际交付成本
```

### 4. 技术服务费

技术服务费是平台或技术公司向某个商户、品牌或业务收取的技术服务费用。

它通常和拉卡拉手续费有关，但不是拉卡拉手续费本身。

技术服务费可以按品牌或业务配置：

| 配置项 | 说明 |
| --- | --- |
| 品牌 | 不同品牌可配置不同技术服务费规则 |
| 技术服务费类型 | 比例 / 固定金额 |
| 技术服务费值 | 例如1.5%、2%、固定金额 |
| 计算基数 | 客户实付金额、拉卡拉支付金额或指定金额 |
| 是否抵扣拉卡拉手续费 | 是 / 否 |
| 补差来源 | 通常从总部成本承担中预留 |
| 入账账户 | 技术服务费账户 |

如果配置为按客户实付金额的1.5%收取：

```text
技术服务费总额 = 客户实付金额 × 1.5%
```

如果拉卡拉已经扣除0.38%手续费，则可记录为：

```text
技术服务费总额 = 拉卡拉已扣手续费 + 技术服务费补差
技术服务费补差 = 技术服务费总额 - 拉卡拉已扣手续费
```

示例：

```text
客户实付金额 = 100元
技术服务费率 = 1.5%
拉卡拉手续费率 = 0.38%

技术服务费总额 = 100 × 1.5% = 1.50元
拉卡拉已扣手续费 = 100 × 0.38% = 0.38元
技术服务费补差 = 1.50 - 0.38 = 1.12元
```

### 5. 商户主体预留

商户主体预留是为满足收款商户或拉卡拉分账限制而预留的金额。

它通常不是某个角色赚到的钱，也不一定作为收益名目进入账本。

常见处理方式：

| 项目 | 说明 |
| --- | --- |
| 是否是收益名目 | 通常不是 |
| 是否进入收益账本 | 可不进入，仅作为资金预留或分账额度校验 |
| 计算来源 | 可从总部成本承担中预留 |
| 计算方式 | 按客户实付金额、拉卡拉支付金额或业务配置比例计算 |
| 用途 | 满足支付通道和收款商户分账约束 |

示例：

```text
商户主体预留 = 拉卡拉支付金额 × 预留比例
```

如果业务要求从总部成本承担中预留，则：

```text
总部实际交付成本
= 总部成本承担
- 技术服务费
- 商户主体预留
```

### 6. 总部实际交付成本

总部实际交付成本用于归集总部侧真实承担的业务成本。

包括但不限于：

1. 商品成本；
2. 供应链成本；
3. 仓储成本；
4. 物流履约成本；
5. 服务交付成本；
6. 售后成本；
7. 总部运营成本。

## 七、资金模块四：团购待核销释放金额

### 1. 模块定义

团购待核销释放金额不是最终收益名目。

它是团购类业务在支付成功后暂时还未完成最终分账、等待用户核销后再释放的钱。

适用业务：

1. 980团购套餐卡；
2. 多次套餐卡；
3. 跨店核销套餐；
4. 后续需要按核销释放收益的业务。

### 2. 支付成功阶段

支付成功后，先完成固定资金模块拆分，例如：

```text
客户实付金额
- 商城推广费
- 总部成本承担
= 团购待核销释放金额
```

团购待核销释放金额进入待核销释放池。

它不是给某个人的钱，也不是最终收益名目。

### 3. 核销成功阶段

用户每次核销时，从待核销释放池释放一部分金额，并转化为真正的收益名目。

常见收益名目：

| 收益名目 | 定义 | 接收方 |
| --- | --- | --- |
| 履约服务费 | 实际完成服务履约的一方获得的收入 | 服务门店或其配置收款主体 |
| 履约销售费 | 团购订单销售归属方获得的销售收入 | 订单支付时固化的销售归属门店 |

示例：

```text
团购待核销释放金额 = 747.30元
套餐次数 = 10次
单次释放金额 = 747.30 ÷ 10 = 74.73元

单次释放金额
= 履约服务费
+ 履约销售费
```

### 4. 收款规则

| 收益名目 | 收款规则 |
| --- | --- |
| 履约服务费 | 按本次核销门店对应的服务门店关联配置确定收款主体 |
| 履约销售费 | 按订单支付成功时固化的销售归属门店确定收款主体 |

即使履约服务费和履约销售费最终进入同一个主体，也必须生成两条不同收益名目的账本记录。

### 5. 尾差处理

团购类业务按次数释放时，应按分为单位计算。

建议规则：

1. 前N-1次按配置比例计算并四舍五入；
2. 最后一次按剩余金额兜底；
3. 确保累计释放金额等于待核销释放金额；
4. 不允许多释放或少释放。

## 八、资金模块五：奖金池

### 1. 模块定义

奖金池是专项奖励资金池，通常用于后续周期性发放、线下发放或导入发放明细。

适用业务：

1. 1万共建奖金池；
2. 30万联盟店奖金池；
3. 后续其他身份或市场奖励业务。

### 2. 入账规则

奖金池通常在业务办理完成后入账。

示例：

| 业务 | 奖金池 | 入账规则 |
| --- | --- | --- |
| 1万共建 | 共建奖金池 | 业务金额 × 10% |
| 30万联盟店 | 联盟店奖金池 | 业务金额 × 20% |

奖金池入账后，形成奖金池账本余额，但不代表已经真实发放。

### 3. 支出规则

奖金池支出通常通过线下发放或发放批次确认完成。

```text
奖金池账本余额 = 历史入账总额 - 历史支出总额
```

规则：

1. 每笔入账必须生成账本明细；
2. 每笔支出必须生成账本明细；
3. 支出金额不得大于当前可用账本余额；
4. 发放确认后再计入合伙人线下奖金；
5. 奖金池真实出款可走线下发放或既有财务流程。

## 九、资金模块六：收款商户留存

### 1. 模块定义

收款商户留存是线下 POS 或聚合码收款场景中，扣除需要进入平台收益账本、奖金池、总部成本、销售收益等部分后，留在实际收款商户侧的钱。

适用业务：

1. 1万共建；
2. 30万联盟店；
3. 后续线下收款并由员工办理的业务。

### 2. 处理规则

收款商户留存通常不进入合伙人收益钱包，而是作为收款商户侧留存金额进行财务核对。

同时，POS手续费、信用卡手续费、部分通道费用也可能由收款商户承担。

示例：

```text
业务金额 = 10000元
已配置入账模块合计 = 8900元
收款商户留存 = 1100元
POS手续费由收款商户承担
```

## 十、账户类型

### 1. 真实收益账户

真实收益账户用于承接可以展示给业务主体、后续可提现或结算的收益。

示例：

| 账户 | 承接收益 |
| --- | --- |
| 共建钱包 | 直推推广佣金、共建销售提成 |
| 店家钱包 | 间推推广佣金、店家销售提成、上级店家销售提成 |
| 门店钱包 | 销售服务费、履约销售费 |
| 合伙人钱包 | 履约服务费、销售收益 |

### 2. 系统虚拟账户

系统虚拟账户用于承接平台、总部、技术公司或内部成本归集金额。

它通常不面向商家提现，但需要支持后台查询、对账和报表。

示例：

| 账户 | 承接金额 |
| --- | --- |
| 平台推广营销费账户 | 商城推广费中的平台推广营销费和缺位托底份额 |
| 技术服务费账户 | 技术服务费总额或技术服务费补差 |
| 总部交付成本账户 | 总部实际交付成本 |
| 总部兜底账户 | 销售关系异常、缺位兜底金额 |

### 3. 专项池账户

专项池账户用于承接需要后续发放或释放的专项资金。

示例：

| 账户 | 说明 |
| --- | --- |
| 共建奖金池 | 1万共建奖金池 |
| 联盟店奖金池 | 30万联盟店奖金池 |
| 团购待核销释放池 | 团购支付后等待核销释放的金额 |

### 4. 不进入收益账本的项目

以下项目通常不作为收益名目进入统一收益账本：

| 项目 | 处理方式 |
| --- | --- |
| 拉卡拉手续费 | 记录为支付通道成本或技术服务费抵扣明细 |
| POS手续费 | 记录在支付流水或收款商户承担成本中 |
| 商户主体预留 | 作为资金预留或分账额度校验，不一定生成收益流水 |
| 拉卡拉可分账额度 | 支付通道层校验，不作为收益账户 |

## 十一、统一配置模型

每个业务可按以下结构配置分账规则：

```text
业务分账规则
= 业务事件
+ 资金模块
+ 模块计算方式
+ 模块内部收益名目
+ 接收方识别规则
+ 缺位托底规则
+ 账户类型
+ 规则快照
```

### 1. 业务基础配置

| 字段 | 说明 |
| --- | --- |
| 业务名称 | 例如399控制糖卡、980团购套餐卡、1万共建 |
| 业务类型 | 商品销售、团购套餐、身份办理、服务项目 |
| 关联系统套餐 | 选择系统中已有的套餐、权益包或服务项目 |
| 客户实付金额来源 | 订单实付、办理金额、支付单金额 |
| 触发事件 | 支付成功、办理完成、核销成功 |
| 是否启用分账 | 是 / 否 |
| 规则状态 | 启用 / 停用 |
| 规则版本 | V1、V2等 |

基础配置规则：

1. 每一个分账业务都必须关联系统已有套餐、权益包或服务项目；
2. 统一分账业务不负责新建套餐，只负责给已有套餐绑定分账规则；
3. 同一个套餐在同一时间只能绑定一套启用中的同类型分账规则；
4. 规则调整后只影响新订单、新办理单或新核销记录，历史业务按规则快照处理。

### 2. 资金模块配置

| 字段 | 说明 |
| --- | --- |
| 资金模块名称 | 商城推广费、销售提成、总部成本承担等 |
| 模块定义 | 这部分钱的业务含义 |
| 计算基数 | 客户实付金额、商城推广费池、销售提成池、单次释放金额等 |
| 计算方式 | 固定比例、固定金额、剩余金额、按次数释放 |
| 计算值 | 比例或金额 |
| 是否生成账本 | 是 / 否 |
| 是否为中间池 | 是 / 否 |

### 3. 收益名目配置

| 字段 | 说明 |
| --- | --- |
| 收益名目 | 直推推广佣金、平台推广营销费、履约服务费等 |
| 所属资金模块 | 商城推广费、销售提成、总部成本承担等 |
| 接收方类型 | 共建、店家、平台、总部、服务门店、奖金池 |
| 分配比例 | 在所属资金模块内的比例 |
| 账户类型 | 真实收益账户、系统虚拟账户、专项池账户 |
| 是否前端可见 | 是 / 否 |
| 是否可提现 | 是 / 否 |
| 缺位处理 | 平台托底、总部兜底、异常待处理、阻断业务 |

### 4. 接收方识别规则

| 接收方类型 | 识别方式 |
| --- | --- |
| 直推共建 | 订单或办理单中的直接推广人、直接销售共建 |
| 间推店家 | 直推共建的上级店家，或销售归属门店 |
| 销售人 | 订单销售人、办理员工确认的销售关系 |
| 上级店家 | 共建上级店家或关系快照中的上级 |
| 销售归属门店 | 支付成功时固化的销售归属门店 |
| 服务门店 | 核销发生时的系统门店账号 |
| 履约收款主体 | 服务门店关联配置中的门店、合伙人或平台账户 |
| 平台账户 | 系统固定账户 |
| 总部账户 | 系统固定账户或业务配置账户 |
| 奖金池账户 | 业务绑定的专项奖金池账户 |

### 5. 缺位托底规则

| 缺位类型 | 默认处理 |
| --- | --- |
| 商城推广费无直推 | 对应份额进入平台推广营销费 |
| 商城推广费无间推 | 对应份额进入平台推广营销费 |
| 销售提成无销售人 | 进入总部兜底账户或异常待处理 |
| 共建无上级店家 | 上级部分进入总部兜底或异常待处理 |
| 团购核销无服务门店配置 | 阻断核销，不生成履约服务费 |
| 履约收款主体不可用 | 阻断核销或进入异常待处理 |
| 销售归属门店缺失 | 支付阶段进入异常，或销售服务费进入总部兜底 |

## 十二、新增分账业务交互建议

### 1. 统一入口

建议设置一个统一入口：

```text
分账业务
```

所有会产生分账、收益、成本归集或待释放金额的业务，都放在同一个列表中管理。

列表中通过业务类型区分：

| 业务名称 | 业务类型 | 关联系统套餐 | 触发节点 | 状态 |
| --- | --- | --- | --- | --- |
| 1万共建 | 身份招商业务 | 1万共建权益包 | 办理完成 | 启用 |
| 30万联盟店 | 身份招商业务 | 30万联盟店权益包 | 办理完成 | 启用 |
| 399控制糖卡 | 商品销售业务 | 399家庭膳食纤维日常补给卡 | 支付成功 | 启用 |
| 980团购套餐 | 团购核销业务 | 980团购10次卡 | 支付成功 + 核销成功 | 启用 |

### 2. 新增业务第一步

新增分账业务时，第一步先选择业务类型和关联系统套餐。

```text
新增分账业务
→ 选择业务类型
→ 关联系统已有套餐
→ 按业务类型展示对应配置表单
```

业务类型建议：

| 业务类型 | 适用场景 | 默认展示模块 |
| --- | --- | --- |
| 身份招商业务 | 1万共建、30万联盟店、身份权益购买 | 奖金池、总部成本承担、销售收益、收款商户留存 |
| 商品销售业务 | 399控制糖卡、普通商城商品 | 商城推广费、销售提成、总部成本承担 |
| 团购核销业务 | 980团购套餐、多次卡、跨店核销项目 | 商城推广费、总部成本承担、团购待核销释放金额、履约服务费、履约销售费 |

### 3. 商品销售业务配置表单

商品销售业务中，商城推广费相关规则应引用现有分销模块。

本页面不配置：

1. 分销毛利率；
2. 直推比例；
3. 间推比例。

本页面配置：

1. 关联系统套餐；
2. 关联分销规则；
3. 平台推广营销费账户；
4. 直推缺位托底；
5. 间推缺位托底；
6. 销售提成规则；
7. 总部成本承担规则；
8. 测算预览。

### 4. 团购核销业务配置表单

团购核销业务必须拆成两个阶段展示。

支付成功阶段：

1. 关联系统套餐；
2. 关联分销规则；
3. 商城推广费托底规则；
4. 总部成本承担规则；
5. 团购待核销释放金额计算方式。

核销成功阶段：

1. 履约服务费比例；
2. 履约销售费比例；
3. 服务门店收款主体规则；
4. 销售归属门店规则；
5. 尾差处理规则。

### 5. 身份招商业务配置表单

身份招商业务重点是办理链路和身份权益。

建议展示：

1. 关联系统权益包或套餐；
2. 办理金额；
3. 绑定合伙人身份；
4. 奖金池规则；
5. 总部成本承担规则；
6. 销售收益规则；
7. 收款商户留存规则；
8. 是否生成办理码；
9. 是否需要合同。

### 6. 交互原则

1. 列表统一，表单按业务类型变化；
2. 底层规则统一，页面不展示无关字段；
3. 商城推广费只引用分销规则，不重复配置分销毛利率、直推、间推；
4. 每个业务必须关联已有套餐、权益包或服务项目；
5. 每个业务保存前必须支持测算预览；
6. 测算预览要展示最终入账结果和缺位托底结果。

## 十三、典型业务套用示例

### 1. 399控制糖卡

399控制糖卡可套用以下资金模块：

| 资金模块 | 计算方式 | 内部收益名目 |
| --- | --- | --- |
| 商城推广费 | 客户实付金额 × 分销毛利率 | 直推推广佣金、间推推广佣金、平台推广营销费 |
| 销售提成 | 客户实付金额 × 销售提成比例 | 店家销售提成、共建销售提成、上级店家销售提成 |
| 总部成本承担 | 客户实付金额 × 总部成本承担比例 | 技术服务费、商户主体预留、总部实际交付成本 |

特点：

1. 商城推广费复用现有直推、间推分销逻辑；
2. 新增平台推广营销费和缺位托底；
3. 销售提成独立于商城推广费；
4. 技术服务费从总部成本承担中预留；
5. 商户主体预留从总部成本承担中预留或作为通道约束处理。

### 2. 980团购套餐卡

980团购套餐卡可套用以下资金模块：

支付成功阶段：

| 资金模块 | 计算方式 | 说明 |
| --- | --- | --- |
| 商城推广费 | 客户实付金额 × 分销毛利率 | 分给直推、间推、平台推广营销费 |
| 总部成本承担 | 固定金额或比例 | 技术服务费、商户主体预留、总部实际交付成本 |
| 团购待核销释放金额 | 剩余金额 | 暂不最终分账，等待核销释放 |

核销成功阶段：

| 收益名目 | 计算方式 | 接收方 |
| --- | --- | --- |
| 履约服务费 | 单次释放金额 × 履约服务费比例 | 服务门店关联的收款主体 |
| 履约销售费 | 单次释放金额 × 履约销售费比例 | 支付时固化的销售归属门店 |

特点：

1. 团购待核销释放金额不是收益名目；
2. 真正最终分账发生在核销成功后；
3. 履约服务费和履约销售费必须分别记账；
4. 即使收款主体相同，也不能合并为综合收入；
5. 历史订单必须保存规则快照。

### 3. 1万共建和30万联盟店

身份办理类业务可套用以下资金模块：

| 资金模块 | 说明 |
| --- | --- |
| 奖金池 | 按业务金额进入共建奖金池或联盟店奖金池 |
| 总部成本承担 | 进入总部保本、总部成本或总部交付成本账户 |
| 销售收益 | 按销售关系进入销售方或销售门店收益账户 |
| 收款商户留存 | 剩余金额留在实际收款商户侧 |

特点：

1. 业务事件通常是办理完成；
2. 需要关联 POS 或聚合码支付单；
3. 平台收益账本记录应得收益，不等于立即真实出款；
4. POS手续费、预留金可由收款商户留存部分承担；
5. 后续真实出款继续走余额分账提现或线下发放。

## 十四、账本明细字段建议

每一笔最终收益、成本归集、虚拟账户入账、专项池入账，都应生成账本明细。

| 字段 | 说明 |
| --- | --- |
| 明细ID | 系统唯一流水号 |
| 业务类型 | 商品销售、团购套餐、身份办理等 |
| 业务单号 | 订单号、办理单号、核销单号 |
| 触发事件 | 支付成功、办理完成、核销成功、发放确认 |
| 资金模块 | 商城推广费、销售提成、总部成本承担等 |
| 收益名目 | 直推推广佣金、平台推广营销费、履约服务费等 |
| 计算基数 | 客户实付金额、商城推广费池、单次释放金额等 |
| 计算方式 | 比例、固定金额、剩余金额 |
| 计算比例 | 当前收益名目的分配比例 |
| 金额 | 入账金额，冲正时为负数 |
| 接收方类型 | 共建、店家、总部、平台、服务门店、奖金池 |
| 接收方ID | 对应业务主体ID |
| 接收方名称快照 | 生成时的名称快照 |
| 账户类型 | 真实收益账户、系统虚拟账户、专项池账户 |
| 账户ID | 入账账户 |
| 结算状态 | 待结算、可提现、已提现、已结算、已冲正、异常 |
| 规则版本 | 生成该明细时使用的规则版本 |
| 关系快照 | 直推、间推、销售归属、服务门店等快照 |
| 创建时间 | 系统生成时间 |
| 冲正关联ID | 如为冲正流水，关联原明细 |

## 十五、规则快照要求

所有涉及后续结算、核销、退款、对账的业务，都必须保存规则快照。

保存快照的原因：

1. 后续规则可能调整；
2. 历史订单不能被新规则覆盖；
3. 核销时不能重新读取最新规则影响历史订单；
4. 退款冲正必须按原始收益明细计算；
5. 财务对账需要追溯当时规则。

快照至少包含：

| 字段 | 说明 |
| --- | --- |
| 业务规则ID | 当前使用的规则 |
| 规则版本 | V1、V2等 |
| 客户实付金额 | 本次计算基础 |
| 启用资金模块 | 本业务启用哪些模块 |
| 各模块计算方式 | 比例、固定金额、剩余金额 |
| 各收益名目比例 | 直推、间推、平台推广营销费等 |
| 接收方关系快照 | 直推、间推、销售归属、服务门店 |
| 缺位处理结果 | 是否发生平台托底、总部兜底、异常 |
| 生成时间 | 快照生成时间 |

## 十六、标准口径总结

### 1. 商城推广费

商城推广费是分销毛利率形成的推广费池。

```text
商城推广费池 = 客户实付金额 × 分销毛利率
```

商城推广费池内部再分给：

1. 直推推广佣金；
2. 间推推广佣金；
3. 平台推广营销费。

无直推、无间推时，对应份额进入平台推广营销费。

### 2. 销售提成

销售提成是销售成交收益池。

它可以分给：

1. 共建；
2. 上级店家；
3. 店家；
4. 总部兜底账户。

销售提成和商城推广费是两类不同资金模块，不能混账。

### 3. 总部成本承担

总部成本承担不是单一名目，而是成本承担池。

它内部可拆为：

1. 技术服务费；
2. 商户主体预留；
3. 总部实际交付成本。

技术服务费是平台或技术公司基于品牌、商户或交易收取的服务费用，可按品牌配置。

商户主体预留通常是通道或收款商户约束，不一定作为收益名目。

### 4. 团购待核销释放金额

团购待核销释放金额不是最终收益名目。

它只是支付成功后等待核销释放的中间池。

真正分账发生在核销成功后，转化为：

1. 履约服务费；
2. 履约销售费。

### 5. 奖金池

奖金池是专项池。

它先按业务规则入账，后续通过发放批次、线下发放或财务流程支出。

奖金池余额不等于真实资金账户余额。

### 6. 账本和真实出款

统一收益账本负责记录业务归属、收益名目、成本归集、专项池余额和对账依据。

真实出款继续通过：

1. 余额待分账池；
2. 提现流程；
3. 线下发放；
4. 既有财务流程。

账本入账不等于立即真实打款。
