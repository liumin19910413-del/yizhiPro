# 第一部分：现状态（线上已上线）

⸻

一、业务背景

1.1 业务模式

平台引进第三方供应链商家，由供应链商家负责选品、发货、物流全流程，将商品上架到平台商城销售。平台内有运营公司角色，负责活动运营，按比例参与分账。

订单金额按以下顺序依次拆分：先扣除支付通道手续费，再按SKU配置的货款比例分给供应链商家、运营比例分给运营公司，剩余部分为平台毛利（商城营销推广费池），再按分佣规则分配给共建、店家和总部营销推广费。

1.2 现状问题

商城营销推广费池中，共建（直推）30%、店家（间推）30% 已按分佣规则处理，但其中 **40% 的总部营销推广费**部分，目前处理方式不规范：

- 若用户使用现金支付（微信/支付宝/拉卡拉），这 40% 停留在收款商户账户，**未单独拆分记账**；
- 若用户使用积分等非现金支付，收款方侧**没有对应的分账记录**，这部分钱的归属不清晰。

1.3 分账方式

当前全部使用**订单分账**模式：每笔订单支付成功后，按订单维度实时调用拉卡拉分账接口，将各方款项直接打出。

积分支付场景下，因无拉卡拉资金，货款和运营费需从**供应链商家预充的保证金**中扣除。


⸻

二、现状资金拆解规则

2.1 核心设计原则

- 所有比例以**扣除支付通道手续费后的可分配金额**为计算基数
- 货款比例 + 运营比例 + 平台毛利比例 = 100%，按 SKU 维度配置
- 支付方式影响手续费和资金来源，不影响各方应分比例

2.2 名词定义

名词	定义
订单金额	用户在商城购买供应链商品的实付金额
供应链商家	与平台合作、提供选品/发货/物流的第三方供应商
运营公司	平台内负责活动运营的分账主体，按比例参与分账
可分配金额	订单金额扣除支付通道手续费后的金额，作为分账计算基数
货款比例	供应链商家从可分配金额中获得的比例，按 SKU 配置
运营比例	运营公司从可分配金额中获得的比例，按 SKU 配置
平台毛利比例	平台从可分配金额中获得的比例 = 1 - 货款比例 - 运营比例，范围 5%~20%
商城营销推广费池	平台毛利作为佣金池，按分佣规则分配给各角色
保证金	供应链商家预充至平台的资金，用于积分支付时货款和运营费的结算

2.3 整体资金结构

```
订单金额 = 支付通道手续费 + 可分配金额

可分配金额 = 货款 + 运营费 + 商城营销推广费池

商城营销推广费池 = 可分配金额 × 平台毛利比例
```

2.4 各方资金计算公式

项目	计算方式
支付通道手续费	订单金额 × 0.38%（仅现金支付时收取，积分支付为 0）
可分配金额	订单金额 - 支付通道手续费
货款	可分配金额 × 货款比例
运营费	可分配金额 × 运营比例
商城营销推广费池	可分配金额 × 平台毛利比例

2.5 商城营销推广费池分配

分佣角色	比例	账户类型	现状处理
共建（直推）	30%	真实收益账户	正常入账共建钱包
店家（间推）	30%	真实收益账户	正常入账店家钱包
总部营销推广费	40%	——	**现金支付停留收款商户；非现金支付无记账** ← 现状问题所在

缺失角色兜底规则：

销售场景	共建（直推）	店家（间推）	总部营销推广费
共建销售	30%	30%	40%
店家直接销售	—	30%	70%
无销售人	—	—	100%


2.6 计算示例

**示例一：现金支付，订单100元，货款70%、运营5%、毛利25%**

项目	计算方式	金额	去向
支付通道手续费	100 × 0.38%	0.38元	通道成本，不进账本
可分配金额	100 - 0.38	99.62元	—
货款	99.62 × 70%	69.73元	打给供应链商家（订单分账）
运营费	99.62 × 5%	4.98元	打给运营公司（订单分账）
商城营销推广费池	99.62 × 25%	24.91元	二次分配

商城营销推广费池分配（共建销售）：

分佣角色	计算方式	金额	现状去向
共建（直推）	24.91 × 30%	7.47元	共建钱包（正常）
店家（间推）	24.91 × 30%	7.47元	店家钱包（正常）
总部营销推广费	24.91 × 40%	9.96元	**停留收款商户（问题）**

三、现状保证金机制

3.1 用途

供应链商家向平台**一次性预充**保证金，用于积分等非现金支付场景下货款和运营费的结算。

3.2 扣款规则

触发条件	扣款金额
订单使用积分支付成功	货款 + 运营费（以可分配金额为基数）
混合支付（积分+现金）	积分部分对应的货款 + 运营费

3.3 保证金不足处理

1. 支付成功后，系统校验保证金余额是否足以覆盖本单货款 + 运营费；
2. 保证金不足 → **订单挂起，不发货**；
3. 供应链商家补充保证金后 → **系统自动推送，订单恢复**。

⸻

四、现状 SKU 配置规则

4.1 配置维度

每个 SKU 独立配置三个比例，按 SKU 维度管理：

字段	类型	说明
货款比例	数值	供应链商家从可分配金额中获得的比例
运营比例	数值	运营公司从可分配金额中获得的比例
平台毛利比例	自动计算	= 1 - 货款比例 - 运营比例，范围 5%~20%

4.2 配置规则

1. 货款比例 + 运营比例 + 平台毛利比例 = 100%，系统强制校验；
2. 同一 SKU 同一时间只能有一套启用中的规则；
3. 规则修改后只影响新订单，不影响历史订单；
4. 支付成功时保存规则快照。

⸻

五、现状通道费用

- 现金支付：拉卡拉手续费 0.38%，不进入统一钱包账本
- 积分/非现金支付：无手续费
- 本业务**无技术服务费**（不同于本地商家的 1.5%）

# 第二部分：目标态（未来改造方向）

⸻

一、改造目标概述

本次改造有两个核心目标：

**目标1：分账模式从"订单分账"升级为"余额分账"**

将现有订单维度的实时分账模式，改造为与本地商家、399卡、980卡一致的余额待分账池模式。所有支付方式的款项统一进入余额待分账池，系统账本足额记账，实际出款由资金池统一承接。

改造后，供应链商家**不再需要预充保证金**，积分支付场景下的货款和运营费同样由余额待分账池兜底出款。

**目标2：总部营销推广费独立为系统虚拟账户**

参考 399卡、980卡中"平台营销费账户"的做法，将供应链业务的总部营销推广费 40% 从收款商户中独立拆分出来，在分销模块中建立独立的**系统虚拟账户（总部营销推广费账户）**，所有支付方式下均生成明细记账，实现全链路可追溯。

⸻

二、目标态分账模式：余额分账

2.1 核心机制

参考本地商家业务和 399/980 卡的余额待分账池机制（详见《余额待分账池》）：

1. 用户下单支付后，拉卡拉自动扣除通道手续费（0.38%），**剩余金额全部进入余额待分账池**；
2. 系统账本按订单可分配金额**足额记账**，各方收益不受支付方式影响；
3. 实际出款（货款、运营费、推广佣金）均由**余额待分账池统一承接**，跨单滚动使用；
4. 积分、混合支付场景下，**货款和运营费同样由余额待分账池兜底**，不再依赖供应链商家保证金；
5. 拉卡拉商户主体预留金（10%）留在收款商户，T+N 天后释放补回资金池。

2.2 与现状对比

对比项	现状（订单分账）	目标态（余额分账）
分账触发	单笔订单支付后实时触发	资金进池，按规则统一出款
积分支付货款来源	供应链商家保证金	余额待分账池
供应链商家是否需要充保证金	**是**	**否，取消保证金机制**
资金池依赖	依赖单笔订单拉卡拉可分账额度	跨单滚动，不依赖单笔额度
架构一致性	与其他业务不一致	与本地商家/399/980 统一架构

⸻

3.1 账户定义

3.2 记账规则

五、目标态订单收益生成流程

**阶段一：支付成功时（记账，不出款）**

```
用户购买供应链商品
→ 订单支付成功
→ 读取该 SKU 的分账规则快照
→ 判断支付方式，计算可分配金额
→ 现金支付：扣手续费后剩余资金进入余额待分账池
  积分支付：无资金入池，由余额待分账池后续兜底
→ 按比例计算货款、运营费、平台毛利，生成各方收益明细（状态：待结算）
→ 判断销售人身份，按分佣规则生成推广佣金明细
→ 总部营销推广费生成系统虚拟账户明细
→ 手续费记录在通道成本明细（不进钱包账本）
```

**阶段二：确认收货后（触发实际出款）**

```
用户确认收货（或超时自动确认）
→ 触发余额分账出款
→ 货款从余额待分账池打款给供应链商家
→ 运营费从余额待分账池打款给运营公司
→ 共建/店家推广佣金从待结算转为可提现
→ 总部营销推广费虚拟账户明细完成结算
```

> 说明：账本记账在支付时即刻完成，实际资金出款在确认收货后触发，两个动作解耦。

八、目标态补充说明

**1. 货款/运营费出款方式**

确认收货后，系统支持**自动出款**，无需人工触发，从余额待分账池直接打款给供应链商家和运营公司。

**2. 财务对账导出**

总部营销推广费虚拟账户**支持财务对账导出**，可按订单/时间/金额等维度导出明细，供财务团队核对。

**3. 保证金退还（现状→目标态迁移）**

改造完成后，供应链商家已预充的保证金余额，由**财务线下退款**，打款给对应供应链商家，系统记录退款流水。

**4. 混合支付手续费规则**

混合支付（部分积分 + 部分现金）场景：**手续费仅按现金部分收取**，积分部分不收手续费，与现状保持一致。

九、目标态仍待确认项

1. 货款/运营费自动出款的触发时机：确认收货后立即触发，还是 T+N 天后结算？
2. 自动出款失败时的重试和告警机制如何设计？
