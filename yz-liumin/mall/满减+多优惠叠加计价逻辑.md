# 📊 多优惠计价业务逻辑梳理 v2.0（增加满减满赠）

> 版本：v2.0 | 状态：草稿 | 适用业务：线下门店、微商城、秒杀、拼团、积分订单

> ⚠️ **重要说明**：线上满减满赠活动仅支持多优惠电商模式，标准模式暂不可用满减满赠活动。

---

## 一、优惠类型总览

### 1.1 折扣类

| 类型 | 作用对象 | 计算方式 | 备注 |
|---|---|---|---|
| 会员卡折扣 | 所有品 | 原价 × 折扣率 | 改变计价基准，最先执行 |

### 1.2 营销活动类（新增）

| 类型 | 作用对象 | 门槛 | 计算方式 | 分摊方式 |
|---|---|---|---|---|
| 满减满赠活动 | 指定品类+指定订单类型 | 满X元 or 满X件 | 满足门槛后减固定金额或赠商品/券 | 按适用品当前价比例分摊 |

### 1.3 优惠券类（3种）

| 类型 | 作用对象 | 门槛 | 计算方式 | 分摊方式 |
|---|---|---|---|---|
| 项目抵用券 | 绑定指定品/SKU | 无门槛，品存在即可用 | 直接从该品扣（全额或固定金额） | 不分摊，直接作用于绑定品 |
| 现金券 | 指定品类+指定订单类型 | 满X元 or 满X件 | 满足门槛后减固定金额 | 按适用品当前价比例分摊 |
| 折扣券 | 指定品类+指定订单类型 | 满X元 or 满X件 | 满足门槛后打折 | 按适用品当前价比例分摊 |

### 1.4 虚拟资产类

| 类型 | 性质 | 来源 | 使用限制 |
|---|---|---|---|
| 积分 | 平台虚拟积分 | 消费累积 | 有余额门槛、单笔抵扣上限 |
| 赠送金 | 平台赠送资产 | 平台赠送/活动奖励 | 有有效期、适用项目限制、订单类型限制、是否叠加会员折扣开关 |

### 1.5 预支付类（新增）

| 类型 | 性质 | 适用场景 | 说明 |
|---|---|---|---|
| 预定金 | 用户预支付资金 | 仅限秒杀、拼团订单 | 用户为参与特定秒杀/拼团活动而预先支付的资金；有时效性；在预定金活动中配置适用范围 |

### 1.6 支付类

| 类型 | 性质 | 说明 |
|---|---|---|
| 会员卡卡金 | 用户真实储值 | 用户充值的真实资金；可配置适用品类/项目范围；最后消耗；若商品标记为「仅限现金支付」则该商品不可使用卡金 |
| 现金/第三方支付 | 真实资金 | 微信/支付宝/银行卡，兜底支付；亦为「仅限现金支付」商品的唯一可用支付方式 |

## 三、完整计价链（核心）

```plaintext
【订单内商品分类】
将订单内所有商品拆分为两个子集：
  子集A：普通商品（无「仅限现金支付」标记）
  子集B：仅限现金支付商品（有标记）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
子集A（普通商品）完整计价链
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
STEP1  会员折扣
  ├ 判断：订单类型是否开启会员折扣 + 商品是否在折扣适用品范围
  ├ 判断：订单是否使用赠送金 + 赠送金与折扣是否配置为「互斥」
  │   → 若互斥且已勾选赠送金：子集A所有商品折扣率强制 = 1（按原价）
  │   → 否则：各品按折扣率正常计算
  └ 输出：各品折后价（= 原价 × 折扣率，或原价）

STEP1.5 预定金抵扣（仅秒杀、拼团订单）
  ├ 判断：订单类型是否为秒杀或拼团
  ├ 判断：当前秒杀/拼团活动是否在预定金活动的适用范围内
  ├ 判断：用户是否有适用于当前活动的预定金余额
  ├ 判断：预定金是否在有效期内
  ├ 判断：用户是否选择使用预定金（支持开启/关闭）
  ├ 叠加规则检查：
  │   → 若后续满减活动配置「不与预定金共用」：需用户选择使用预定金或满减活动
  │   → 若选择使用预定金且存在冲突：标记满减活动不可用
  ├ 计算可抵扣金额：min(预定金余额, STEP1后的订单金额)
  ├ 按适用品STEP1后价格比例分摊预定金抵扣额
  └ 更新各品 current_price，为后续优惠计算提供基准

STEP2  满减满赠活动（新增）
  ├ 前置检查：若STEP1.5使用了预定金且活动配置「不与预定金共用」：跳过此活动
  ├ 获取订单适用的满减满赠活动
  ├ **重要限制：一个订单只能参与一个满减满赠活动**
  │   → 如有多个符合条件的活动，按以下规则选择：
  │   → 优先级1：用户优惠最大的活动
  │   → 优先级2：活动配置的优先级（数字越大优先级越高）
  │   → 优先级3：创建时间最新的活动
  ├ 计算门槛基准：
  │   → 基于原价：sum(各商品原价 × 数量)
  │   → 基于会员价：sum(各商品STEP1后价格) 💡 推荐，不受预定金影响
  │   → 基于预定金后价格：sum(各商品STEP1.5后价格) 💡 若使用了预定金
  ├ 按阶梯匹配（从高到低）：满足最高阶梯执行
  ├ 减免类型处理：
  │   → 满减：按适用品当前价比例分摊减免额
  │   → 满赠商品：添加赠品到订单，价格=0
  │   → 满赠券：发放券到用户券包
  ├ 叠加规则检查：
  │   → 若配置「不与券共用」：标记券不可用
  │   → 若配置「不与积分共用」：标记积分不可用
  │   → 若配置「不与赠送金共用」：标记赠送金不可用
  │   → 若配置「不与预定金共用」：标记预定金不可用（需重新选择）
  └ 更新各品 current_price

STEP3  项目抵用券（若有且未被满减活动禁用）
  ├ 每张券直接作用于其绑定品/SKU
  ├ 全额抵用：该品 current_price = 0，退出分摊池
  ├ 固定金额：该品 current_price -= 抵用金额（最低 = 0）
  └ 检查：current_price = 0 的品移出分摊池

STEP4  现金券 & 折扣券（若有且未被满减活动禁用）
  ├ 按「满金额门槛从高到低」排序执行
  ├ 门槛基准计算：
  │   → 统一基于券前价格：sum(各商品当前current_price)
  │   → 说明：券前价格 = 经过会员折扣、预定金抵扣、满减活动后的价格
  ├ 验证门槛后按「适用品当前价比例」分摊减免额
  └ 更新各品 current_price

STEP5  积分抵扣（若开启且用户选择使用且未被满减活动禁用）
  ├ 验证：用户积分余额 ≥ 最低余额门槛
  ├ 计算可抵金额：min(用户积分/换算比例, 单笔金额上限, 订单应付额×比例上限)
  ├ 按「分摊池内各品当前价比例」分摊积分减免额
  └ 更新各品 current_price

STEP6  赠送金（若有且用户选择使用且未被满减活动禁用）
  ├ 按赠送金适用品范围过滤（全品/白名单/黑名单）
  ├ 计算可用金额：赠送金余额（不超过适用品当前价总和）
  ├ 按「赠送金适用品当前价比例」分摊减免额
  └ 更新各品 current_price

STEP7  卡金支付（若开启且用户选择使用）
  ├ 按卡金适用品范围过滤（全品/白名单/黑名单）
  ├ 计算可用金额：min(卡金余额, 适用品当前价总和)
  ├ 按「卡金适用品当前价比例」分摊卡金支付额
  └ 更新各品 current_price

STEP8  现金支付
  └ 各品 cash_amount = current_price（STEP7后剩余值）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
子集B（仅限现金支付商品）计价链
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
STEP1  会员折扣：✅ 正常执行（若在折扣适用品范围内）
STEP1.5 预定金抵扣：❌ 跳过（仅限现金支付商品不可使用预定金）
STEP2  满减满赠：❌ 跳过
STEP3  项目抵用券：❌ 跳过
STEP4  现金券&折扣券：❌ 跳过
STEP5  积分抵扣：❌ 跳过
STEP6  赠送金：❌ 跳过
STEP7  卡金支付：❌ 跳过
STEP8  现金支付：支付 STEP1 折后价（若折扣不适用则支付原价）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
邮费（独立核算，不进入商品分摊池）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 五、券类详细说明

### 5.1 项目抵用券
- **作用对象**：绑定指定商品/SKU
- **门槛**：无门槛，商品存在即可用
- **计算方式**：直接从该商品扣除（全额或固定金额）
- **分摊方式**：不分摊，直接作用于绑定商品

### 5.2 现金券 & 折扣券
- **作用对象**：指定品类+指定订单类型
- **门槛计算基准**：**固定基于券前价格**
  - 券前价格 = 经过会员折扣、预定金抵扣、满减活动后的价格
  - 计算公式：sum(各适用商品的current_price)
  - ✅ **简化配置**：统一按券前价格计算，无需选择其他基准
- **计算方式**：
  - 现金券：满足门槛后减固定金额
  - 折扣券：满足门槛后打折
- **分摊方式**：按适用商品当前价比例分摊
- **执行顺序**：按满金额门槛从高到低排序执行

### 5.3 券类互斥规则
- **现金券 vs 折扣券**：默认互斥（可配置允许叠加）
- **与满减活动**：满减活动可配置「不与券共用」
- **与预定金**：无直接冲突，按计价链顺序执行

## 六、预定金抵扣详细说明

### 6.1 预定金基本概念
- **定义**：用户为参与特定秒杀/拼团活动而预先支付的资金
- **适用范围**：仅限秒杀订单和拼团订单
- **时效性**：有明确的有效期限制
- **配置方式**：在预定金活动中配置适用的秒杀/拼团活动范围

### 6.2 预定金可用性判断
1. **基础条件**：
   - 订单类型为秒杀或拼团
   - 用户有预定金余额且在有效期内

2. **活动关联性**：
   - **关键限制**：当前秒杀/拼团活动必须在预定金活动的适用范围内
   - 预定金活动后台会设置可适用的秒杀活动和拼团活动清单
   - 只有被设置的活动才能使用预定金，否则不可用

3. **用户选择性**：
   - 用户可以选择使用或不使用预定金
   - 支持在结算页面开启/关闭预定金使用

### 6.3 预定金抵扣逻辑
1. **抵扣计算**：
   ```
   可抵扣金额 = min(预定金余额, STEP1后的订单金额)
   ```

2. **分摊方式**：
   - 按适用商品STEP1后价格比例分摊预定金抵扣额
   - 更新各商品的current_price，为后续优惠计算提供基准

### 6.4 预定金与满减满赠的互斥规则
1. **配置层面**：
   - 满减满赠活动可配置「是否与预定金共用」
   - 若配置为「不与预定金共用」，则两者互斥

2. **执行逻辑**：
   - 若用户已使用预定金，且满减活动不允许共用：跳过满减活动
   - 若满减活动不允许与预定金共用：用户需在两者间选择一个
   - 用户可以取消预定金使用，改为使用满减活动

3. **优先级处理**：
   - 系统应计算两种方案的优惠金额，推荐用户选择优惠更大的方案
   - 最终决策权在用户

### 6.5 预定金与其他优惠的关系
- **执行顺序**：在会员折扣之后，满减满赠活动之前执行
- **叠加规则**：可与券类、积分、赠送金、卡金叠加使用（除非特殊配置）
- **限制条件**：「仅限现金支付」商品不可使用预定金

### 6.6 预定金示例
**场景**：秒杀活动，用户有100元预定金，该秒杀活动在预定金适用范围内
- 商品A：原价200元，会员折扣后180元
- 商品B：原价100元，会员折扣后90元
- STEP1后订单总额：270元

**预定金抵扣**：
- 可抵扣金额：min(100, 270) = 100元
- 商品A分摊：100 × (180/270) = 66.67元
- 商品B分摊：100 × (90/270) = 33.33元
- 抵扣后：商品A = 113.33元，商品B = 56.67元

**互斥场景**：
- 若同时存在满200减50的满减活动，且配置「不与预定金共用」
- 用户需选择：使用预定金优惠100元 vs 使用满减优惠50元
- 系统推荐：选择预定金方案（优惠更大）

## 七、配置管理要点

### 8.1 预定金活动配置
1. **适用活动设置**：
   - 预定金活动需配置可适用的秒杀活动清单
   - 预定金活动需配置可适用的拼团活动清单
   - 只有在清单内的活动才能使用预定金

2. **有效期管理**：
   - 预定金有明确的有效期限制
   - 超期预定金不可使用

### 7.1 满减满赠活动配置
1. **叠加规则配置**：
   - ☐ 不与预定金共用
   - ☐ 不与券共用
   - ☐ 不与积分共用
   - ☐ 不与赠送金共用

2. **门槛计算基准**：
   - 基于原价
   - 基于会员价（推荐）

### 7.2 系统交互设计建议
1. **优惠选择提示**：
   - 当预定金与满减活动互斥时，展示两种方案的优惠金额
   - 推荐用户选择优惠更大的方案
   - 提供一键切换功能

2. **实时计算反馈**：
   - 用户切换预定金使用状态时，实时更新订单金额
   - 清晰显示各项优惠的抵扣明细

3. **错误提示**：
   - 当前活动不在预定金适用范围内时，给出明确提示
   - 预定金余额不足或已过期时，给出相应说明
  其他优惠均不可抵扣邮费
```

## 四、满减满赠活动配置规则（新增）

### 4.1 活动基础配置
```plaintext
满减满赠活动配置 = {

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【基础信息】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  活动名称：[文本]
  活动类型：满减 / 满赠商品 / 满赠券 / 阶梯满减
  活动时间：开始时间 - 结束时间
  活动状态：启用 / 停用

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【门槛与奖励】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  门槛类型：满金额 / 满件数
```

## 五、业务场景示例

### 5.1 完整优惠叠加场景
```plaintext
订单：商品A 500元 + 商品B 800元 = 1300元
用户：金卡会员（8折）
可用优惠：
- 满减活动A：满1000减150（与券共用✅）
- 满减活动B：满800减100（与券不共用❌）
- 现金券：满500减50（门槛基于券前价格）
- 积分：可抵60元

系统选择：活动A优惠更大（150>100），自动选择活动A

计算过程：
STEP1 会员折扣：1300 × 0.8 = 1040元
STEP2 满减活动A：满1000减150 ✅ = 890元
STEP3 现金券：满500减50 ✅（基于890元）= 840元  
STEP4 积分抵扣：抵60元 = 780元

最终：用户实付780元，总优惠520元
```

### 5.2 满减活动自动选择场景
```plaintext
订单：1200元商品，会员8折=960元
可参与满减活动：
- 活动A：满800减120（优先级5）
- 活动B：满900减100（优先级3）  
- 活动C：满1000减80（优先级10）

系统选择逻辑：
1. 按优惠金额排序：A(120) > B(100) > C(80)
2. 选择活动A：满800减120
3. 活动B、C标记为不可用

计算过程：
STEP1 会员折扣：1200 × 0.8 = 960元
STEP2 满减活动A：满800减120 ✅ = 840元
STEP3 其他优惠：正常执行

最终：用户享受最大优惠120元的满减活动
```

### 5.3 满减活动禁用其他优惠场景
```plaintext
订单：同上1300元订单
满减活动配置：满1000减200，但不与券共用❌

计算过程：
STEP1 会员折扣：1300 × 0.8 = 1040元
STEP2 满减活动：满1000减200 ✅ = 840元
STEP3 券类优惠：❌ 被满减活动禁用，跳过
STEP4 积分抵扣：抵60元 = 780元

最终：用户实付780元，但券无法使用
```

## 八、满减活动互斥限制业务规则（新增）


## 九、设计核心原则总览（更新）

```plaintext
【最高优先级规则】
①（最高） 商品级「仅限现金支付」标识高于一切订单级、活动级配置

【满减活动互斥规则】（新增）
② 每个订单只能参与一个满减满赠活动（强制限制）
③ 系统自动选择最优活动，用户可手动切换
④ 满减活动间互斥，不可叠加

【计价顺序原则】  
⑤ 身份权益优先：会员折扣最先执行
⑥ 营销活动次之：满减满赠在券类之前执行
⑦ 个人资产最后：券→积分→赠送金→卡金→现金的顺序

【门槛基准原则】
⑧ 满减活动门槛：可配置基于原价或会员价
⑨ 券类门槛：固定基于券前价格（简化配置）
⑩ 推荐配置：满减基于会员价，券固定基于券前价格

【叠加规则原则】
⑪ 满减活动可配置是否与券、积分、赠送金叠加
⑫ 叠加冲突时，满减活动优先级更高
⑬ 会员折扣与满减活动固定可叠加

【分摊原则】（保持不变）
⑭ 项目抵用券直接作用于绑定品，不分摊给其他品
⑮ 免单品退出分摊池，不参与后续任何计算
⑯ 每种优惠只在"适用范围内的品"中分摊
⑰ 统一使用最大余数法处理分摊尾差，保证分摊无损

【数据原则】（保持不变）
⑱ 每品保存完整的支付明细快照，退款直接读快照，不重算
⑲ 分账只计算真实资金（卡金+现金），积分/赠送金为平台成本
⑳ 所有操作写流水记录，支持审计和对账
㉑ 退款按品级快照原路逆向，不影响其他品
```
