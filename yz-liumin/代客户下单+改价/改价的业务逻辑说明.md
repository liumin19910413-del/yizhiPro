# 改价的业务逻辑说明

## 1. 功能定位

改价是指客户已经在线上下单、订单状态仍为 `待付款` 时，商户后台可对该订单的 `商品还需支付金额` 和 `邮费还需支付金额` 分别进行人工调整，客户在客户端按最新金额完成支付。

该功能只适用于：

- 客户自己在线上下单生成的订单
- 订单状态为 `待付款`

该功能不等同于修改订单商品内容，也不等同于重新跑一遍优惠计算。

本方案明确约定：

`改价调整的是客户最终仍需支付的金额拆分，其中商品与邮费分开调价、合并结算。`

本期交互草图先以你提供的 `微商城订单` 页面作为案例展开，但业务能力本身不是只给微商城订单使用，后续应复用于其他订单类型的待付款订单。

## 2. 功能需求

### 2.1 入口需求

- 商户后台 `PC 端` 订单列表，对客户自己下单的待付款订单提供 `改价` 入口
- 商户后台 `移动端` 订单列表，对客户自己下单的待付款订单提供 `改价` 入口
- 各订单类型沿用各自现有列表页面，仅在 `待付款` 状态订单上新增改价操作

### 2.2 可操作订单范围

- 仅 `客户自己下单` 的订单支持该能力
- 仅 `待付款` 状态支持改价
- 已支付、已关闭、已取消、已发货等状态不可改价

### 2.3 后台可操作按钮

客户自己下单的待付款订单，后台可操作按钮为：

- `改价`
- `删单`

### 2.4 改价能力需求

- 商户可查看订单当前金额构成
- 商户可分别输入 `商品人工调价金额` 与 `邮费人工调价金额`
- 商户可查看每个维度调价后的金额，以及订单调价后的最终还需支付金额
- 商户需填写或选择改价原因
- 提交改价后，订单金额立即更新
- 客户在客户端查看该订单时，应展示最新待支付金额
- 客户支付时，应按最新金额支付
- 若订单本身不存在邮费，则不展示邮费调价项

## 3. 业务逻辑

### 3.1 核心口径

客户下单后，订单已经完成一次商品、营销、优惠、运费等价格计算。

为了避免优惠重算、价格口径混乱、财务与客服难以追溯，本方案明确采用如下规则：

`客户提交订单后，原订单商品明细与已生效优惠结果锁定，不因商家改价而重新计算营销优惠。`

因此，改价不是去改：

- 商品销售价
- 商品明细行金额
- 优惠券结果
- 满减结果
- 会员折扣结果

改价只改：

- `商品还需支付金额`
- `邮费还需支付金额`
- 以及二者汇总后的 `订单还需支付金额`

### 3.2 金额模型与计算公式

建议订单金额展示与计算统一为以下结构：

1. 商品金额
2. 优惠合计
3. 运费
4. 商品当前还需支付金额
5. 邮费当前还需支付金额
6. 商品人工调价金额
7. 邮费人工调价金额
8. 商品调价后还需支付
9. 邮费调价后还需支付
10. 调价后订单还需支付金额

字段定义如下：

- `商品当前还需支付金额`：当前订单中商品部分尚需客户支付的金额
- `邮费当前还需支付金额`：当前订单中邮费部分尚需客户支付的金额
- `商品人工调价金额`：对商品待支付部分做的人工加减值
- `邮费人工调价金额`：对邮费待支付部分做的人工加减值

计算公式为：

`商品调价后还需支付 = 商品当前还需支付金额 + 商品人工调价金额`

`邮费调价后还需支付 = 邮费当前还需支付金额 + 邮费人工调价金额`

`调价后订单还需支付金额 = 商品调价后还需支付 + 邮费调价后还需支付`

其中：

- 调价金额为负数表示减价
- 调价金额为正数表示加价
- 商品与邮费两个维度分别计算，再做最终汇总

业务语义保持一致：

`人工调价不会重算原有优惠，只是在原订单应付基础上对商品部分和邮费部分做人工增减。`

### 3.3 校验规则与交互限制

本期改价明确遵循以下校验规则：

- `商品调价后还需支付金额 >= 0`
- `邮费调价后还需支付金额 >= 0`
- `调价后订单还需支付金额 > 0`

具体限制如下：

- 允许把商品侧单独调成 `0`
- 允许把邮费侧单独调成 `0`
- 但不允许商品与邮费合计后使订单最终 `还需支付金额 = 0`
- 若订单本身不存在邮费，则交互层面不展示邮费调价项，也不允许出现“把邮费调低多少”的操作语义

### 3.4 为什么不直接改商品总价

如果直接修改商品总价，会带来以下问题：

1. 原本已满足的满减、优惠券门槛可能被破坏
2. 订单优惠是否需要重新计算会变得不确定
3. 客户下单时看到的价格明细与后台修改后的价格口径会不一致
4. 财务、客服、客户三方都难以理解金额变化原因

因此，本期方案不建议支持“修改商品行价格”或“重算整单优惠后的新价格”。

### 3.5 优惠处理逻辑

本期改价明确遵循以下规则：

- 客户下单时已生效的优惠券继续保留
- 客户下单时已生效的满减结果继续保留
- 客户下单时已生效的会员折扣继续保留
- 客户下单时形成的商品明细不因改价改变
- 商家改价后，不重新判断是否满足优惠门槛

换句话说：

`优惠结果锁定，改价只改变最终现金应付部分的商品/邮费拆分。`

### 3.6 订单更新逻辑

商家提交改价后，系统应立即更新：

- 商品当前待支付金额
- 邮费当前待支付金额
- 订单当前待支付金额
- 订单的改价标记
- 改价时间
- 改价操作人
- 改价原因
- 商品改价前金额
- 邮费改价前金额
- 商品改价后金额
- 邮费改价后金额
- 商品人工调价金额
- 邮费人工调价金额
- 汇总人工调价金额

支付完成后，系统还应补充完成以下落账处理：

- `邮费实付金额` 直接按调价后的 `邮费还需支付金额` 落账
- `商品实付总额` 直接取支付完成时订单的 `商品还需支付金额`
- `商品实付总额` 需按各商品自身的 `商品还需支付金额占比` 自动等比分配到各商品行
- 该自动分配只作用于商品部分，邮费不参与商品分摊
- 若订单只有单个商品，则该商品承接全部 `商品实付总额`
- 若某商品调价后 `商品还需支付金额 = 0`，则该商品不再参与商品实付金额分摊

建议分配公式明确为：

`商品实付总额 = 支付完成时订单商品还需支付金额`

`单商品分配实付金额 = 商品自身还需支付金额 / 全部商品还需支付金额 × 商品实付总额`

### 3.7 前台展示与转水单口径

客户打开该订单时，应看到最新订单金额。

订单展示侧满足以下要求：

- 最终待支付金额按最新金额展示
- 消费者端订单详情、PC 商城订单列表等页面继续展示 `人工调价`
- 若商品和邮费分别调价，前台可按汇总人工调价展示，同时后台需保留拆分明细，便于追溯
- 若有必要，可展示“订单金额已调整”提示
- 若客户在支付前刷新页面，应拉取最新订单金额

线上订单转水单场景满足以下要求：

- 不把 `人工调价` 作为独立优惠项输出
- 需将人工调价折算进商品价格口径
- 支付完成后，商品行展示金额以等比分配后的商品实付结果为准
- 支付完成后，邮费按最终邮费实付金额单独体现
- 例如：商品原价 `200`，人工调价后成交按 `150`，则转水单时商品原价按 `150` 体现，不再额外展示一条 `人工调价 -50`
- 这样处理是为了不改动既有报表与优惠逻辑

### 3.8 链接与支付逻辑

- 改价后，客户通过原订单详情页或原支付入口进入时，应按最新金额支付
- 若系统已有付款链接，则该链接对应的订单支付金额需同步为最新金额
- 若客户正在支付中，需防止并发导致旧金额被支付
- 支付完成后，系统自动完成商品金额分摊与邮费单独落账，无需人工干预

### 3.9 删单逻辑

- 仅待付款状态允许删单
- 删单需二次确认
- 删单后客户不可再继续支付该订单

## 4. 业务流程

### 4.1 主流程

```text
客户在线上下单
→ 系统生成待付款订单
→ 商户后台在订单列表查看该订单
→ 商户点击改价
→ 查看订单当前金额构成与商品/邮费待支付拆分
→ 分别输入商品人工调价金额、邮费人工调价金额
→ 填写改价原因
→ 系统实时计算每个维度调价后金额与订单调价后还需支付金额
→ 校验最终还需支付金额大于 0
→ 商户确认提交
→ 系统更新订单待支付金额
→ 客户在订单详情页看到最新金额
→ 客户按最新金额完成支付
→ 订单流转到后续正常履约状态
```

### 4.2 金额计算流程

```text
读取原订单商品金额、优惠金额、运费、商品当前待支付金额、邮费当前待支付金额
→ 商户分别输入商品人工调价金额、邮费人工调价金额
→ 系统实时计算商品调价后还需支付、邮费调价后还需支付
→ 汇总计算订单调价后还需支付金额
→ 校验每个维度调价后金额不能小于 0
→ 校验订单调价后还需支付金额必须大于 0
→ 提交成功后写回订单
```

### 4.3 异常流程

#### 4.3.1 非待付款订单改价

```text
商户点击改价
→ 系统发现订单状态不是待付款
→ 阻止改价并提示当前订单不可改价
```

#### 4.3.2 任一维度调价后金额非法

```text
商户输入商品或邮费人工调价金额
→ 系统计算某一维度调价后金额小于 0
→ 阻止提交并提示调整金额不合法
```

#### 4.3.3 调价后整体金额为 0

```text
商户完成商品与邮费调价
→ 系统计算订单最终还需支付金额等于 0
→ 阻止提交并提示订单最终仍需客户支付金额
```

#### 4.3.4 无邮费订单调价

```text
商户打开改价界面
→ 系统识别该订单邮费当前待支付金额为 0 且无邮费项
→ 页面不展示邮费调价输入项
→ 商户仅可调整商品还需支付金额
```

#### 4.3.5 客户支付并发

```text
商户提交改价
→ 系统校验订单支付状态发生变化
→ 若订单已支付或进入支付中
→ 阻止本次改价并提示订单状态已变化
```

## 5. 用户故事

### 5.1 商户后台视角

#### 用户故事 1

作为商户后台操作人员，我希望在订单列表中直接对客户自己下的待付款订单进行改价，这样我可以在议价、补差价、减免零头等场景下快速处理订单。

#### 用户故事 2

作为商户后台操作人员，我希望改价时能看到订单当前金额构成以及商品、邮费分别还需支付多少，这样我能明确这次调整影响的是待支付金额拆分，而不是商品明细。

#### 用户故事 3

作为商户后台操作人员，我希望分别输入商品和邮费的人工调价金额后，系统能立即算出每个维度调价后的金额及最终应付金额，这样我能避免手工算错。

#### 用户故事 4

作为商户后台操作人员，我希望系统保留改价原因、改价前后金额和操作记录，这样后续客服、财务和运营都能追溯。

### 5.2 客户视角

#### 用户故事 5

作为客户，我希望商家改价后，我在订单详情页里看到的是最新金额，这样我不会按旧金额付款。

#### 用户故事 6

作为客户，我希望我之前使用的优惠结果不会因为商家改价而被重新计算或失效，这样我能理解这次变化只是商家对商品应付部分或邮费应付部分做了调整。

6. 验证只调邮费金额时，系统实时显示邮费调价后金额与订单最终还需支付金额
7. 验证同时调商品与邮费时，系统正确汇总最终还需支付金额
8. 验证支持减价场景
9. 验证支持加价场景
10. 验证提交改价后订单金额立即更新
11. 验证客户订单详情显示最新待支付金额
12. 验证客户按最新金额可完成支付
13. 验证订单列表保留 `删单` 能力

### 6.2 规则测试

1. 验证只有 `待付款` 状态允许改价
2. 验证已支付订单不可改价
3. 验证已关闭订单不可改价
4. 验证人工调价不会触发优惠券重新计算
5. 验证人工调价不会触发满减重新计算
6. 验证人工调价不会触发会员折扣重新计算
7. 验证人工调价不会变更商品明细
8. 验证商品调价后金额不能小于 `0`
9. 验证邮费调价后金额不能小于 `0`
10. 验证订单最终还需支付金额必须大于 `0`
11. 验证商品可单独调成 `0`，但邮费仍需支付时可提交
12. 验证邮费可单独调成 `0`，但商品仍需支付时可提交
13. 验证无邮费订单不展示邮费调价项
14. 验证改价前金额、改价后金额、商品/邮费调价金额、改价原因、操作人、操作时间均被正确记录
15. 验证原支付入口进入后支付金额以最新金额为准
16. 验证订单展示侧显示人工调价汇总金额
17. 验证转水单时人工调价折算进商品价格，不作为优惠项输出
18. 验证多商品场景下，支付完成后按各商品还需支付金额占比正确分摊商品实付金额
19. 验证单商品场景下，支付完成后商品实付金额全部落到该商品
20. 验证邮费存在时，邮费实付金额单独落账，不参与商品等比分摊
21. 验证某商品调价后商品还需支付金额为 `0` 时，该商品不参与商品实付金额分摊

### 6.3 异常测试

1. 验证无改价权限的账号不可执行改价
2. 验证客户正在支付中时，后台改价被正确拦截或做并发控制
3. 验证订单状态在改价提交前发生变化时，系统阻止写入旧数据
4. 验证输入超大金额、非法字符、空值时，系统正确拦截
5. 验证网络异常时不会出现部分成功、部分失败导致的脏数据

## 8. 调价日志功能

### 8.1 功能定位

调价日志用于记录商户对订单进行改价操作的完整历史，方便：
- 商家内部客服、财务、运营人员追溯每次改价的明细
- 客户咨询时，客服可快速查看该订单的调价历史
- 财务对账时核对金额变化是否合理

### 8.2 入口规则

- **入口位置**：订单列表操作列，"删单"按钮之后
- **显示条件**：仅对"有调价记录的订单"显示"调价日志"入口
- **未调价订单**：不显示该入口
- **适用端**：PC端、移动端均按此规则处理

### 8.3 数据记录要求

每次改价操作需记录以下信息：

| 字段 | 说明 |

### 8.8 转水单口径说明

对于线上订单转水单场景，金额处理口径与前台展示侧不同，需明确如下：

- 前台订单页、订单列表继续展示 `人工调价`
- 转水单时，不把 `人工调价` 作为独立优惠项输出
- 转水单时，需将人工调价折算进商品价格口径
- 支付完成后，商品部分按各商品 `商品还需支付金额` 占比自动分摊到商品行
- 支付完成后，邮费部分按最终邮费实付金额单独体现
- 例如：某商品原价 `200`，人工调价后成交按 `150`，则转水单时该商品原价按 `150` 体现，不再额外展示 `人工调价 -50`
- 这样可避免影响既有报表统计逻辑与优惠口径
