# Demo 输出边界

本文档用于约束 OpenDesign 可点击 Demo 的输出范围，避免把开发、测试或产品评审说明内容混入真实产品界面。

## 1. 输出目标

第一版 Demo 用于演示商家自营售后系统的真实产品体验，重点跑通：

```text
C 端消费者申请售后
→ B 端商家处理售后
→ C 端消费者查看处理结果
```

Demo 应像真实系统，而不是流程说明文档、培训材料或测试说明页面。

## 2. 只做两类页面

### 2.1 C 端消费者页面

C 端只展示消费者真实需要看到和操作的内容：

- 我的订单
- 订单详情
- 申请退款 / 申请售后入口
- 售后类型选择
- 售后申请表单
- 售后详情
- 退款进度
- 退货物流填写
- 退款成功
- 申请驳回
- 售后关闭

### 2.2 B 端商家后台页面

B 端只展示商家后台真实处理售后需要看到和操作的内容：

- 售后列表
- 售后详情
- 订单与商品信息
- 用户申请原因和凭证
- 审核处理弹窗
- 退款金额调整
- 责任归因
- 退货收货 / 质检
- 补寄处理
- 退款处理
- 操作日志

## 3. 不做的内容

Demo 中不要出现以下内容：

- 开发说明页
- 测试用例页
- PRD 摘要页
- 状态机说明图
- 字段字典页
- 技术架构页
- Demo 使用说明页
- 给内部评审人员看的流程解释卡片
- 直接解释“这个按钮用于演示某某规则”的页面文案

## 4. 规则如何体现

开发和测试需要理解的规则，应藏在真实产品体验里：

- 通过不同售后状态下的按钮显隐体现操作权限。
- 通过 C 端售后详情状态变化体现 B 端处理结果。
- 通过真实退款信息展示商品退款、原单运费、寄回邮费、优惠、积分、余额等金额口径。
- 通过 B 端审核弹窗、退款弹窗、补寄弹窗体现商家需要填写和判断的信息。
- 通过操作日志体现商家后台留痕。
- 通过申请驳回、退款失败、补寄、售后关闭等真实状态体现异常流。

## 5. 优先依据

Demo 输出优先级如下：

1. `PRD/商家自营订单售后C-B联动完整PRD.md` 决定业务流程和状态流转。
2. `设计规范/设计规范.md` 决定双端视觉气质和交互风格。
3. `C端参考图/` 和 `B端参考图/` 用于参考真实页面布局、组件和视觉细节。
4. 如果参考图与 PRD 流程冲突，以 PRD 为准。
