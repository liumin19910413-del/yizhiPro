## 1. 项目上下文包

### 1.1 业务背景

商家自营订单按**产品类型**和**营销方式**两个维度理解：

**产品类型（决定售后链路）：**

| 产品类型 | 履约方式 | 售后链路 |
| --- | --- | --- |
| 商品 - 物流发货 | 需物流发货，消费者收货 | 完整售后（仅退款/退货退款/补寄） |
| 商品 - 自提 | 到店自提 | 轻量退款（仅退款，不寄回） |
| 套餐次卡 | 不发货，资产直达客户账户 | 轻量退款（仅退款，不寄回） |
| 开通卡（储值卡/会员卡） | 不发货，卡金/赠送金进入客户资产 | 轻量退款（仅退款，不寄回） |

**营销方式（不决定售后链路）：**

| 营销方式 | 可搭载的产品 | 对售后的影响 |
| --- | --- | --- |
| 微商城 | 商品、套餐次卡 | 不影响售后流程，影响金额分摊 |
| 拼团 | 商品、套餐次卡 | 成团后按产品类型走售后 |
| 秒杀 | 商品、套餐次卡 | 按秒杀实付金额计算退款 |
| 积分 | 商品、套餐次卡 | 需同时处理积分退回 |
| 预定金 | 预定资格（不发货） | 轻量退款（仅退款，不寄回） |

**核心原则：产品类型决定走哪条售后路径，营销方式只影响金额分摊和特殊规则。**

当前需要统一售后流程，让 C 端消费者、B 端商家客服、移动商家版、仓库、财务、旺店通、分账佣金、水单系统围绕同一套售后单进行处理。

不做的影响：

1. **电商场景基本不可用**：真正跑电商业务的品牌商家，当前售后链路不完善，无法正常支撑日常运营。
2. **分账订单无法退货寄回**：分账订单目前不支持发货商品的退货退款寄回流程，导致分账订单售后能力严重缺失。
3. **云仓退款对接不完善**：对接旺店通云仓后，退款相关流程（重推、补寄发货、物流回传）未打通。
4. **不支持补寄**：电商场景下常见的少件/漏发/错发补寄场景，目前完全不支持。
5. **不支持审核调额和退款复核调额**：客服审批时需要协商调整退款金额、仓库验收后财务退款时需要调额，当前均不支持。
6. **不支持客服后台代创建售后单**：实际运营中客服需要代客户发起售后，当前无此能力。

### 1.2 用户角色

| 角色 | 目标 | 典型操作 | 关注点 |
| --- | --- | --- | --- |
| C端消费者 | 对可售后订单发起退款/退货并查看进度 | 申请退款、填写物流、查看状态、确认补寄收货 | 能否申请、退多少、进度到哪 |
| B端商家客服（电脑端） | 审核售后、代客创建售后单、处理退款退货补寄 | 审核、调额、拒绝、转补寄、代创建 | 金额、责任、凭证、处理留痕 |
| 移动商家版 | 非物流订单快捷退款 | 点击退款一步完成 | 快速处理非物流退款 |
| 仓库 | 退货商品签收验收 | 签收、质检、完好入库、异常记录 | 完好数量、异常数量 |
| 财务 | 退款执行和水单记录 | 操作退款、查看水单 | 退款金额、水单完整性 |
| 系统 | 自动化处理 | 超时关闭、补寄自动确认收货 | 规则准确执行 |

### 1.3 核心流程

入口：

1. C端个人中心「退款/退货」列表。
2. C端订单列表/订单详情。
3. B端电脑端订单列表「申请售后」。
4. B端售后管理工作台「创建售后单」。
5. 移动商家版非物流订单「退款」。

主路径：

```text
判断订单类型和状态
  -> 非物流订单：创建售后单 → 待商家退款 → 退款中 → 售后完成
  -> 物流待发货：创建售后单 → 待商家审批 → 待商家退款 → 退款中 → 售后完成
  -> 物流已发货仅退款：创建售后单 → 待商家审批 → 待商家退款 → 退款中 → 售后完成
  -> 物流已发货退货退款：创建售后单 → 待商家审批 → 待客户寄回 → 待仓库收货 → 待商家退款 → 退款中 → 售后完成
  -> 补寄：创建补寄单 → 待补寄发货 → 待客户收货 → 售后完成
```

结束状态：售后完成 / 售后关闭

### 1.4 业务术语

| 术语 | 定义 |
| --- | --- |
| 产品类型 | 商品（物流发货/自提）、套餐次卡、开通卡（储值卡/会员卡），决定售后链路走向 |
| 营销方式 | 微商城、拼团、秒杀、积分，可搭载商品或套餐次卡，不决定售后链路 |
| 预定金 | 特殊营销活动，卖预定资格，不发货，属于非物流轻量退款 |
| 商品物流发货订单 | 商品类型中需要物流发货的订单，走完整售后链路 |
| 商品自提订单 | 商品类型中到店自提的订单，走轻量退款链路 |
| 仅退款 | 不要求消费者寄回商品的退款 |
| 退货退款 | 消费者寄回商品，仓库验收后退款 |
| 补寄 | B端客服代发起，补发商品替代退款 |
| 负向水单 | 退款成功后写入的退款水单记录 |
| 旺店通重推 | 待发货部分退款后重新推送最新订单给旺店通 |

### 1.5 系统边界

本需求包含：

1. C端售后列表、申请、详情、物流填写、补寄确认收货。
2. B端电脑端订单列表按钮、售后处理全流程、客服代创建。
3. 移动商家版非物流订单快捷退款。
4. 售后状态机（C端状态 + B端状态）。
5. 旺店通重推、补寄推送、物流回传。
6. 负向水单记录。
7. 分账回退失败后台数据记录。
8. 佣金钱包回退（允许负数）。
9. 退货入库（完好部分）和异常记录。
10. 历史售后数据兼容。

本需求不包含：

1. 发货订单移动商家版售后处理。
2. C端主动申请补寄。
3. 换货。
4. 非物流订单部分退款。
5. 异常商品残次库存和出入库。
6. 补寄出入库记录。
7. 分账回退失败页面展示和报表。
8. 退回运费补贴报表、补寄责任方报表。

## 3. 业务结果说明

### 3.1 为什么做

将商家自营订单的退款、退货退款、补寄能力统一为一套标准售后体系，消除 C/B 端流程割裂，让消费者、客服、仓库、财务使用同一套状态和金额语言。

### 3.2 目标指标

| 指标 | 当前状态 | 目标状态 | 验证方式 |
| --- | --- | --- | --- |
| 售后处理时长 | 各端流程割裂 | 缩短申请到完成时间 | 状态耗时统计 |
| 退款准确率 | 金额口径混乱 | 金额拆分清晰可追溯 | 财务对账 |
| C端售后可见性 | 进度不清 | 状态清晰可理解 | 客服咨询量下降 |
| 旺店通同步率 | 部分退款后无重推 | 退款后自动重推 | 重推成功率 |
| 水单完整率 | 部分场景缺水单 | 所有线上退款写水单 | 水单覆盖率检查 |

### 3.3 当前问题

1. **电商场景不可用**：真正跑电商业务的品牌商家，当前售后链路不完善，基本无法正常使用。
2. **分账订单售后缺失**：分账订单无法走退货寄回流程，导致发货商品订单售后能力严重受限。
3. **云仓退款对接不完善**：对接旺店通云仓后，退款重推、补寄发货、物流回传未打通。
4. **不支持补寄**：电商场景常见的少件/漏发/错发补寄，当前不支持。
5. **不支持调额**：客服审核时的协商退款金额调整、仓库验收后财务退款时的调额，均不支持。
6. **不支持客服代创建**：实际运营中客服需要代客户发起售后单，当前无此能力。
7. **非物流和物流订单规则混淆**：缺乏统一状态机，容易把非物流订单误套进物流寄回链路。

### 3.4 验证方式

| 验证类型 | 判断标准 |
| --- | --- |
| 业务验证 | 产品能从入口讲到售后完成/关闭 |
| 用户验证 | C端能看懂状态和金额 |
| 数据验证 | 水单、分账失败记录、旺店通重推记录完整 |
| 流程验证 | 测试能覆盖全部主流程和异常流 |

### 4.1 分类结论

## 5. PRD 主体

### 5.1 需求目标

**用户目标：**

1. C端消费者可从独立售后列表和订单入口发起售后。
2. 消费者可查看售后状态、退款金额、物流和关闭原因。
3. 补寄商品可确认收货，7天后系统自动确认。

**业务目标：**

1. 统一非物流和商品物流订单售后流程。
2. 商品物流订单不区分分账/非分账、不区分营销方式（微商城/拼团/秒杀/积分），统一走售后。
3. 让分账订单也能走退货寄回流程，补齐电商场景缺失能力。
4. 沉淀责任方、运费补贴、分账失败数据。

**产品目标：**

1. C端独立售后列表和完整售后详情。
2. B端电脑端标准售后处理工作台。
3. 移动商家版非物流快捷退款。

**技术目标：**

1. 统一售后状态机，支持多次售后。
2. 旺店通重推和补寄打通。
3. 所有线上退款写负向水单。

### 5.2 范围边界

| 类型 | 内容 |
| --- | --- |
| 本期做 | C端独立售后列表入口；非物流退款；待发货仅退款（全额/部分）；已发货仅退款；退货退款；补寄；B端代创建；旺店通重推和补寄推送；负向水单；分账回退失败后台记录；佣金钱包回退；退货入库和异常记录；历史数据兼容 |
| 本期不做 | 发货订单移动商家版售后；C端主动补寄；换货；非物流部分退款；异常商品残次库存；补寄出入库；分账回退失败页面/报表；退回运费补贴报表；补寄责任方报表 |
| 后续可能做 | 补寄出入库；分账回退失败报表和追回；责任方报表；换货 |
| 明确排除 | C端申请补寄；移动商家版处理发货订单售后 |

### 5.3 业务场景

| 场景编号 | 场景名称 | 触发条件 | 参与角色 | 期望结果 |

| S2 | 待发货全额仅退款 | 物流订单待发货，选择全部商品 | C端/B端 | 退商品金额+原单运费 → 待审批 → 退款完成 |
| S3 | 待发货部分仅退款 | 物流订单待发货，选择部分商品 | C端/B端 | 退所选商品金额，不退运费 → 待审批 → 退款完成 |

| S6 | 补寄 | B端客服判断需补发，原订单未全部退款 | B端客服/云仓 | 补寄发货 → 客户确认收货 → 完成 |
| S7 | 退款失败重试 | 退款渠道失败 | 财务/系统 | 重新退款 → 完成 |
| S8 | 超时关闭 | 待寄回超时未填物流 | 系统 | 自动关闭售后单 |
| S9 | 旺店通重推 | 待发货部分退款 | 系统 | 重新推送最新订单 |

### 5.4 主流程

### 5.6A B端售后详情页信息块与按钮规则

本节用于指导前端按「订单模式 + 售后类型 + 当前状态」渲染 B 端售后详情页。详情页当前不展示处理日志。

#### 判断顺序

1. 判断订单模式：物流发货订单 / 非物流订单。
2. 判断售后类型：仅退款 / 退款退货 / 补寄。
3. 判断当前状态。
4. 根据规则表展示信息块和底部按钮。

核心规则：

1. 非物流订单申请售后后，直接进入「待商家退款」，不进入「待商家审批」。
2. 「待商家退款」状态还没有发起退款，不展示退款记录。
3. 「退款中」「退款失败」都展示「退款记录」；退款失败原因放在退款记录里。
4. 已完成、已关闭等终态统一展示「处理结果」。
5. 非物流订单的信息块也统一叫「申请信息」，不单独命名为「退款申请信息」。

**权限配置规则：**

1. 每个 Tab 的可见性和可操作性均可在后台权限管理中按角色独立配置。

3. 角色有查看权限但无操作权限时，Tab 可见但操作按钮置灰或隐藏。
4. 权限变更实时生效，无需重新登录。
5. 超级管理员默认拥有全部 Tab 的查看和操作权限。

## 7. 测试与验收

### 7.1 测试覆盖点

**主流程：**

1. 非物流订单退款（C端/B端/移动商家版）。
2. 待发货全额仅退款。
3. 待发货部分仅退款。
4. 已发货仅退款。
5. 已发货退货退款（全额/部分）。
6. 补寄（云仓/自有仓）。
7. 退款成功后写水单。

**分支流程：**

1. 商家拒绝申请。
2. 消费者撤销申请。
3. 超时未寄回关闭。
4. 退款失败和重试。
5. 旺店通重推。
6. 补寄7天自动确认收货。
7. 审核环节调额。
8. 退款复核环节调额。
9. 仓库验收后调额。
10. 优惠券未满足整券退回条件时的申请金额计算。
11. 卡金、赠送金、积分按 SKU+数量原路退回。

**异常流程：**

1. 资产已核销不可退。
2. 会员卡余额不足不可退。
3. 申请金额超最大可退。
4. 分账回退失败记录。
5. 佣金钱包退成负数。
6. 旺店通重推失败。
7. 水单写入失败。
8. 同一原订单存在售后中单据时再次申请。
9. 调额为0或超过上限。

**多次售后：**

1. 同一订单多次售后。
2. 售后中数量占用可售后数量。
3. 全部退完后不可继续申请。
4. SKU购买2件，第一次调额退款后，第二次仍按剩余数量原始成交金额计算可退上限。

### 7.3 验收标准

1. 非物流订单退款走完整流程且写水单；C端需商家确认，B端/移动商家版代客申请后直接退款。
2. 物流订单不区分分账/非分账，统一走售后流程。
3. 待发货全额退默认退运费，部分退不退运费。
4. 已发货退款默认不退运费，调整需留痕。
5. 退货退款仓库验收后才能退款。
6. 补寄只能B端发起，7天自动确认。
7. 旺店通部分退款后重推最新订单。
8. 所有线上退款写负向水单。
9. 分账回退失败仅后台记录。
10. 佣金钱包允许负数，无需审批。
11. 一张订单支持多次售后，剩余数量正确计算。
12. 移动商家版只支持非物流退款，发货订单引导电脑端。
13. 申请金额按 SKU+数量和原支付构成计算，优惠券未满足整券退回条件前不计入可申请退款金额。
14. 调额必须原因和凭证齐全，暂不允许调为0。
15. 分账、合伙人佣金、员工业绩按原商品/数量对应的原始分摊结果回退。
13. 历史售后数据可在新详情页打开。

## 10. 金额规则补充

### 10.1 运费退回规则

| 场景 | 运费规则 | 备注 |
| --- | --- | --- |
| 待发货全额退款 | 默认全额退运费 | — |
| 待发货部分退款 | 默认不退运费 | 只退所选 SKU/数量对应商品金额 |
| 已发货仅退款 | 默认不退原单运费 | 商家审核时可记录协商结果 |
| 退货退款 | 默认不退原单运费 | 客户寄回产生的退回邮费只记录，不随订单退款自动退 |
| 补寄 | 不涉及退邮费 | 记录补寄责任方 |

补充规则：

1. 退货寄回邮费只做记录，不随订单退款自动退给客户。
2. 退回邮费可用于后续向物流、仓库、商家、消费者或其他责任方追责。
3. 运费是否补贴客户、补贴多少、责任方是谁，本期只做售后记录沉淀，不进入自动退款主链路。

### 10.2 售后维度与多次售后规则

售后维度按「SKU + 数量」计算，不按整笔订单或整条商品行一次性耗尽。

核心规则：

1. 同一原订单可以产生多条售后单，但同一时间只允许存在一张售后中单据。
2. 若原订单存在未完成、未关闭的售后单，C端和B端均不允许再次发起新售后，只展示售后详情。
3. 售后单关闭、撤销、拒绝或完成后，系统重新计算剩余可售后数量、剩余可申请退款金额和可补寄数量。
4. 某 SKU 购买数量大于 1 时，可分多次发起售后。例如购买 SKU-A 2件，第一次退1件完成后，第二件仍可继续发起售后。
5. 直到原订单内所有 SKU 的可售后数量均为0，才视为该订单不可再发起售后。
6. 每一张售后单详情页永远展示该售后单当次申请的 SKU、数量、金额、原因、凭证和状态，不随后续售后单变化而改写。
7. 每一次进入售后申请页时，都必须基于原订单最新售后结果重新计算：已购数量、已完成售后数量、售后中占用数量、剩余可售后数量、最多可申请退款金额。
8. 已完成退款/退货的 SKU+数量不允许再次申请退款或退货；已完成补寄的数量不扣减可退款数量，但补寄总数量受原购买数量限制。

```text
剩余可售后数量 = 原购买数量 - 已完成退款/退货数量 - 售后中占用数量
剩余可补寄数量 = 原购买数量 - 已完成补寄数量 - 售后中补寄占用数量
```

示例：

```text
原订单：SKU-A × 2件，单件原始成交价 ¥50
第一次售后：退 SKU-A × 1件，商家协商退款 ¥30，售后完成
第二次售后：剩余 SKU-A × 1件仍按该件原始成交价 ¥50 作为可退上限
第二次不能因为第一次只退了 ¥30，就把剩余可退上限理解为 ¥70 或 ¥80
```

### 10.3 申请退款金额与支付构成规则

申请退款金额按客户购买时每个 SKU、每个数量实际支付构成计算。

基础规则：

1. 系统必须把原订单支付金额拆到 SKU+数量维度，记录每一件商品对应的现金、微信、卡金、赠送金、积分、优惠券等支付构成。
2. C端可申请退款金额只包含当前可直接退回给客户的现金类/资产类金额，不包含当前不满足退回条件的优惠券抵扣金额。
3. 客服审核可调金额以该次申请可退现金/资产金额为上限，只能向下调整，不能向上调整。
4. 仓库验收后如需再次调额，以客服一审后的金额为上限，只能向下调整，不能向上调整。
5. 最终退给客户的金额以最后一个有效退款决策点为准：有仓库验收/财务退款复核时，以财务操作退款时金额为准；无寄回验收流程时，以客服审核通过金额为准。
6. C端售后详情、退款结果和退款明细展示的金额，均展示最终与客户协商确认的退款金额。

支付方式处理规则：

| 支付/资产类型 | 是否按 SKU+数量原路回退 | 退款申请金额是否计入 | 特殊规则 |
| --- | --- | --- | --- |
| 微信/现金等现金支付 | 是 | 是 | 按当前退款 SKU+数量对应实付金额计算 |
| 卡金 | 是 | 是 | 按当前退款 SKU+数量对应卡金金额原路退回 |
| 赠送金 | 是 | 是 | 按当前退款 SKU+数量对应赠送金金额原路退回 |
| 积分 | 是 | 是 | 按当前退款 SKU+数量对应积分金额/积分数原路退回 |
| 抵用券 | 条件退回 | 视抵用券分摊口径计算 | 若某一数量完全由抵用券抵扣，优先退现金支付的数量；退到抵用券对应数量时再退抵用券 |
| 折扣券/现金券 | 必须整张券覆盖的所有品项全部退款后才退 | 未满足整券退回条件前不计入可申请退款金额 | 部分退款时不退券，只扣减可申请退款金额 |

现金券/折扣券示例：

```text
原订单：SKU-A × 2件，单价 ¥50，合计 ¥100
支付构成：微信 ¥50 + 现金券 ¥50，现金券分摊到 SKU-A 两件
拆分后：每件 SKU-A = 微信 ¥25 + 现金券 ¥25

客户第一次退 SKU-A × 1件：
最多可申请现金退款 = ¥25
现金券 ¥25 因未满足整张券覆盖品项全部退款，不退给客户，也不计入可申请退款金额
客服可调金额上限 = ¥25，只能向下调
```

抵用券示例：

```text
原订单：SKU-A × 2件，其中一件由抵用券抵扣，一件由现金 ¥50 支付
客户第一次退 SKU-A × 1件：
系统默认优先退现金支付的那一件，最多可申请退款金额 = ¥50

客户第二次退 SKU-A × 1件：
此时退抵用券对应数量，按抵用券规则退回对应券资产
```

### 10.4 调额与留痕规则

| 环节 | 操作人 | 必须条件 | 留痕要求 |
| --- | --- | --- | --- |
| 审核环节调额 | B端客服 | 金额不得高于系统可退上限；暂不允许调为0 | 填写调额原因 + 上传凭证，写入操作日志 |
| 仓库验收后调额 | 仓库/财务按权限 | 金额不得高于客服审批金额；暂不允许调为0 | 填写调额原因 + 上传凭证，写入操作日志 |
| 退款复核环节调额 | B端/财务 | 金额不得高于上一环节确认金额；暂不允许调为0 | 填写调额原因 + 上传凭证，同步C端展示最终金额 |

规则说明：

1. 所有调额必须有原因和凭证，没有原因或凭证不允许提交。
2. 调额只能向下调整，不能超过当前环节的可退上限。
3. 本期调额暂不允许调整为0。
4. 退款中对客户展示的金额，以最终协商金额为准。

### 10.5 分账、合伙人佣金、员工业绩回退规则

分账、合伙人佣金、员工业绩回退按「原商品 SKU+数量」对应的原始分摊结果回退，不按本次实际退款给客户的金额重新均分。

规则：

1. 当前售后退哪个 SKU、退几个数量，就回退这些 SKU+数量在原订单中对应的分账金额、合伙人佣金和员工业绩。
2. 即使客服或财务对客户退款金额做了调额，分账、合伙人佣金、员工业绩仍按原商品/数量对应的原始分摊结果回退。
3. 分账方原先获得多少，就按该 SKU+数量对应的原始分账口径回退多少；不因为客户最终只退了较低金额而重新计算比例。
4. 合伙人佣金同理，按原 SKU+数量对应佣金回退；如扣回后钱包为负数，本期允许负数。
5. 员工业绩/提成同理，按原单分配给哪些员工、每个 SKU+数量对应多少钱进行回退。
6. 若本次退款有扣回合伙人佣金，需要写入佣金负向记录，便于后续对账和追溯。

示例：

```text
原订单：SKU-A × 2件，单件成交价 ¥50
其中某一件拆分为微信 ¥25 + 卡金 ¥25
该件原分账给分账方 ¥5

客户退该件时，客服协商后只退客户 ¥20
分账回退仍按原该件对应分账 ¥5 回退
不能按 ¥20 再重新均分一次
```

### 10.6 水单与资产回退规则

所有线上退款订单类型都需要写负向水单记录。

水单规则：

1. 每次退款成功后，均需向水单系统写入一条负向记录。
2. 退款金额类水单按调额后的最终实际退款金额写入。
3. 现金、微信、卡金、赠送金、积分等资产按退款 SKU+数量对应的原始支付构成原路回退。
4. 折扣券/现金券只有在整张券使用到的所有品项全部退款时，才执行整券退回。
5. 券资产实际退回给客户时，后台按完整券退回；水单呈现层只记录本次售后 SKU+数量对应的券抵扣金额。
6. 部分退款时，如果优惠券尚未满足退回条件，水单不写优惠券退回，只写本次实际退款金额和已发生的资产回退。
7. 水单写入失败不能影响退款结果，但必须记录失败原因并进入补偿或人工排查。

### 10.7 非物流订单退款规则

非物流订单包括自提商品、套餐次卡、会员储值卡/开通卡、预定金等不走寄回链路的订单。

规则：

1. 非物流订单本期只支持全额仅退款，不支持部分退款。
2. C端仍可按原有入口申请退款，但提交前必须校验对应资产是否未核销、是否可退、是否可全额退回。
3. 资产已使用、已核销、已用完或不满足全额退回条件时，不允许C端申请退款。
4. C端申请后进入「待商家确认退款」，商家可确认退款或拒绝退款。
5. B端后台、移动商家版、拼团/积分/秒杀订单后台中原「退款」按钮统一调整为「申请售后」。
6. B端客服对非物流订单点击「申请售后」后，填写弹窗并确认提交，即直接调用原退款流程完成退款动作，不再额外进入待商家退款后再操作一次。
7. 非物流订单线上退款必须写负向水单，并调用原线下套餐、会员储值卡、卡金、赠送金、积分等资产退款/退卡能力。
8. 非物流订单退款也需要回退员工业绩、合伙人佣金、分账等原订单相关权益。

### 10.8 补寄规则

补寄是 B端客服发起的售后类型，C端消费者不能主动申请补寄。

规则：

1. 只有 B端客服可创建补寄售后单。
2. 原订单未全部退款时，允许发起补寄；原订单全部退款后，不允许补寄。
3. 补寄数量不得大于原购买数量；同一原订单可多次补寄，但累计补寄数量不得超过原购买数量。
4. 补寄货值按原订单对应单品单价计算。
5. 补寄不影响分账、不影响员工业绩、不影响合伙人佣金。
6. 本期补寄不做出入库处理。
7. 创建补寄时需要选择履约方式：云仓或商家自有发货。
8. 若门店开启旺店通云仓，B端可选择云仓履约；未开启云仓时不展示云仓选项，默认商家自有发货。
9. 云仓履约：客服创建补寄售后单后，系统直接向云仓发送一条待补寄订单；云仓发货后回传物流信息，售后单进入待客户收货。
10. 商家自有发货：售后单进入待补寄发货，由商家在B端填写物流公司和物流单号，填写完成后进入待客户收货。
11. 待客户收货状态下，客户可手动确认收货；超过7天未确认时系统自动确认收货并完成售后。

### 10.9 退货质检与入库规则

退货质检在仓库签收后进行，按本次售后申请退回的 SKU+数量录入结果。

规则：

1. 质检结果为「全部完好」时，表示本次申请退回的所有物品均完好、可二次销售；确认入库后系统自动生成退货入库单。
2. 质检结果为「部分损耗」时，仓库必须分别填写客户寄回总数量、完好可二次销售数量、异常数量。
3. 部分损耗场景只对完好数量做入库处理，异常数量只记录数据，不做入库。
4. 货品存在异常、破损、严重污染、缺失等情况时，必须上传对应凭证，如破损图片、拍照凭证等。
5. 完好数量 + 异常数量不得超过客户实际寄回数量，也不得超过本次申请退货数量。
6. 仓库验收后的退款调额以客服审批金额为上限，只能向下调，且必须填写原因和凭证。

### 10.10 极速退款规则

极速退款由门店功能参数控制，但本期仅对发货订单生效，对非物流订单不生效。

规则：

1. 非物流订单不适用极速退款开关，按非物流订单退款规则执行。
2. 发货订单待发货状态下，若门店开启极速退款，客户申请退款后系统自动原路退款，不需要客服审批。
3. 发货订单待发货状态下，若门店未开启极速退款，客户申请后进入客服审批流程。
4. 已发货订单、退货退款、补寄等链路不因极速退款开关直接跳过必要审核、寄回或验收流程。
