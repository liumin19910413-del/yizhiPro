商家自营订单售后流程-AI-Native-PRD.md

1. 需求背景

商家自营订单包含两大类：

1. 非物流发货订单：如开充值卡、套餐次卡、会员卡、预定金订单、商品自提订单等。
2. 物流发货订单：如微商城、拼团、秒杀、积分订单等需要商品发货的订单。

当前需要统一商家自营订单的售后流程，让 C 端消费者、B 端商家客服、电脑端后台、移动商家版、仓库、财务、旺店通、分账与佣金回退、水单退款记录等围绕同一套售后单进行处理。

本期重点建设：

1. C 端消费者申请退款/退货退款。
2. B 端电脑端商家后台处理发货订单售后。
3. B 端客服代客户创建售后单。
4. 非物流发货订单退款处理。
5. 物流发货订单仅退款、退货退款、补寄流程。
6. 售后状态、售后详情、订单列表售后信息展示。
7. 旺店通订单重推、补寄发货、物流回传。
8. 分账回退失败记录、佣金钱包回退、负向水单记录。
9. 历史售后数据兼容。

⸻

2. 需求目标

2.1 C 端目标

消费者可以：

1. 从个人中心进入独立的「退款/退货」售后列表。
2. 查看所有售后单，按申请时间倒序展示。
3. 对可售后的订单发起退款或退货退款。
4. 根据订单状态、售后时长、资产核销情况判断是否展示「申请退款/申请售后」按钮。
5. 查看售后当前状态、进度、退款金额、退货物流、补寄物流、拒绝原因、关闭原因。
6. 对补寄商品进行确认收货。
7. 补寄发货 7 天后未确认收货时，由系统自动确认收货。

2.2 B 端目标

商家客服可以：

1. 在电脑端订单列表中通过「申请售后」入口代客户创建售后单。
2. 在原订单存在售后中状态时，通过「售后详情」查看当前售后进度。
3. 审核消费者发起的售后申请。
4. 对退款金额进行协商调整。
5. 处理退货退款、仅退款、补寄。
6. 查看仓库验收结果后，进行退款复核和二次调价。
7. 记录责任方、退回运费责任方、补寄责任方、退回运费补贴记录。
8. 处理分账回退失败、佣金钱包回退等数据记录。
9. 查看售后全链路操作日志。

2.3 移动商家版目标

本期移动商家版只支持：

1. 非物流发货订单退款。
2. 点击退款后，系统模拟客服代客户发起退款单。
3. 同时完成退款确认操作。
4. 退款成功后进入售后完成。
5. 发货订单不在移动商家版处理售后，统一引导到电脑端 B 端后台处理。

4. 订单类型定义

4.1 非物流发货订单

非物流发货订单包括：

1. 开充值卡订单。
2. 套餐次卡订单。
3. 会员卡订单。
4. 预定金订单。
5. 商品自提订单。
6. 其他不需要物流发货、不需要寄回的订单。

4.1.1 非物流订单退款规则

规则项	说明
C 端是否可申请	可申请退款
B 端是否可代申请	可代客户申请退款
移动商家版是否支持	支持非物流退款快捷处理
是否支持部分退款	本期不支持
是否支持全额退款	支持，但需满足可退条件
是否走寄回流程	不走
是否支持补寄	不支持
默认售后类型	仅退款
默认售后原因	我不想要了
创建售后单后状态	待商家退款
退款完成后状态	售后完成
是否写水单	是，写负向水单记录

4.1.2 非物流订单可退判断

非物流订单能否申请退款，需要判断：

1. 是否在可售后时间范围内。
2. 是否未核销。
3. 是否存在可退资产。
4. 是否满足原路退回所需的资产余额条件。
5. 是否已全退。

会员卡特殊规则

如果是会员卡订单，需要判断：

1. 会员卡内资产是否仍然足够退回本笔订单。
2. 如果会员卡内余额/资产已经不足以支持本笔订单原路退回，则不允许申请退款。
3. 已核销或已消耗导致无法完整退回的，不展示申请退款按钮。

套餐次卡/资产订单规则

1. 只要卡被核销过，就不能申请退款。
2. 未核销且在售后期内，才允许申请退款。
3. 本期不支持部分退。

⸻

5. 物流发货订单定义

物流发货订单包括：

1. 微商城发货订单。
2. 拼团发货订单。
3. 秒杀发货订单。
4. 积分发货订单。
5. 其他商品物流发货订单。

5.1 标准化原则

只要是物流发货订单，不管是否分账，不管来源于微商城、拼团、秒杀、积分订单，只要在可售后时间范围内，都按统一售后流程处理。

规则项	说明
是否区分分账	售后流程不区分分账订单和非分账订单
是否区分来源	售后流程不区分微商城、拼团、秒杀、积分
核心判断	是否发货订单、是否在售后期内、当前履约状态是否允许售后
是否生成售后单	必须生成
是否写水单	发生线上退款/退货退款时必须写负向水单
B 端处理端	本期只做电脑端
移动商家版	发货订单不支持移动端售后处理

⸻

6. 售后时长与按钮展示规则

6.1 售后时长来源

售后时长根据商城设置配置。

配置示例：

交易完成/确认收货后 N 天内允许申请售后

6.1.1 C 端按钮展示规则

场景	C 端按钮
在可售后时间内，且有可售后内容	展示申请退款/申请售后
超过可售后时间	不展示申请退款/申请售后
资产已核销	不展示申请退款
会员卡资产不足以原路退回	不展示申请退款
原订单已全部退完	不展示申请退款/申请售后
原订单还有剩余可售后内容	展示申请退款/申请售后

6.1.2 B 端按钮展示规则

场景	B 端按钮
还有可售后内容，且没有售后中状态	展示申请售后
有售后中状态	展示售后详情
原订单已全部退完	不允许继续申请售后
不满足售后时间	不允许申请售后
非物流订单满足退款条件	可申请售后/退款
发货订单移动商家版	不展示直接售后操作，引导电脑端处理

7. 一张原订单多次售后规则

一张原订单支持多次售后，但每次售后必须基于「剩余可售后内容」发起。

7.1 规则

1. 一张原订单可以有多条售后单。
2. 每次售后都要扣减该订单剩余可售后商品/资产。
3. 已退款、已退货、已补寄中的数量不能重复申请。
4. 售后中数量需要占用可售后数量。
5. 如果整张原订单已全部退完，则不能继续申请售后。
6. 如果原订单还有剩余商品/资产，则可以继续申请新的售后。
7. 订单列表需要继续按原逻辑展示具体哪个商品退了多少，是部分退还是全退。

7.2 可售后数量计算

剩余可售后数量 = 原购买数量 - 已完成退款/退货数量 - 售后中占用数量

7.3 可售后金额计算

剩余可售后金额 = 原可退金额 - 已完成退款金额 - 售后中占用退款金额

退货退款	已发货订单	C 端/B 端	是	是	否
补寄	已发货订单	B 端客服	否	否	是

8.1 发起方规则

发起方	支持类型
C 端消费者	仅退款、退货退款
B 端电脑端客服	仅退款、退货退款、补寄
移动商家版	非物流订单退款快捷处理

8.2 补寄限制

1. 补寄只能由 B 端客服代发起。
2. C 端消费者不支持主动申请补寄。
3. 本期不支持换货。
4. 补寄不涉及退款金额。

9. 售后状态模型

9.1 C 端状态

C 端状态	含义
待商家审批	售后申请已提交，等待商家审核
待客户寄回	商家已同意退货退款，等待消费者寄回商品
待商家收货	消费者已寄回，等待商家/仓库验收
待商家退款	等待商家操作退款或系统处理退款
待补寄发货	补寄单已创建，等待发货
待客户收货	补寄已发货，等待消费者确认收货
售后完成	退款完成或补寄完成
售后关闭	拒绝、撤销、超时关闭、退款拒绝等

9.1.1 非物流订单 C 端状态口径

非物流订单无论是消费者申请，还是客服代申请：

1. 创建售后单后，进入「待商家退款」。
2. 退款完成后，进入「售后完成」。
3. 不展示寄回、物流、仓库验收流程。

9.2 B 端状态

B 端状态	说明
待商家审批	待审核消费者或客服代发起的售后
待客户寄回	已同意退货退款，等待客户寄回
待仓库收货	客户已填写物流，等待仓库签收
待商家退款	等待商家/客服/财务操作退款
退款中	已发起退款，等待渠道、分账、佣金、水单等处理结果
退款失败	退款或回退失败，需要记录或重试
待补寄出库	补寄单已创建，等待云仓/自有仓发货
待客户收货	补寄已发货，等待客户确认收货
已完成	退款完成或补寄完成
已关闭	售后终止，不再继续处理

10. 主流程一：非物流订单仅退款

flowchart TD
A[C端消费者申请退款 / B端客服代申请 / 移动商家版点击退款] --> B{是否满足可退条件}
B -->|否| C[不可申请退款/不展示按钮]
B -->|是| D[创建仅退款售后单]
D --> E[待商家退款]
E -->|操作退款| F[退款中]
F -->|退款完成| G[写入负向水单]
G --> H[售后完成]
F -->|退款失败| I[退款失败]
I -->|重新退款| E

10.1 关键规则

1. 非物流发货订单包括开充值卡、套餐次卡、会员卡、预定金订单、商品自提订单等。
2. 不管是消费者申请，还是客服代申请，都是先创建售后单。
3. 创建售后单后，状态统一进入「待商家退款」。
4. 进入待商家退款后，再进行操作退款。
5. 退款完成后进入「售后完成」。
6. 本期不支持非物流订单部分退款。
7. 非物流订单不走退货寄回流程。
8. 售后原因默认「我不想要了」。
9. 退款完成后写入负向水单记录。

10.2 移动商家版特殊规则

移动商家版本期只支持非物流发货订单退款：

1. 点击退款时，系统模拟客服帮客户发起退款单。
2. 同时完成退款确认操作。
3. 系统仍需生成售后单。
4. 售后单状态可按流程记录为：待商家退款 → 退款中 → 售后完成。
5. 如果退款失败，则记录退款失败。
6. 发货订单在移动商家版不支持售后处理，需到电脑端 B 端操作。

11. 主流程二：待发货物流订单仅退款

flowchart TD
A[C端申请退款/B端客服代申请] --> B[选择退款商品]
B --> C{全额退款/部分退款}
C -->|全额退款| D[默认退商品金额+原单运费]
C -->|部分退款| E[默认仅退所选商品金额，不退运费]
D --> F[待商家审批]
E --> F
F -->|商家同意| G[待商家退款]
G -->|操作退款| H[退款中]
H -->|退款完成| I[售后完成]
F -->|商家拒绝| J[售后关闭]

11.1 规则

1. 待发货订单只能申请仅退款。
2. 支持部分退款和全额退款。
3. 必须选择退款商品。
4. 全额退款默认全额退运费。
5. 部分退款默认不退运费。
6. 申请后进入待商家审批。
7. 商家审核通过后进入待商家退款。
8. 商家操作退款后进入退款中。
9. 退款完成后进入售后完成。
10. 退款失败则进入退款失败，允许重新退款。

11.2 旺店通重推规则

如果待发货订单已经推送过旺店通，但发生部分退款：

1. 系统需要根据退款后的剩余商品，重新生成最新订单数据。
2. 将最新订单重新推送给旺店通。
3. 旺店通侧以后续最新订单为准发货。
4. 系统需要记录原推送订单、售后调整后的最新订单、重推时间、重推状态。
5. 如果重推失败，需要记录失败原因，供后台排查。

12. 主流程三：已发货物流订单仅退款

B --> C[选择退款商品]

12.1 规则

1. 已发货订单可申请仅退款。
2. 可根据选择商品形成部分退款或全额退款。
3. 已发货仅退款默认不退原单运费。
4. 如需退运费，需要商家在审核时调整。
5. 商家审核时可协商调价。
6. 调价结果需要同步 C 端展示。
7. 商家同意后进入待商家退款。
8. 退款完成后进入售后完成。

13.1 规则

1. 已发货订单可申请退货退款。
2. 支持部分退货退款和全额退货退款。
3. 退款范围根据客户选择的商品确定。
4. 申请后进入待商家审批。
5. 商家审核通过后，消费者寄回商品。
6. 仓库签收后，对完好部分做退货入库。
7. 异常商品暂不做任何出入库处理，只做系统数据记录。
8. 异常记录用于后续统计、追责和退款复核。
9. 退货退款默认不退原单运费。
10. 需要记录退货责任方、退回运费责任方、补贴退回运费记录。
11. 本期这些责任和补贴数据只做记录，暂不做报表统计。
12. 退款完成后进入售后完成。
13. 线上退货退款需要写负向水单记录。

19. 旺店通打通规则

19.1 发货订单推送

1. 商品发货订单需要推送旺店通。
2. 旺店通完成发货后，需要回传发货物流信息。
3. 售后发起时，需要同步订单最新发货信息。
4. 系统售后判断必须基于最新履约状态。

19.2 待发货部分退款后的订单重推

如果待发货订单已经推送给旺店通，但之后发生部分退款：

1. 系统需要重新计算订单剩余商品。
2. 生成最新订单数据。
3. 将最新订单重新推送给旺店通。
4. 旺店通以后续最新订单为准发货。
5. 原推送订单与重推订单关系需要记录。
6. 如果重推失败，需要记录失败原因和失败时间。
7. 后台需要保留重推记录，方便排查。

19.3 售后发起时同步最新发货信息

1. 发起退款或退货退款时，需要同步最新发货信息。
2. 如果订单已发货，需要基于旺店通回传状态判断是否可仅退款、是否需要退货。
3. 售后单中需要记录旺店通物流信息。

19.4 补寄打通

1. 发起补寄时，需要往旺店通推送补寄订单。
2. 旺店通发货后，需要回传补寄物流信息给系统。
3. C 端售后详情展示补寄物流。
4. B 端售后详情展示补寄发货状态和物流回传状态。

28. 验收标准

28.1 非物流订单

1. 非物流订单包括开充值卡、套餐次卡、会员卡、预定金、商品自提等。
2. 非物流订单本期不支持部分退款。
3. 已核销的资产订单不能申请退款。
4. 会员卡资产不足以原路退回时，不能申请退款。
5. C 端申请后，先创建售后单。
6. B 端客服代申请后，先创建售后单。
7. 售后单创建后进入待商家退款。
8. 操作退款完成后进入售后完成。
9. 非物流订单退款需要写负向水单。
10. 移动商家版只支持非物流订单退款快捷处理。

28.2 物流发货订单

1. 微商城、拼团、秒杀、积分发货订单均支持统一售后流程。
2. 发货订单本期只支持电脑端 B 端处理。
3. 移动商家版不支持发货订单售后操作。
4. 待发货订单只能申请仅退款。
5. 待发货订单支持部分和全额退款。
6. 待发货全额退款默认退运费。
7. 待发货部分退款默认不退运费。
8. 已发货订单支持仅退款和退货退款。
9. 已发货订单根据选择商品判断部分或全额退款。
10. 已发货退款退货默认不退原单运费。
11. 售后申请后进入待商家审批。
12. 商家审核通过后进入待商家退款或待客户寄回。
13. 退款完成后进入售后完成。
14. 所有线上退款/退货退款写负向水单。

28.3 原订单多次售后

1. 一张原订单支持多次售后。
2. 每次售后基于剩余可售后内容。
3. 售后中数量需要占用剩余可售后数量。
4. 原订单全部退完后，不允许继续售后。
5. 原订单列表仍需展示具体哪个商品退了多少。
6. 原订单列表需展示部分退或全退状态。

28.4 B 端订单列表

1. 没有售后中状态且有可售后内容时，展示申请售后。
2. 点击申请售后后，打开客服创建售后单页面/弹窗。
3. 有售后中状态时，展示售后详情。
4. 点击售后详情后，打开当前售后中的售后单详情。
5. 不满足售后条件时，不展示申请售后或置灰并说明原因。

28.5 旺店通

1. 待发货订单已推送旺店通后，如果发生部分退款，需要重新推送最新订单。
2. 最新订单应按退款后的剩余商品生成。
3. 旺店通以后续最新订单为准发货。
4. 需要记录重推成功/失败状态。
5. 补寄订单需要推送旺店通。
6. 旺店通发货后回传补寄物流。

28.6 分账与佣金

1. 分账回退失败只做后台数据记录。
2. 本期不在任何页面展示分账回退失败。
3. 本期不做分账回退失败报表。
4. 佣金钱包允许退成负数。
5. 佣金钱包退成负数不需要财务审批。

28.7 补寄

1. 补寄只能 B 端客服发起。
2. C 端不支持主动申请补寄。
3. 不支持换货。
4. 补寄不涉及退款金额。
5. 补寄记录责任方。
6. 补寄不做出入库记录。
7. 补寄发货后 7 天未确认收货，系统自动确认收货。
8. 自动确认按补寄单发货时间起算。
9. 自动确认后售后完成。

29. 待技术确认点

1. ✅ **已确认**：移动商家版非物流退款快捷处理时，需要完整记录处理轨迹「待商家退款 → 退款中 → 售后完成」三个状态，但操作上只需一步——商家点击退款后，系统自动完成整个流程（创建售后单 → 发起退款 → 退款完成），用户无需多步操作。
