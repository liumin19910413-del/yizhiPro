# 商家自营订单售后 C-B 联动完整 PRD

> 文档定位：本文是商家自营订单售后的端到端产品需求文档，覆盖 C 端消费者从订单列表、订单详情发起售后，到 B 端商家审核、退货、仓库验收、退货入库/异常记录、退款、补寄、财务处理，再回到 C 端进度与结果展示的完整联动流程。
>
> 使用对象：产品、研发、测试、财务/测算、运营、客服、AI 开发辅助。
>
> 来源说明：本文整合《商家自营售后流程-最新版PRD.md》中自营业务规则，以及《消费者C端售后流程完整PRD.md》中适用于自营订单的 C 端真实操作路径。原文档保留为来源与备份，本文作为自营订单售后端到端联动口径。

---

## 1. 需求背景与价值判断

### 1.1 需求性质

本需求属于迭代需求，也是旧能力重构补齐。

不是新增一个孤立退款功能，而是将商家自营订单已有的普通退款、局部退货退款能力，升级为一套覆盖 C 端申请、B 端审核、退款退货、补寄、资产回退、分账冲正、财务报表和结果展示的标准售后体系。

### 1.2 业务问题

当前商家自营售后存在以下问题：

| 问题 | 当前表现 | 影响 |
| --- | --- | --- |
| C/B 端流程割裂 | C 端申请、B 端审核、C 端结果展示缺少统一端到端口径 | 研发、测试需要跨多份文档拼流程 |
| 订单类型边界不清 | 发货商品、自提、套餐/次卡、储值卡、预定金退款规则混在一起 | 容易把非物流订单误套进退货寄回链路 |
| 旧售后能力不完整 | 部分旧开关只覆盖非分账物流发货订单的部分场景 | 分账订单、部分退款、补寄、责任归因难以承接 |
| 金额口径混乱 | 商品退款、原单运费退款、寄回邮费补贴容易混算 | 财务对账、退款解释、报表统计不准确 |
| 责任字段缺失 | 退货责任方、寄回邮费责任方、补寄责任方未系统化记录 | 成本归属和后续追偿没有依据 |
| C 端进度表达不足 | 消费者难以理解当前是待审核、待寄回、待退款还是退款失败 | 客服咨询量增加，用户体验变差 |
| 仓库验收结果未标准化 | 退回货品是否可二次销售、是否异常不入库缺少清晰流程 | 库存账、异常记录和退款判断容易脱节 |
| 财务与分账回退不完整 | 已分账订单退款缺少回退成功、回退失败、不可冲正、垫付记录 | 账实不一致，失败数据无法在数据库沉淀，后续报表和追回缺少数据基础 |
| 商家后台售后管理不够标准 | 原有售后状态流不全面，缺少补寄、责任归属、运费责任、调额等专业化处理节点 | 商家客服处理依赖线下判断，研发和测试难以覆盖完整流程 |

### 1.3 目标用户

| 目标用户 | 遇到的问题 | 本文需要解决的理解点 |
| --- | --- | --- |
| C 端消费者 | 不知道哪些订单能申请、怎么申请、进度到哪、退多少钱 | 订单入口、售后类型、申请页、详情页、结果页 |
| 商家客服 | 审核时无法一次看清商品、金额、责任、凭证、分账状态 | B 端待审核、同意、拒绝、转补寄、调额、调责 |
| 商家仓库/云仓 | 退货寄回后缺少签收质检、退货入库、异常不入库口径 | 待收货、质检、退货入库、异常记录、补寄出库 |
| 财务/测算 | 退款、资产回退、分账冲正、补寄成本难以统计 | 金额拆分、分账冲正、报表字段 |
| 研发 | 页面、接口、状态、字段、权限和日志影响范围不清 | 状态机、数据字段、端到端链路 |
| 测试 | 无法从单文档拆出主流程、异常流、边界流 | 验收场景与测试用例 |
| 运营/管理者 | 无法评估售后效率、责任成本、退款原因分布 | 指标、报表、责任统计 |

### 1.4 目标指标

本期希望改善以下指标：

| 指标 | 目标方向 |
| --- | --- |
| 售后处理时长 | 缩短从申请到审核、退款、完成的整体时间 |
| 退款准确率 | 减少退款金额、运费、资产退回、分账冲正错误 |
| 客服咨询量 | 降低消费者因进度不清、金额不清产生的咨询 |
| 售后通过/拒绝可解释性 | 每次审核、调额、拒绝、关闭都有原因和展示 |
| 财务对账准确率 | 商品退款、原单运费、寄回邮费、分账冲正可追溯 |
| 仓储处理准确率 | 退回商品可二销、破损、少件、空包等结果可追踪 |
| 分账回退失败记录完整率 | 回退失败金额、失败原因、分账方、订单和售后单关系可完整入库 |
| 测试覆盖完整度 | 能覆盖订单类型、订单状态、售后类型和异常边界 |

### 1.5 机会点

本期改进机会集中在以下环节：

1. 订单入口：订单列表、订单详情能够准确展示 `申请退款 / 申请售后 / 售后详情 / 不可申请`。
2. 售后申请：按订单类型和状态展示正确售后类型，避免消费者选错。
3. B 端审核：在待审核阶段集中处理金额、责任、运费、补寄判断。
4. 退货寄回：待寄回、待收货、质检、退货入库、异常记录和退款节点串联清楚。
5. 退款执行：退款金额、资产退回、分账回退成功/失败分开处理。
6. 失败记录：分账回退失败形成数据库明细记录，为后续报表、查询和追回功能预留数据基础。
7. 补寄：少件、漏发、错发等场景可由后台转补寄，不强迫全部走退款。
8. C 端回显：后台每个处理结果都能转成消费者可理解的状态、按钮和结果说明。
9. 商家后台标准化：补齐状态流、处理按钮、中央弹窗、责任归属、运费责任、调额留痕。
10. 财务报表：保留责任、金额、库存、补寄、分账字段，支持后续测算。

### 1.6 方案假设

本方案可能有效的原因：

1. 按“商家自营订单”独立成文，减少读者在 C 端文档和 B 端文档之间切换。
2. 按端到端链路梳理，从 C 端入口开始，到 B 端处理，再回到 C 端结果。
3. 按订单业务形态分层处理，避免把非物流、资产、储值卡、预定金订单误套进发货商品逆向物流。
4. 统一售后主状态和金额口径，让研发、测试、财务使用同一套字段和状态语言。
5. 将已有实现、本期新增、影响范围写在同一份文档，便于评审和拆任务。

### 1.7 验证方式

| 验证方式 | 判断标准 |
| --- | --- |
| 产品评审走查 | 能从订单列表一路讲到退款/补寄完成 |
| AI 拆任务 | AI 只读本文可拆出 C 端、B 端、后端、数据、测试任务 |
| 研发评估 | 研发能识别页面、接口、字段、状态、权限、日志影响 |
| 测试写用例 | 测试能直接写主流程、异常流、边界流 |
| Mock 场景覆盖 | 每个主状态和关键订单类型都有 mock 样例 |
| 关键流程演示 | 能演示仅退款、退货退款、补寄、退款失败、拒绝、撤销 |

## 2. 现状、已有实现与本期新增

### 2.1 已有实现

当前商家自营售后已有以下基础：

1. C 端订单列表和订单详情已有退款/售后入口基础。
2. 自营非物流订单存在普通一键退款链路。
3. 部分非分账物流发货订单已支持全额退款、部分退款、退款退货、消费者寄回、商家/仓库确认收货。
4. C 端已有售后类型页、申请页、售后详情页、填写退货物流页等页面结构。
5. B 端已有订单管理和售后管理基础能力，可查看售后单、处理部分退款审核。
6. 退款结果已具备一定原路退回能力，包括微信/现金、卡金、积分等资产退回基础。

### 2.2 当前不足

1. 已分账物流发货订单缺少完整售后处理。
2. 旧开关语义过窄，只表达“非分账订单部分退款退货”，不适合新能力。
3. 自提、套餐/次卡、储值卡、预定金等退款型订单缺少统一售后闭环描述。
4. 补寄、责任归因、运费承担、库存回补、退货异常记录、分账冲正和报表统计不完整。
5. 分账回退成功和失败缺少明细记录，失败数据无法沉淀为后续报表和追回处理的数据基础。
6. 商家后台售后管理模块不够标准，状态流、处理动作、调额、调责、补寄、运费责任等流程不全面。
7. C 端统一体验与 B 端处理结果之间缺少状态映射。

### 2.3 本期新增/迭代

| 能力 | 本期口径 |
| --- | --- |
| 标准售后模式 | 废弃旧开关准入口径，改为按订单业务形态分层判断售后能力 |
| 发货商品订单完整售后 | 支持仅退款、退款退货、全额、部分、寄回、收货、退款、补寄 |
| 非物流退款型订单闭环 | 自提、套餐/次卡、储值卡、预定金按各自校验规则进入退款型售后 |
| 审核调额 | 支持按配置在待审核或待退款节点调整审批退款金额 |
| 责任归因 | 记录退货责任方、寄回邮费责任方、补寄责任方、补寄运费承担方 |
| 运费拆分 | 商品退款金额、原单运费退款金额、寄回邮费补贴/赔付金额分开 |
| 补寄链路 | 后台审核后可转补寄，并形成补寄出库记录 |
| 仓库验收与库存处理 | 退回货品由仓库判断是否可二次销售；完好可二销数量生成退货入库，破损/污损/缺件只做异常记录不入库 |
| 分账回退记录 | 已分账订单退款时记录应回退、回退成功、回退失败、不可冲正、垫付金额 |
| 分账回退失败数据记录 | 回退失败数据仅在数据库形成明细记录，本期不做报表页面、待追回列表或追回处理功能 |
| 合伙人佣金回退 | 合伙人佣金可正常回退则回退；无法正常回退时扣减合伙人钱包，余额可扣为负数并记录明细 |
| 商家后台标准化 | 售后管理模块补齐状态流、详情卡片、中央弹窗、责任归属、运费责任、调额、补寄、日志 |
| C/B 端联动 | 后台处理结果同步为 C 端状态、按钮、金额、物流和结果展示 |
| Mock 与验收 | 按订单类型、状态、金额、异常覆盖测试场景 |

## 3. 范围与影响范围

### 3.1 本期覆盖的订单业务形态

商家自营线上订单本期按以下业务形态纳入统一售后体系：

| 一级订单类型 | 二级形态 | 订单完成/资产变化 | 本期售后链路 |
| --- | --- | --- | --- |
| 商品订单 | 商品物流发货订单 | 订单发货后进入物流履约，消费者收货后完成 | 仅退款、退款退货、补寄；支持全额、部分、多次售后 |
| 商品订单 | 商品自提订单 | 消费者到店自提，完成自提后交易完成 | 仅退款；未自提前可退，已自提后不可退；只支持全额退款，不走寄回 |
| 套餐/次卡订单 | 线下套餐项目、次卡线上销售 | 支付完成后订单交易完成，对应项目/次数直达客户资产 | 仅退款；资产未使用且在可退款期内可退；只支持全额退款，不走寄回 |
| 开卡充值订单 | 会员储值卡开卡、充值 | 开卡/充值完成后订单交易完成，卡金、赠送金、赠品等进入客户资产 | 仅退款；资产未使用或满足回退条件时可退；只支持全额退款，不走寄回 |
| 预定金订单 | 活动预定金 | 支付后形成预定金权益 | C 端开放退款入口；按权益是否已使用校验，只支持全额退款，不走寄回 |

说明：

1. 商品订单分为 `商品自提订单` 和 `商品物流发货订单` 两类，二者售后路径不同。
2. 套餐/次卡和开卡充值订单支付完成后通常直接进入交易完成，并把项目、次数、卡金、赠送金、赠品等写入客户资产。
3. 非物流订单的 C 端售后链路相对更轻：消费者确认提交后直接进入商家售后管理的待退款，由财务操作退款后完成。
4. 预定金订单本期在 C 端开放退款入口，但操作流复用非物流退款型订单流程，不进入待审核、寄回、仓库验收和补寄链路。

### 3.1A 订单来源与营销类型

商品订单和套餐/次卡订单还会叠加不同订单来源或营销类型。

| 订单来源/营销类型 | 适用订单 | 售后影响 |
| --- | --- | --- |
| 普通微商城订单 | 商品自提、商品物流发货、套餐/次卡 | 按对应业务形态进入售后 |
| 拼团订单 | 商品自提、商品物流发货、套餐/次卡 | 成团后按对应业务形态进入售后；未成团或拼团失败按拼团系统退款规则处理 |
| 秒杀订单 | 商品自提、商品物流发货、套餐/次卡 | 按对应业务形态进入售后，退款金额按秒杀实付和优惠分摊计算 |
| 积分商城订单 | 商品自提、商品物流发货、套餐/次卡 | 按对应业务形态进入售后，需同时处理积分退回 |

原则：

1. 订单来源/营销类型不直接决定是否走寄回链路，真正决定售后主流程的是订单业务形态。
2. 商品物流发货订单进入完整售后体系；商品自提、套餐/次卡、开卡充值、预定金进入轻量退款型售后体系。
3. 营销类型主要影响金额分摊、积分退回、优惠回退和特殊活动限制。

### 3.2 本期暂不展开

1. 国际物流、跨境税费逆向处理。
2. 自动向物流或外部供应商追偿，本期先保证字段和台账可记录，具体外部扣款自动化后续迭代。
3. 复杂残次品定价、维修、报废审批流不在本期展开；本期先完成退货入库和异常留痕的标准流程，不做残次库存和破损出库单。

### 3.3 影响范围

| 角度 | 影响内容 |
| --- | --- |
| C 端页面 | 订单列表、订单详情、售后类型页、申请页、售后详情页、填写物流页、退款/退货列表、结果页 |
| B 端页面 | 售后列表、状态筛选、售后详情、中央弹窗、退款清分账单、分账冲正信息区 |
| 订单状态 | 已有售后中、售后完成、售后关闭、部分退款后的剩余商品状态 |
| 售后状态 | 待审核、待寄回、待收货、待退款、退款中、退款失败、待补寄出库、待买家收货、已完成、已关闭 |
| 退款 | 商品退款、原单运费退款、资产原路退回、退款失败重试 |
| 水单 | 购买成功保留购买销售单；退款成功新增退款单/退款水单，不删除或覆盖原购买水单 |
| 库存 | 退货验收、退货入库、异常不入库、少件/空包不回补、补寄出库 |
| 分账 | 分账回退、回退成功、回退失败、中止未完成分账、不可冲正、垫付记录 |
| 财务数据 | 退款金额、责任方、寄回邮费补贴、补寄成本、分账回退成功/失败记录、合伙人佣金回退、合伙人钱包负数扣减；普通回退失败仅做数据沉淀 |
| 商城设置 | 原先用于限制“非分账订单可退款退货/分账订单不可退款退货”的旧开关本期废弃，不再作为商品物流发货订单是否允许退款退货的判断条件 |
| 权限 | 客服、仓库、财务、管理员、总部、分店操作边界 |
| 日志 | 审核、调额、调责、质检、退款、补寄、关闭都需留痕 |
| Mock 数据 | 订单入口、售后列表、售后详情、金额、原因、异常 |
| 测试用例 | 主流程、异常流、边界流、金额流、权限流 |

## 4. 业务对象与核心规则

### 4.1 售后类型

| 售后类型 | 适用订单 | C 端是否可主动选择 |
| --- | --- | --- |
| 仅退款 | 发货商品、自提、套餐/次卡、储值卡、预定金 | 可按订单类型展示 |
| 退款退货 | 发货商品订单 | 可在已发货/待收货/售后期内选择 |
| 补寄 | 发货商品订单 | C 端不主动申请，由 B 端审核转补寄 |

### 4.2 售后范围

| 范围 | 说明 |
| --- | --- |
| 全额退款 | 覆盖订单当前剩余可售后的全部商品金额 |
| 部分退款 | 覆盖部分商品、部分数量或部分金额；待发货商品物流订单支持部分仅退款 |
| 全额退货退款 | 退回订单当前剩余可售后的全部商品和数量 |
| 部分退货退款 | 退回部分商品或同一商品部分件数 |

### 4.2A 多次售后规则

同一笔订单允许多次发起售后，但每次售后只能基于当前仍可售后的剩余商品、剩余数量或剩余资产。

规则：

1. 每次售后创建时，系统必须计算订单当前剩余可售后范围。
2. 已退款、已退货、已补寄关闭且不再可退的商品或数量，不允许再次申请。
3. 商品物流发货订单可多次申请部分售后，直到剩余可售后数量或金额为 0。
4. 商品自提订单、套餐/次卡、开卡充值等只支持全额退款的订单，一旦已退款或资产已使用，不允许再次申请。
5. 多次售后之间必须保留独立售后单号、申请商品明细、申请金额、审批金额、退款记录和操作日志。
6. 订单详情和售后列表需要能区分同一原订单下的多个售后单，并展示每个售后单的当前状态。
7. 商品物流发货订单待发货时如果只退部分商品或部分数量，退款成功后剩余商品继续保留待发货或进入后续发货履约。

### 4.3B 撤销、超时与重新申请规则

1. `待审核` 状态下，消费者可撤销售后申请；撤销后如仍在售后期内，且仍有可售后商品、数量或金额，可重新发起售后。
2. `待寄回` 状态下，消费者可撤销；若超过寄回截止时间仍未填写物流，系统自动关闭为 `已关闭`，关闭类型记录为 `超时关闭`，并记录超时关闭原因。
3. `待收货` 状态下，消费者已填写物流，原则上不允许自行撤销；如需关闭，由商家客服或管理员按异常流程处理并留痕。
4. `待退款` 和 `退款中` 状态下，消费者不允许撤销，避免退款处理中途状态不一致。
5. `退款失败` 状态下，消费者不允许撤销；由财务在商家账户补足资金或排除渠道异常后重新操作退款，也可由财务/管理员人工关闭并记录原因。
6. `已关闭` 且关闭类型为 `已拒绝` 时，消费者可在售后期内基于剩余可售后范围重新申请；超过售后期不可重新申请。
7. `已完成` 状态不可撤销、不可重复退款；如原订单仍有剩余可售后商品、数量或金额，可对剩余部分再次申请。
8. `已关闭` 且关闭类型为 `已撤销` 或 `超时关闭` 时，如仍满足售后期和可售后范围，可重新申请；重新申请必须生成新的售后单号，不复用已关闭售后单。

### 4.3C 原订单状态与售后状态联动

售后状态不直接覆盖原订单履约状态。原订单保留主状态，同时增加售后标识和商品行级售后状态，用于 C 端展示、B 端筛选和防止重复申请。

总原则：

1. 创建售后单后，原订单增加 `售后中` 标识。
2. 售后中涉及的商品、SKU、数量或金额需要锁定，不允许重复申请。
3. 售后未完成前，原订单不直接改成 `已退款` 或 `已关闭`。
4. 售后拒绝或撤销后，若该订单无其他进行中的售后单，原订单恢复原履约状态，仅保留售后记录。
5. 同一订单存在多个售后单时，只要任一售后单处于进行中状态，原订单展示 `售后中` 或 `部分售后中`。
6. 售后完成后，按全额/部分、退款/退货/补寄结果更新原订单售后标识和商品行状态。

按售后场景联动：

| 售后场景 | 售后中原订单展示 | 售后完成后原订单展示 |
| --- | --- | --- |
| 待发货整单仅退款 | `待发货 / 售后中`，订单不可继续发货 | `已关闭 / 已退款` |
| 待发货部分仅退款 | `待发货 / 部分售后中`，锁定申请退款商品和数量 | 仍为 `待发货`，追加 `部分退款`，剩余商品继续发货 |
| 已发货仅退款 | 保留 `已发货 / 待收货`，追加 `售后中` | 全额退款展示 `售后完成 / 已退款`；部分退款展示 `部分退款` |
| 已发货退货退款 | 保留 `待收货 / 已完成`，追加 `退货退款中` | 全部剩余商品退货退款完成展示 `售后完成 / 已退款`；部分完成展示 `部分退货退款` |
| 补寄 | 保留原履约状态，追加 `补寄中` | 补寄签收或确认后展示 `补寄完成`，原订单不改成退款 |
| 非物流退款型订单 | `售后中 / 待退款` | 退款成功后展示 `已关闭 / 已退款` |

按售后状态联动：

| 售后状态 | 原订单展示建议 | 原订单按钮建议 |
| --- | --- | --- |
| 待审核 | 原状态 + `售后中` | 展示 `售后详情` |
| 待寄回 | 原状态 + `待买家寄回` | 展示 `售后详情 / 填写物流` |
| 待收货 | 原状态 + `退货待商家收货` | 展示 `售后详情 / 查看物流` |
| 待退款 | 原状态 + `待退款` | 展示 `售后详情` |
| 退款中 | 原状态 + `退款中` | 展示 `售后详情` |
| 退款失败 | 原状态 + `退款失败` | 展示 `售后详情`，不恢复申请入口 |
| 待补寄出库 | 原状态 + `补寄处理中` | 展示 `售后详情` |
| 待买家收货 | 原状态 + `补寄待收货` | 展示 `查看补寄物流` |
| 已完成 | 按全额/部分更新 `已退款 / 部分退款 / 退货退款完成 / 补寄完成` | 展示 `售后详情` |
| 已关闭 | 恢复原履约状态，保留 `售后已关闭` 记录，并展示关闭类型 | 售后期内按关闭类型判断是否可重新申请 |

商品行级状态：

| 商品行状态 | 含义 |
| --- | --- |
| 正常可售后 | 该商品和数量没有售后中或已完成退款 |
| 售后中 | 该商品和数量正在售后，不可重复申请 |
| 已退款 | 该商品和数量已退款，不可再次申请 |
| 已退货退款 | 该商品和数量已退货退款，不可再次申请 |
| 部分退款 | 同一商品部分数量或金额已退款，剩余数量仍可售后 |
| 补寄中 | 该商品已转补寄，等待出库或收货 |
| 补寄完成 | 补寄流程完成，不代表原商品已退款 |

建议订单展示模型：

```text
订单主状态：待发货 / 已发货 / 待收货 / 已完成 / 已关闭
售后标识：售后中 / 部分退款 / 退货退款中 / 退款失败 / 补寄中 / 售后完成
商品行状态：正常 / 售后中 / 已退款 / 已退货退款 / 补寄中
```

### 4.4 金额拆分原则

必须分开记录：

1. `商品退款金额`：本次退还消费者商品货款部分。
2. `原单运费退款金额`：是否退回正向订单运费，以及退回多少。
3. `寄回邮费补贴/赔付金额`：消费者寄回商品产生的邮费补贴，不并入商品退款。
4. `实际退款金额`：最终调用退款渠道退回的合计金额。
5. `资产退回明细`：微信/现金、卡金、赠送金、积分、优惠券等。

运费与最多可退金额：

1. 待发货商品物流订单申请仅退款时，原订单运费默认可退，并计入最多可退金额。
2. 待发货部分仅退款后如果订单仍有商品需要发货，原订单运费默认不退；只有本次退款覆盖订单当前剩余全部可售后商品和数量时，才按全额仅退款处理并可退原订单运费。
3. 已发货商品物流订单申请仅退款或退货退款时，C 端默认不退原订单运费。
4. 已发货订单申请页必须明确提示 `商品已发货，原订单运费默认不可退；如有特殊情况，将由商家审核处理` 或同等语义。
5. 商品物流发货订单的退款申请页必须展示 `最多可退金额`，并展示该金额由哪些支付类型组成，例如现金、卡金、赠送金、积分、优惠券状态等，避免消费者或商家误以为退款金额全部通过微信/现金原路退回。
6. 商品物流发货订单的 `申请退款金额` 由消费者填写，系统可默认带出最多可退金额作为建议值，但消费者提交金额不得超过系统计算的最多可退金额。
7. 如商家审核时特殊补退原订单运费，必须在 B 端单独调整并留痕。

### 4.4A 支付资产、优惠与活动权益回退规则

自营线上订单在支付时可能同时使用多种支付资产、优惠和活动权益。退款时必须按商品、SKU、数量维度拆分本次退款对应的支付构成，并按规则原路退回或保留优惠使用状态。

#### 4.4A.1 参与计算的支付与优惠类型

| 类型 | 示例 | 退款处理 |
| --- | --- | --- |
| 现金支付 | 微信、支付宝、银行卡等现金支付 | 按本次退款商品/SKU/数量分摊金额原路退回 |
| 卡金支付 | 会员储值卡卡金 | 按本次退款商品/SKU/数量分摊金额退回卡账户 |
| 赠送金支付 | 会员卡赠送金 | 按本次退款商品/SKU/数量分摊金额退回赠送金账户 |
| 积分抵扣 | 积分商城或订单支付时积分抵扣 | 按本次退款商品/SKU/数量分摊积分退回 |
| 优惠券抵扣 | 满减券、折扣券、商品券等 | 只有该券关联商品全部退款成功时才退回；部分退款不退券 |
| 会员折扣 | 会员等级折扣、会员价优惠 | 折扣本身不返还给消费者 |
| 活动减免 | 满减、秒杀优惠、拼团优惠、活动直减 | 减免金额不返还给消费者 |
| 满赠赠品 | 满减满赠活动赠送的商品或权益 | 整单或活动关联商品全部退款时需退回赠品或记录赠品回收 |
| 预定金权益 | 花钱购买抵扣/折扣名额，例如 10 元买 20 元抵扣权益 | 使用该权益的订单全退时，预定金权益需回退为未核销状态 |

#### 4.4A.2 SKU 与数量维度拆分

规则：

1. 退款金额和资产退回必须按订单原始支付快照计算，不允许只按当前商品售价反推。
2. 同一订单多 SKU、多数量时，需要拆到商品、SKU、数量维度。
3. 部分退款或部分退货退款时，只退本次售后涉及的 SKU 和数量对应的现金、卡金、赠送金、积分。
4. 历史已退款金额和已退资产必须从当前剩余可退金额中扣除。
5. 同一 SKU 购买多件时，按单件分摊金额乘以本次退款数量计算。

示例：

```text
SKU A 购买 2 件，现金分摊 ¥60，卡金分摊 ¥20，积分抵扣 1000
本次只退 1 件：
  现金退回 ¥30
  卡金退回 ¥10
  积分退回 500
```

#### 4.4A.3 优惠券退回规则

优惠券按“整张券关联商品是否全部退款成功”判断是否退回。

规则：

1. 如果该优惠券关联的商品全部退款成功，优惠券按优惠券系统规则退回。
2. 如果只退了部分关联商品，优惠券不退回。
3. 如果优惠券已过期、已作废、不可退，按优惠券系统既有规则处理，但必须记录退回状态。
4. C 端需要展示优惠券退回状态，例如 `优惠券暂不退回`、`优惠券已退回`、`优惠券不满足退回条件`。

#### 4.4A.4 会员折扣、活动减免与满赠规则

规则：

1. 会员折扣、会员价优惠、满减金额、活动直减金额属于价格优惠，不作为独立资产返还给消费者。
2. 秒杀价、拼团价等活动优惠同样不返还优惠差额，只按实际支付构成退款。
3. 满减满赠中的“减金额”不返还；但赠品属于可回收权益或商品时，需要按活动规则处理。
4. 如果整单退款，或满赠活动关联商品全部退款，赠品需退回或记录赠品回收状态。
5. 如果只是部分退款，且仍满足赠品保留条件，赠品可不退回；如不再满足赠品条件，需按活动规则要求退回赠品或在退款金额中处理。

#### 4.4A.5 预定金权益回退规则

预定金活动本质是消费者提前支付一笔金额，获得后续订单中的抵扣或折扣资格。

示例：

```text
消费者支付 ¥10 购买预定金权益
权益内容：可在指定订单抵扣 ¥20
后续订单使用该权益并抵扣 ¥20
如果后续订单整单退款，则该预定金权益需要回退为未核销状态
```

规则：

1. 使用预定金权益的订单整单退款成功后，需要将关联预定金权益恢复为未核销或可重新使用状态。
2. 预定金权益回退必须关联原预定金订单、使用该权益的订单和本次售后单。
3. 如果只退部分商品，是否回退预定金权益需按活动规则判断；默认只有使用权益的订单整单退款时才回退权益。
4. C 端可展示与消费者相关的退款结果，但不展示内部权益冲正明细。
5. 后台必须记录预定金权益回退状态，避免权益已使用但售后后未恢复。

#### 4.4A.6 水单与资产回退规则

退款不删除原购买水单，而是按业务发生方向新增对应退款水单，保持购买、退款两条业务记录可追溯。

规则：

1. 消费者购买成功时，系统按原有逻辑形成购买销售单或购买水单。
2. 消费者退款成功时，系统新增一条退款单或退款水单，关联原购买销售单、原订单号、本次售后单号和退款单号。
3. 购买水单和退款水单必须同时保留，不允许通过删除或覆盖原购买水单来表达退款。
4. 商品订单退款时，退款水单记录本次商品退款金额、原单运费退款金额、支付方式退回明细和售后单关系。
5. 非商品订单退款时，退款水单除记录退款金额外，还必须记录对应资产回退结果。
6. 套餐/次卡订单退款时，需要回退客户项目/次数资产，并在退款水单中记录回退资产类型、回退数量、回退前余额/次数、回退后余额/次数。
7. 开卡充值订单退款时，需要回退卡金、赠送金、赠品等资产，并在退款水单中记录卡金回退、赠送金回退、赠品回收或回退状态。
8. 预定金订单退款时，需要回退预定金权益状态，并在退款水单中记录预定金退款金额、权益回退状态和关联预定金订单号。
9. 退款水单必须支持后续财务、门店、客户资产流水查询，不要求消费者看到内部水单编号。

水单字段至少包含：

| 字段 | 说明 |
| --- | --- |
| 水单类型 | 购买销售单、退款单 |
| 原购买水单号 | 退款水单关联的原购买销售单 |
| 原订单号 | 原交易订单 |
| 售后单号 | 本次售后单 |
| 退款单号 | 本次退款记录 |
| 订单业务形态 | 商品物流、自提、套餐/次卡、开卡充值、预定金 |
| 资产类型 | 项目/次数、卡金、赠送金、赠品、预定金权益等 |
| 资产回退数量/金额 | 本次退款回退的资产数量或金额 |
| 回退前资产余额 | 退款前客户资产余额、次数或权益状态 |
| 回退后资产余额 | 退款后客户资产余额、次数或权益状态 |
| 水单生成时间 | 购买或退款水单生成时间 |

### 4.5 责任与运费承担原则

必须分开记录：

1. `退货责任方`：本次售后问题的主要责任归属。
2. `寄回邮费责任方`：消费者寄回商品的邮费由谁承担。
3. `补寄责任方`：补寄产生的货值成本归属。
4. `补寄运费承担方`：补寄发货运费由谁承担。

责任方可选：

- 消费者责任
- 商家责任
- 物流责任
- 云仓责任
- 其他责任
- 线下协商

任一人工调责或调运费承担方，必须填写原因并记录调整前后值、调整人、调整时间。

### 5.2 订单详情入口

订单详情页负责入口判断，不承接后台审核信息。

必须展示：

1. 订单状态和消费者可理解说明。
2. 商品信息、数量、实付金额。
3. 商品实付、原订单运费、支付合计、支付方式摘要。
4. 售后期提示。
5. 底部主按钮：`申请退款 / 申请售后 / 售后详情 / 不可申请`。

待发货商品物流订单入口规则：

1. 点击 `申请退款` 后直接进入 `仅退款申请页`，不进入退货退款类型页。
2. 虽然订单尚未发货，但消费者仍需要在 `仅退款申请页` 选择本次要退款的商品、SKU 和数量。
3. `仅退款申请页` 展示当前订单仍可退款的商品、SKU、数量和对应可退金额。
4. 页面默认可选全部剩余可退款商品，消费者可取消部分商品或调整退款数量。
5. 如果消费者选择当前剩余全部商品和数量，系统判定为待发货全额仅退款。
6. 如果消费者只选择部分商品、SKU 或数量，系统判定为待发货部分仅退款。
7. 因为商品尚未发货，所以待发货仅退款不展示退货地址，不要求填写退货物流，不进入仓库验收和补寄链路。

非物流退款型订单入口规则：

1. 适用商品自提订单、套餐/次卡订单、开卡充值订单等不需要物流寄回的订单。
2. C 端点击 `申请退款` 后沿用当前线上普通退款交互，只弹出确认弹窗。
3. 弹窗只需要告知 `退款申请已提交，将等待商家处理退款` 或同等语义。
4. 消费者确认后即提交退款申请，不进入新的退款申请页。
5. 消费者不需要填写退款原因、申请说明、图片凭证，也不需要选择商品或数量。
6. 提交后系统创建售后单，并直接进入商家 B 端售后管理 `待退款`。
7. C 端售后详情展示 `待退款`，说明等待商家处理退款。

### 5.3 售后类型页

发货商品订单已发货后，C 端只展示两类入口：

| 入口 | 说明 | 是否寄回 | 商品选择 |
| --- | --- | --- | --- |
| 退货退款（已收到货） | 消费者已收到货，需要退回商品后退款 | 是 | 必须选择本次退货退款的商品、SKU 和数量 |
| 仅退款（未收货仅退款） | 消费者未收到货，申请不寄回商品直接退款 | 否 | 必须选择本次仅退款的商品、SKU 和数量 |

规则：

1. 两个入口都支持全额和部分。
2. 是否全额或部分不由入口决定，而由消费者在申请页选择的商品、SKU 和数量决定。
3. 选择当前剩余全部可售后商品和数量时，系统判定为全额售后。
4. 只选择部分商品、SKU 或数量时，系统判定为部分售后。
5. 已发货后 C 端申请页默认不退原订单运费，并需要在金额区提示原订单运费默认不可退。
6. 两个入口都需要展示最多可退金额及其支付构成，包括现金、卡金、赠送金、积分、优惠券状态等。
7. 两个入口都需要由消费者填写申请退款金额；系统默认可带出最多可退金额，但提交金额不得超过最多可退金额。
8. C 端不展示 `申请补寄`，补寄只作为商家后台审核后的处理结果。
9. C 端不展示分账、财务、责任调账等后台概念。

## 6. 端到端业务流程

### 6.0 商品物流发货订单完整售后原则

商品物流发货订单本期不再按是否分账限制售后能力。

规则：

1. 无论订单是否参与分账，只要是商品物流发货订单，均可按订单状态和售后期进入标准售后体系。
2. C 端售后入口按 `仅退款` 和 `退货退款` 两类呈现；全额或部分由消费者选择的商品、SKU 和数量自动判定，并支持后台审核后的补寄结果。
3. 待发货商品物流订单支持全额仅退款和部分仅退款，消费者需要选择退款商品和数量，但不走寄回流程。
4. 已发货后的售后申请进入商家 B 端待审核，由客服判断同意、拒绝、调额同意或转补寄。
5. 若是退款退货，消费者必须寄回商品，仓库验收后再进入待退款。
6. 若退回商品完好可二销，仓库生成退货入库记录并回补可售库存；若有破损、污损、缺件、空包，则只记录异常和追溯数据，不做二次入库，不回补可售库存。
7. 待退款节点由财务确认退款。默认不调整退款金额；如因质检结果、协商结果或责任认定需要调整，必须填写调额原因。
8. C 端需展示申请退款金额、最终实际退款金额和调额原因，避免消费者只看到金额变化却不知道原因。

旧开关处理：

1. 原商城设置中用于限制 `非分账订单可退款退货/分账订单不可退款退货` 的开关，本期不再作为商品物流发货订单售后能力的判断条件。
2. 商品物流发货订单无论是否参与分账，均按本章节进入标准售后体系，支持仅退款、退货退款、全额、部分和多次售后。
3. 非物流退款型订单不受该旧开关影响，统一按轻量全额退款链路处理，不进入退货寄回和仓库验收流程。
4. 若系统中仍保留该开关配置项，本期需从售后准入逻辑中移除引用；页面是否隐藏或保留展示需至少标记为旧配置不生效，避免运营误以为它仍会限制本期售后能力。

### 6.1 发货商品订单仅退款流程

```text
C端订单列表/订单详情
  -> 申请退款/申请售后
    -> 待发货直接进入仅退款申请页
      -> 选择退款商品、SKU、数量
      -> 选择全部剩余商品：待发货全额仅退款
      -> 选择部分商品/SKU/数量：待发货部分仅退款
      -> C端填写申请退款金额，系统提示最多可退金额和支付构成
    -> 已发货选择仅退款（未收货仅退款）
      -> 选择退款商品、SKU、数量
      -> 选择全部剩余商品：已发货全额仅退款
      -> 选择部分商品/SKU/数量：已发货部分仅退款
      -> C端填写申请退款金额、原因、说明、凭证
      -> 系统提示最多可退金额、支付构成和原订单运费默认不可退
        -> B端待审核
          -> 商家同意/调额同意
            -> 待退款
              -> 财务操作退款
                -> 退款中
                  -> 退款成功进入已完成 / 失败进入退款失败
          -> 商家拒绝
            -> 已关闭（关闭类型：已拒绝）
          -> 商家转补寄
            -> 待补寄出库
```

关键规则：

1. 待发货全额仅退款默认可退商品金额和原订单运费。
2. 待发货部分仅退款默认只退本次选择商品、SKU、数量对应的商品实付分摊金额。
3. 待发货部分仅退款后，如果订单仍有商品需要发货，原订单运费默认不退。
4. 待发货场景下，如果本次退款覆盖订单当前剩余全部可售后商品和数量，则按全额退款处理，可退原订单运费。
5. 待发货全额仅退款和部分仅退款统一进入商家售后管理 `待审核`，或按系统参数走自动/人工审核。
6. 审核同意后进入 `待退款`，由财务确认退款。
7. 该链路不进入 `待寄回 / 待收货 / 仓库验收 / 补寄`。
8. 已发货仅退款支持全额和部分，均需要选择商品、SKU 和数量。
9. 已发货仅退款默认不退原订单运费，且必须人工审核。
10. 仅退款申请页必须由消费者填写申请退款金额，并展示最多可退金额及支付构成；已发货仅退款还必须提示原订单运费默认不可退。
11. 审核或财务退款时可调整审批退款金额，但不得超过系统最大可退金额；如有调整，C 端展示申请退款金额、实际退款金额和调额原因。

### 6.2 发货商品订单退货退款流程

```text
C端选择退货退款（已收到货）
  -> 选择退货退款商品、SKU、数量
  -> 选择全部剩余商品：全额退货退款
  -> 选择部分商品/SKU/数量：部分退货退款
  -> C端填写申请退款金额、原因、说明、凭证
  -> 系统提示最多可退金额、支付构成和原订单运费默认不可退
    -> B端待审核
      -> 商家同意
        -> 待寄回
          -> C端填写退货物流
            -> 待收货
              -> 商家/仓库签收质检
                -> 全部完好可二销：确认退货入库
                  -> 待退款
                    -> 操作退款
                      -> 退款中
                        -> 已完成 / 退款失败
                -> 部分破损/污损/缺件/空包：完好数量退货入库，异常数量只记录不入库
                  -> 待退款
                    -> 按质检结果确认最终退款金额
      -> 商家拒绝
        -> 已关闭（关闭类型：已拒绝）
      -> 商家转补寄
        -> 补寄链路
```

关键规则：

1. 审核通过后，C 端售后详情必须展示退货地址、寄回截止时间、寄回提示。
2. 退货寄回邮费责任由 B 端审核阶段判断，C 端申请页不直接让消费者判定。
3. 签收质检后才能进入待退款，不允许在买家寄回时提前回补库存。
4. 仓库签收时必须判断退回货品是否可二次销售，并形成对应库存处理记录。
5. 可二次销售的完好商品生成 `退货入库` 记录并回补可售库存；不可二次销售或缺失的数量按破损、污损、缺件、空包等结果记录异常，不做二次入库。
6. 质检结果会影响待退款节点的最终审批退款金额，若扣减金额，必须填写扣减原因并回显给 C 端。
7. 已发货退货退款申请页默认不退原订单运费，并需要展示原订单运费默认不可退提示。
8. 退货退款申请页必须由消费者填写申请退款金额，并展示最多可退金额及支付构成，避免误认为退款全部退回微信/现金。

### 6.2A 仓库验收、退货入库与异常记录流程

```text
买家填写退货物流
  -> 商品到达仓库/门店
    -> 仓库签收
      -> 系统展示退回商品验收列表
        -> 仓储人员按商品/SKU/数量逐项确认
          -> 全部完好、附件齐全、可二次销售
            -> 确认退货入库
              -> 入库类型：退货入库
              -> 回补可售库存
              -> 售后进入待退款
          -> 部分破损/缺失/严重污染/影响二次销售
            -> 完好可二销数量确认退货入库
              -> 入库类型：退货入库
              -> 回补可售库存
            -> 异常数量记录异常类型、说明和图片凭证
              -> 不生成退货入库
              -> 不回补可售库存
              -> 售后进入待退款或人工复核
          -> 全部破损/缺失/严重污染/影响二次销售
            -> 记录异常类型、说明和图片凭证
            -> 不生成退货入库
            -> 不回补可售库存
            -> 售后进入待退款或人工复核
```

仓库处理规则：

1. 待收货节点必须支持 `签收质检` 操作。
2. 签收后系统必须展示 `退回商品验收列表`，列表按本次售后单商品、SKU、申请退货数量展示，供仓储人员逐项确认。
3. 验收结果至少包含：全部完好、部分破损、部分缺失、严重污染、影响二次销售、空包、错件、其他异常。
4. 全部完好且可二次销售时，系统自动生成退货入库记录，入库类型为 `退货入库`，并回补可售库存。
5. 部分异常时，只对完好且可二次销售的数量生成 `退货入库`；破损、缺失、严重污染、影响二次销售的数量只生成异常记录，不生成退货入库，不回补可售库存。
6. 全部异常时，不生成退货入库，不回补可售库存，只保留异常记录和追溯数据。
7. 无法二次销售的残次品本期不进入二次入库，不生成残次库存，不生成破损出库单。
8. 质检记录必须关联原订单号、原订单商品行、售后单号、售后商品行、商品、SKU、申请退货数量、质检人、质检时间。
9. 质检结果一旦影响最终退款金额，B 端必须记录调额原因，C 端展示最终审批金额和调额说明。

仓库字段：

| 字段 | 说明 |
| --- | --- |
| 原订单号 | 本次退货商品对应的原交易订单 |
| 原订单商品行 | 原订单中对应商品行，用于追溯 SKU、数量和金额 |
| 售后单号 | 本次退货退款对应售后单 |
| 售后商品行 | 售后单中对应商品行 |
| 商品/SKU | 本次验收商品和规格 |
| 申请退货数量 | C 端申请并经 B 端同意退回的数量 |
| 本次签收数量 | 仓库实际收到的数量 |
| 完好可入库数量 | 可二次销售并可回补库存的数量 |
| 破损数量 | 破损且不可二次销售的数量 |
| 缺失数量 | 应退未退、包裹少件或空包对应数量 |
| 污染/影响二销数量 | 严重污染或其他影响二次销售的数量 |
| 不入库数量 | 破损、缺失、污染、影响二销等不生成退货入库的数量 |
| 异常类型 | 破损、缺失、严重污染、影响二次销售、空包、错件、其他 |
| 异常说明 | 仓储人员填写的异常描述 |
| 图片凭证 | 退回商品照片、破损照片、包裹照片 |
| 验收人 | 仓库或门店处理人 |
| 验收时间 | 仓库确认验收的时间 |
| 入库类型 | 完好可入库数量生成 `退货入库` |
| 处理结果 | 退货入库成功、部分退货入库、只记录异常不入库 |

数量校验：

1. `完好可入库数量 + 破损数量 + 缺失数量 + 污染/影响二销数量` 不得超过本次申请退货数量。
2. `完好可入库数量` 不得超过本次签收数量。
3. `缺失数量` 可由申请退货数量减本次签收数量计算，也可由仓储人员在少件/空包场景录入后系统校验。
4. 每条退货入库记录和每条异常记录都必须能追溯到原订单号、原订单商品行、售后单号和售后商品行。

### 6.3 补寄结果流程

```text
C端提交仅退款或退货退款
  -> B端待审核
    -> 商家判断少件/漏发/错发等适合补寄
      -> 转补寄处理
        -> 待补寄出库
          -> 商家创建补寄出库申请/补寄需求单
            -> 推送云仓
              -> 云仓待发货
                -> 云仓发货并回传快递公司、物流单号、发货时间
                  -> 待买家收货
                    -> C端查看补寄物流/确认收货
                      -> 已完成
```

关键规则：

1. 补寄不是 C 端主动入口，是后台审核后的处理结果。
2. `转补寄处理` 阶段，客服只记录补寄原因、补寄责任方、补寄商品、数量和补寄说明，不填写快递公司和物流单号。
3. `待补寄出库` 阶段，商家创建的是补寄出库申请或补寄需求单，只选择出库仓/云仓、补寄商品、数量、补寄运费承担方和给云仓说明。
4. 补寄出库申请创建后推送到云仓；云仓未发货前，C 端和 B 端只展示补寄商品、补寄原因、出库申请状态，不展示补寄物流公司和物流单号。
5. 云仓实际发货后，系统接收云仓回传的快递公司、物流单号和发货时间，售后主状态进入 `待买家收货`。
6. 只有云仓回传补寄物流后，C 端和 B 端才展示补寄物流信息，并提供 `查看补寄物流` 或物流轨迹入口。
7. 补寄通常不退款，但必须形成补寄成本、补寄出库申请、云仓发货结果和出库台账。

### 6.4 非物流退款型订单流程

适用：商品自提订单、套餐/次卡订单、开卡充值订单、预定金订单。

```text
C端点击申请退款
  -> 系统校验订单/资产状态
    -> 校验通过
      -> 弹出普通退款确认弹窗
        -> 消费者确认提交
          -> 系统默认申请理由：不想要了
          -> 创建售后单
          -> B端售后管理待退款
            -> 财务确认退款金额
              -> 默认按订单可退金额全额退款
                -> 退款中
                  -> 退款成功
                    -> 生成退款水单
                    -> 非商品订单回退对应资产
                    -> 已完成
                  -> 退款失败
                    -> 财务充值或排除账户异常后重新退款
    -> 校验不通过
      -> C端不展示入口或展示不可退款原因
```

边界：

1. 非物流退款型订单不进入待寄回、待收货、补寄链路。
2. 非物流退款型订单只支持整单全额退款，不支持部分退款、退款退货和补寄。
3. 商品自提订单未自提前可申请退款；已自提或已确认收货后不允许退款。
4. 套餐/次卡订单需判断资产是否已使用；未使用且在可退款期内可退款，已核销或已使用不允许退款。
5. 开卡充值订单需判断卡金、赠送金、赠品等资产是否满足回退条件；资产已使用或不足回退时不允许退款。
6. 预定金订单需判断预定金权益是否已使用；权益未使用且在可退款期内可退款，权益已使用或不满足退款条件时不允许退款。
7. C 端只弹普通退款确认弹窗，不要求填写退款原因、说明或凭证，保持与当前线上交互一致。
8. C 端确认提交后统一进入商家售后管理待退款，不进入待审核。
9. 非物流退款型订单不提供商家审批同意/拒绝节点。
10. 非物流退款型订单创建售后单时，系统默认申请理由为 `不想要了`，C 端不展示原因选择。
11. 本期非物流退款型订单只要命中可退款条件，均允许退款，原则上不需要业务调额。
12. 财务默认按订单可退金额全额退款；财务系统可保留调额能力，但不作为该类订单的主流程。如发生特殊调额，必须填写调额原因并留痕。
13. 若财务无法退款，通常原因是商家账户余额不足或可用退款资金不足；补足资金后继续发起退款，状态停留在 `待退款` 或 `退款失败`，不应改成 `已关闭`，也不应记录关闭类型 `已拒绝`。
14. C 端结果页展示申请退款金额、实际退款金额、退款方式明细；如发生特殊调额，再展示调额原因。
15. 退款成功后必须新增退款水单，并关联原购买水单、原订单、售后单和退款单。
16. 套餐/次卡、开卡充值、预定金等非商品订单退款成功后，必须同步回退对应资产，并将资产回退结果写入退款水单。

### 6.5 预定金订单流程

```text
C端订单详情
  -> 系统判断预定金权益是否已使用
    -> 权益未使用且在可退款期内：展示申请退款
      -> 点击申请退款
        -> 弹出普通退款确认弹窗
          -> 消费者确认提交
            -> 创建售后单
            -> B端售后管理待退款
              -> 财务确认退款金额
                -> 退款中
                  -> 已完成
    -> 权益已使用或不满足退款条件：不展示入口或展示不可退款原因
```

说明：预定金订单虽然有独立权益校验，但本期退款操作流与非物流退款型订单一致，不填写退款原因、申请说明和图片凭证，系统默认申请理由为 `不想要了`，不进入待审核、待寄回、待收货、仓库验收和补寄链路。

## 7. B 端处理规则

### 7.1 售后列表

列表需支持：

1. 状态 Tab：全部、待审核、待寄回、待收货、待退款、退款中、退款失败、待补寄出库、待买家收货、已完成、已关闭。
   已关闭列表可继续按关闭类型筛选：已拒绝、已撤销、超时关闭、人工关闭。
2. 筛选：申请时间、售后状态、售后类型、订单业务形态、订单来源/营销类型、交易门店、分账状态、责任方、关键词。
3. 列表固定展示交易门店。
4. 总部账号可筛全部门店；分店账号默认锁定当前门店。
5. 订单类型展示门店可识别来源：微商城、拼团、秒杀、积分订单、开充值订单，并补充处理形态：发货订单、自提订单、直接到资产、储值卡直接到资产。
6. 列表必须支持按处理阶段快速识别待处理单据，不再只作为订单摘要页。
7. 列表需固定展示或可筛选责任相关字段：退货责任方、寄回邮费责任方、补寄责任方、补寄运费承担方。
8. 列表需支持识别退款失败；分账回退失败本期只入库记录，不在售后列表增加待追回处理入口。

### 7.2 详情页模块

详情页不是只展示当前状态，而是按已发生处理节点累积展示处理流程结果卡片。每完成一个动作，就在详情页沉淀一张对应结果卡；后续状态页必须能看到前序动作结果。尚未发生的后续节点不展示对应内容卡片，不展示空状态占位，也不预先展示该节点字段。

详情页按状态组合展示以下模块：

1. 顶部状态提示区。
2. 顶部基础摘要区：左侧用户信息，右侧退款项目明细。
3. 售后申请基础信息卡。
4. 处理流程结果卡片区。
5. 审批信息卡。
6. 买家退货物流卡。
7. 质检判断说明卡。
8. 退货入库/异常记录处理卡。
9. 补寄处理结果卡。
10. 补寄物流结果卡。
11. 系统退款清分账单。
12. 分账回退信息区。
13. 操作日志。
14. 中央弹窗交互区。

处理流程结果卡片要求：

1. 结果卡片按时间顺序展示已发生节点，最新处理结果可置顶或高亮，但不得隐藏前序结果。
2. 每张结果卡至少包含处理结论、处理时间、处理人/来源、关键字段和 C 端可见文案。
3. B 端结果卡可额外展示内部字段，例如责任方、运费承担方、审批金额、调额原因、质检数量、异常类型、分账状态、云仓出库申请号、云仓发货回传时间等。
4. C 端结果卡只展示消费者能理解的信息，不展示分账、云仓内部字段、合伙人佣金、钱包扣减、内部责任归因等后台概念。
5. 结果卡片类型至少包括：申请结果、审核结果、退货寄回结果、仓库质检结果、退货入库/异常结果、退款结果、退款失败结果、补寄处理结果、补寄物流结果、关闭结果。
6. 凡结果卡片包含物流信息，必须提供可点击进入物流详情页或物流轨迹页的入口，包括原订单物流、买家退货物流和补寄物流。
7. 结果卡片只在对应动作完成后出现。例如未审核时不展示审核结果卡，买家未填写退货物流时不展示买家退货物流卡，仓库未质检时不展示质检和入库异常卡，财务未发起退款时不展示退款结果卡，云仓未发货时不展示补寄物流卡。
8. `待审核` 状态通常只展示申请结果卡；`待寄回` 在申请结果卡基础上增加审核结果卡；`待收货` 增加买家退货物流结果卡；`待退款` 增加质检/入库结果卡；`退款中` 增加退款发起结果卡；`退款失败` 增加退款失败结果卡；`待补寄出库` 增加补寄处理结果卡；`待买家收货` 增加补寄物流结果卡；`已完成` 展示完整已发生链路；`已关闭` 展示关闭前已发生链路和关闭结果卡。
9. 补寄结果卡片保持精简，只保留 `补寄处理结果` 和 `补寄物流结果` 两类。商家转补寄、创建补寄出库申请、推送云仓等信息合并展示在 `补寄处理结果` 中，通过出库状态表达进度；云仓发货、物流单号、签收信息合并展示在 `补寄物流结果` 中。
10. 仓库质检结果卡必须展示商品级验收明细列表，不得只展示汇总数量。每个商品行至少包含商品图片/名称/SKU、应退数量、签收数量、完好数量、异常数量、异常类型。
11. 退货入库/异常结果卡必须展示商品级入库与异常明细列表，不得只展示汇总数量。每个商品行至少包含商品图片/名称/SKU、完好入库数量、异常不入库数量、处理结果和关联入库/异常记录。

关键结果卡片内容建议：

| 结果卡片 | 核心内容 |
| --- | --- |
| 仓库质检结果 | 处理结论、质检时间、质检人、退回商品验收明细列表、异常说明、图片凭证 |
| 退货入库/异常结果 | 处理结论、处理时间、处理人、入库与异常明细列表、入库仓库、关联入库单号、异常记录单号 |
| 补寄处理结果 | 处理结论、处理时间、处理人、补寄原因、补寄商品列表、补寄说明、出库状态 |
| 补寄物流结果 | 处理结论、发货时间、补寄商品列表、快递公司、快递单号、查看补寄物流入口；若已签收，再追加签收时间和完成方式 |

顶部基础摘要区要求：

1. B 端售后详情页固定展示顶部基础摘要区，位于状态进度和处理结果卡片之前。
2. 左侧为用户信息，至少展示用户昵称、门店、手机号、申请退款金额、原订单号。
3. 右侧为 `退款项目明细`，按行展示本次售后涉及的退款项目。
4. 商品退款按商品行展示，包含商品图片、商品名称/SKU、退款数量和退款金额。
5. 如本次审批或退款包含原单运费退款，必须在 `退款项目明细` 中与商品同列单独展示一行，例如 `原单运费退款  *1  ￥10.00`。
6. 如本次包含寄回邮费补贴/赔付，也可在 `退款项目明细` 中单独展示一行，例如 `寄回邮费补贴  *1  ￥5.00`，但字段口径仍需与商品退款、原单运费退款分开记录。

### 7.4 中央弹窗

所有处理动作统一通过中央弹窗承接，不使用底部抽屉。

弹窗类型：

1. 同意申请：填写审批退款金额、原单运费退款金额、责任方、寄回邮费责任方、寄回邮费补贴、买家留言。
2. 拒绝申请：填写拒绝原因、拒绝说明。

5. 操作退款：确认退款金额、退款明细、退款说明。

本期商家后台售后管理模块需要从“可处理退款”升级为“标准售后工作台”。

标准化要求：

1. 状态流完整覆盖待审核、待寄回、待收货、待退款、退款中、退款失败、待补寄出库、待买家收货、已完成、已关闭；已关闭下记录关闭类型：已拒绝、已撤销、超时关闭、人工关闭。
2. 每个状态都有明确页面重点、可执行操作、不可执行操作和流转结果。
3. 所有业务动作统一通过中央弹窗处理，不在列表页直接完成复杂业务。
4. 审批阶段可记录退货责任方、寄回邮费责任方、审批退款金额、原单运费退款金额、寄回邮费补贴。
5. 补寄阶段可记录补寄责任方、补寄商品、补寄运费承担方、补寄出库申请、给云仓说明和云仓发货回传结果。
6. 质检阶段可记录退货入库、破损、缺失、严重污染、影响二次销售、空包、错件等仓库处理结果。
7. 退款阶段可查看申请金额、审批金额、实际退款金额、资产退回明细、分账回退状态。
8. 所有调额、调责、调运费承担方、拒绝、关闭、退款失败重试都必须进入操作日志。

## 9. 财务、分账与报表规则

### 9.1 分账原则

1. 分账不限制售后入口。
2. 消费者退款优先于分账回退：财务先从商家侧资金账户或可用退款资金中退款给消费者，退款成功后再触发原订单分账回退/冲正。
3. 售后主状态与分账回退状态分开维护。
4. 分账回退成功和失败都必须形成明细记录，不允许只记录汇总结果。
5. 回退失败必须沉淀为数据库明细记录；本期不做前台报表、待追回列表、处理人分配或追回操作功能。
6. 合伙人佣金属于特殊回退对象：能从佣金账户正常回退时按正常回退记录；不能正常回退时，允许直接扣减合伙人钱包余额，钱包余额可扣为负数，并形成负余额扣减记录。
7. 普通分账方回退失败不影响 C 端退款结果和售后完成状态，但必须生成回退失败明细，用于后续报表统计和追回处理。

### 9.2 分账子状态

- 未触发分账
- 待分账冲正
- 分账冲正中
- 分账冲正成功
- 分账冲正失败
- 已提现不可回退
- 回退处理完成
- 合伙人钱包负数扣减

### 9.2A 分账回退记录规则

已分账订单发生退款时，系统必须在消费者退款成功后生成分账回退记录。无论回退成功还是失败，都要保留明细。

处理顺序：

```text
锁定最终审批退款金额
  -> 财务从商家侧资金账户或可用退款资金退款给消费者
    -> 退款成功：售后主状态可进入已完成
      -> 读取原订单分账明细
        -> 计算各分账方应回退金额
          -> 发起分账回退/冲正
            -> 成功：记录回退成功明细，回退金额归还商家或完成冲正
            -> 普通分账方失败：记录回退失败明细
              -> 本期到数据库记录为止，不做追回处理功能
            -> 合伙人佣金无法正常回退
              -> 扣减合伙人钱包余额，可扣为负数
              -> 记录合伙人佣金负余额扣减明细
    -> 退款失败：售后主状态进入退款失败，财务充值或排除账户异常后重新退款
```

规则：

1. C 端只感知消费者退款是否成功，不展示分账回退过程。
2. 消费者退款成功后，售后主状态可进入 `已完成`；后续普通分账方回退失败不反向影响 C 端售后完成结果。
3. 分账回退成功时，记录回退成功明细，并将对应金额回退给商家或完成原分账冲正。
4. 分账回退失败时，记录失败金额、失败原因、分账方、原订单、售后单、退款单、失败时间和后续处理标识。
5. 本期分账回退失败只做数据库记录，不做前台待追回列表、失败报表页面或追回处理动作；后续报表和追回能力基于本期明细扩展。

回退成功记录至少包含：

| 字段 | 说明 |
| --- | --- |
| 原订单号 | 发生售后的订单 |
| 售后单号 | 本次售后单 |
| 退款单号 | 对应退款记录 |
| 分账方 | 需要回退金额的主体 |
| 分账角色 | 商家、门店、总部、代运营方、销售分成方等 |
| 原分账金额 | 该分账方原获得金额 |
| 本次应回退金额 | 本次退款应回退金额 |
| 实际回退金额 | 实际扣回成功金额 |
| 回退结果 | 成功 |
| 回退时间 | 成功时间 |
| 回退流水号 | 支付或分账系统返回流水 |

### 9.2B 合伙人佣金回退规则

合伙人佣金回退独立于普通分账方回退失败边界处理。

处理规则：

1. 订单退款后，如原订单存在合伙人佣金，系统需按本次退款金额和佣金规则计算本次应回退佣金。
2. 合伙人佣金可正常回退时，记录佣金回退成功明细。
3. 合伙人佣金无法正常回退时，不按普通分账失败“仅记录待后续追回”处理，而是直接扣减合伙人钱包余额。
4. 合伙人钱包余额不足时，允许扣成负数，形成合伙人钱包负余额。
5. 合伙人钱包负数扣减必须记录原订单号、售后单号、退款单号、合伙人、原佣金金额、本次应回退佣金、实际扣减金额、扣减前余额、扣减后余额、扣减时间和扣减原因。
6. 合伙人钱包负数扣减为后台财务处理逻辑，C 端不展示，售后主状态不因该扣减结果对消费者暴露。

合伙人佣金记录至少包含：

| 字段 | 说明 |
| --- | --- |
| 原订单号 | 产生合伙人佣金的原订单 |
| 售后单号 | 本次退款对应售后单 |
| 退款单号 | 本次退款记录 |
| 合伙人 ID/名称 | 被扣减佣金的钱包主体 |
| 原佣金金额 | 原订单产生的合伙人佣金 |
| 本次应回退佣金 | 本次退款需回退的佣金金额 |
| 佣金回退结果 | 正常回退、钱包负数扣减 |
| 实际扣减金额 | 本次从合伙人钱包扣减的金额 |
| 扣减前钱包余额 | 扣减前合伙人钱包余额 |
| 扣减后钱包余额 | 扣减后合伙人钱包余额，可为负数 |
| 扣减时间 | 扣减成功时间 |
| 扣减原因 | 售后退款、佣金无法正常回退等 |

### 9.2C 分账回退失败本期边界

回退失败本期只做数据库记录，不做产品功能呈现。

本期不做：

1. 不做待追回列表。
2. 不做分账回退失败报表页面。
3. 不做失败记录导出入口。
4. 不做处理人分配。
5. 不做自动重试、线下扣减、人工收款、坏账标记等追回处理功能。

本期只做：

1. 在消费者退款成功后触发分账回退，并生成分账回退明细。
2. 回退成功记录成功金额、成功时间和回退流水。
3. 回退失败记录失败金额、失败原因、失败时间和分账方信息。
4. 对已提现到银行卡、余额不足、账户冻结、接口失败等失败原因形成枚举或原始返回码记录。
5. 预留后续报表和追回处理所需字段，但本期不在页面呈现。

要求：

1. 回退失败不应静默失败。
2. 回退失败不应覆盖售后主状态；售后是否完成与分账回退记录分开维护。
3. 消费者退款成功后，如普通分账方回退失败，必须记录垫付主体、垫付金额、回退失败金额。
4. 后续如要做报表或追回处理，应基于本期沉淀的数据库明细扩展，不反推历史订单。
5. 合伙人佣金无法正常回退时按 `9.2B 合伙人佣金回退规则` 执行，不进入“仅记录不处理”的普通分账失败边界。

## 10. 权限与风控

### 10.1 权限

| 角色 | 权限 |
| --- | --- |
| 商家客服 | 审核、备注、查看详情、按配置调额/调责 |
| 仓库 | 确认收货、录入质检结果 |
| 财务 | 执行退款、处理分账冲正异常 |
| 管理员 | 关闭售后、修改责任判定、导出报表 |
| 总部账号 | 查看并筛选全部门店售后 |
| 分店账号 | 默认只查看本店售后 |

### 10.2 风控校验

1. 同一商品可退件数不可超过实付件数减已退件数。
2. 累计退款金额不可超过订单实付金额。
3. 已完成、已关闭的售后单不可重复执行退款。
4. 退款失败必须通过重试退款或人工关闭收口。
5. 分账拆分明细与审批退款金额不一致时，不允许执行退款。
6. 调额、调责、调运费承担方未填写原因时，不允许提交。
7. 未命中配置的商品，不展示人工调额、调责入口。
8. 质检结果未录入时，退货退款单不允许直接进入退款完成。
9. 完好可入库数量、破损数量、缺失数量、污染/影响二销数量之和不得超过本次申请退货数量。
10. 分账回退失败记录未生成时，不允许将分账处理记录标记为已记录。

## 13. 执行结论

商家自营订单售后本期按以下口径执行：

1. 按商家自营订单独立形成端到端 PRD，不再要求读者同时翻 C 端文档和 B 端文档。
2. 发货商品订单是主链路，支持仅退款、退款退货、补寄。
3. 自提、套餐/次卡、储值卡、预定金按退款型订单处理；预定金 C 端开放退款入口，但不进入退货寄回和补寄链路。
4. C 端复用现有真实用户路径，只补齐金额、状态、退款方式、补寄结果展示。
5. B 端售后管理承接完整审核、质检、退款、补寄、分账、日志和报表字段。
6. 商品退款金额、原单运费退款金额、寄回邮费补贴/赔付金额必须分开。
7. 退货责任方、寄回邮费责任方、补寄责任方、补寄运费承担方必须分开。
8. 补寄不作为 C 端主动申请入口，只作为后台审核后的处理结果。
9. 退款退货回仓后必须由仓库判断是否可二次销售；完好可二销数量生成 `退货入库` 并回补可售库存，不可二销或缺失数量只记录异常和追溯数据，不做二次入库。
10. 分账回退成功和失败都必须形成数据库记录；回退失败本期只做数据沉淀，不做报表页面、待追回列表或追回处理功能。
11. 合伙人佣金回退按特殊规则处理：可正常回退则回退，无法正常回退时直接扣减合伙人钱包，钱包余额允许扣为负数，并形成扣减明细。
12. 购买与退款都必须形成独立水单：购买保留购买销售单，退款新增退款单；非商品订单退款还必须同步回退对应资产并记录资产回退水单。
13. 商家后台售后管理模块本期需要按标准工作台升级，补齐状态流、详情模块、中央弹窗、责任归属、运费责任、调额、补寄、质检、库存处理和日志。
14. 原订单保留履约主状态，并通过售后标识和商品行级售后状态表达售后中、部分退款、退货退款中、补寄中等逆向进度。
15. 非物流退款型订单 C 端不展示理由选择，创建售后单时默认申请理由为 `不想要了`。
16. 财务先从商家侧资金账户或可用退款资金退款给消费者，消费者退款成功后再触发分账回退；普通分账方回退失败只做记录，供后续报表和追回处理使用。
