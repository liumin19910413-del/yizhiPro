## 1. 项目上下文包

### 1.1 业务背景

当前系统在「营销中心 > 营销应用」下已有支付方案类入口，例如「支付宝人资打款」。系统原先已接入过微信侧的小额打款能力，但由于平台原能力即将下架/下线，现有微信零钱打款链路需要调整为微信支付商家转账的新能力。

因此，本需求不是单纯新增一个可选支付方案，而是一次必要的能力迁移：将原有业务中依赖微信零钱打款的奖励、佣金等场景，迁移到「微信小额打款解决方案」，确保旧能力下架后业务仍可继续发放小额款项。

第一期重点不是直接完成真实支付联调，而是完成可演示、可评审的原型 demo，用于验证：

1. 商家如何开通微信小额打款能力。
2. 总部与分店如何切换视角。
3. 总部统一配置后如何通过「适用门店」分发给分店使用。
4. 分店未被总部配置覆盖时，可自行配置自己的商户号和转账场景。
5. 授权用户、生成授权码和发起打款如何集中在「授权列表」中完成。
6. 总部如何在「数据总览」查看每个店每月打款金额、出款账户拆分和线下应收。
7. 分店如何在「数据总览」通过「打款统计 / 打款明细」切换查看本店数据。
8. 原微信零钱打款链路如何平滑迁移到微信商家转账新能力。

当前已明确的产品边界：

1. 底层能力使用微信支付商家转账的「用户授权免确认模式」。
2. 第一期打款配置只支持「一个商户号 + 一个转账场景」。
3. 安全接口 IP 由系统提供，商家复制后到微信支付商户平台配置。
4. 微信运营账户余额暂不支持系统读取。
5. 总部可统一出资并通过「适用门店」下发给分店，门店打款金额进入总部对门店的线下应收统计。
6. 分店也可能先独立配置自己的微信商户号和商家转账能力；若后续被总部纳入适用门店，总部配置会覆盖分店原配置，分店只能查看不可编辑。
7. 本方案启用后，原依赖旧微信打款能力的微信零钱打款链路不再作为商户侧可选择通道。

不做的影响：

1. 平台原打款能力下架后，原依赖微信零钱打款的业务可能无法继续自动发放。
2. 商家线下奖励、佣金、补贴打款将退回人工处理，缺少统一留痕和报表。
3. 连锁总部无法看到各门店小额打款金额，也无法形成线下收款依据。
4. 门店缺少统一授权用户列表，后续打款容易重复沟通和手工处理。

### 1.2 用户角色

| 角色 | 目标 | 典型操作 | 关注点 |
| --- | --- | --- | --- |
| 连锁总部管理员 | 统一开通、配置小额打款能力，并设置适用门店，查看门店打款统计 | 配置微信参数、选择适用门店、查看总览、导出报表 | 资金安全、门店应收、配置正确、数据可追溯 |
| 适用总部配置的分店管理员 | 使用总部下发能力给本店用户打款 | 查看本店数据、生成授权码、发起打款、查看明细 | 操作简单、只能看本店数据、无法修改总部下发配置 |
| 未适用总部配置的分店管理员 | 自己配置微信商户号并独立出资打款 | 填写打款配置、查看开通说明文档、管理授权用户、发起打款 | 配置指引清晰、能力可独立启用；若后续总部覆盖需明确提示 |
| 财务人员 | 查看打款和门店应收数据 | 查询明细、导出统计、处理失败单 | 金额口径、成功/失败状态、线下收款依据 |
| 收款用户 | 授权后接收奖励、佣金、补贴 | 扫码授权、接收微信零钱 | 授权可信、到账清晰 |
| 系统 | 管理配置、授权、打款记录和报表 | 生成二维码、记录状态、汇总报表 | 状态准确、权限隔离、异常可恢复 |

### 1.3 核心流程

入口：

```text
营销中心
  -> 营销应用
  -> 支付方案
  -> 微信小额打款解决方案
```

入口位置：

```text
支付方案
  1. 微信小额打款解决方案
  2. 支付宝人资打款
  3. 经销商出货分账解决方案
```

主路径：

```text
进入微信小额打款解决方案
  -> 选择当前视角：总部 / 分店A-适用总部配置 / 分店B-独立配置
  -> 判断当前组织是否已启用能力
      -> 未启用：只展示打款配置菜单，进入单页系统配置
      -> 已启用：进入数据总览
  -> 数据总览：查看金额概览，并在打款统计/打款明细之间切换
  -> 授权列表：查看/生成授权码、对已授权用户发起打款
  -> 打款配置：维护微信配置、适用门店
```

关键节点：

1. 总部和未适用总部配置的分店都需要完成打款配置后才能看到数据总览和授权列表。
2. 被总部配置纳入适用门店的分店只查看下发配置，不编辑微信密钥。
3. 若分店先独立配置，后续总部将其加入适用门店，总部配置覆盖分店原配置。
3. 生成授权码时生成二维码，用户扫码完成免确认授权。
4. 只有已授权用户可直接打款。
5. 总部数据总览展示各门店每月打款金额、出款账户拆分和总部应收统计。
6. 分店数据总览只展示本店统计和本店打款明细。

结束状态：

```text
能力未开通 / 待配置 / 待验证 / 待启用 / 已启用 / 运维停用
授权待确认 / 已授权 / 已关闭
打款待处理 / 打款中 / 打款成功 / 打款失败 / 已撤销
```

### 1.4 业务术语

| 术语 | 定义 | 备注 |
| --- | --- | --- |
| 微信小额打款解决方案 | 基于微信支付商家转账能力，面向奖励、佣金、补贴等线下小额打款场景的系统功能 | 本 PRD 命名 |
| 商家转账 | 微信支付向微信用户零钱转账的产品能力 | 本期使用普通商户模式、用户授权免确认模式 |
| 用户授权免确认模式 | 用户首次授权后，后续商户可向授权对应用户发起转账，无需用户逐笔确认 | 适合长期、多次、审核后自动打款 |
| 转账场景 | 微信支付商家转账产品中的官方场景，如现金营销、佣金报酬、企业赔付 | 本期一个商户号只配置一个场景 |
| 业务场景 | 系统内业务发款场景，如抽奖奖励、分销合伙人提现、邮费补贴 | 需映射到已开通的微信转账场景 |
| 来源工具 | 触发打款的具体业务工具或模块，如幸运转盘、零元抽奖、刮刮乐、分销合伙人提现 | 用于打款明细筛选和报表统计 |
| 出款账户 | 实际发起微信商家转账的商户号账户，可能是总部账户，也可能是门店自有账户 | 用于打款明细和门店统计拆分 |
| 接口安全 IP | 微信商户平台中配置的 API 请求来源 IP | 系统提供 IP，商家复制到微信后台配置 |
| 总部统一出资 | 总部商户号统一出款，门店作为成本归属方 | 数据总览中的门店应收统计用于线下收款 |
| 适用门店 | 总部配置中通过门店选择器选中的门店范围 | 被纳入后分店使用总部配置 |
| 继承总部配置 | 分店被纳入总部配置的适用门店后，使用总部下发的微信商户号和场景 | 分店不可编辑密钥；会覆盖分店原配置 |
| 独立配置 | 分店自己配置微信商户号和商家转账能力 | 分店自己出资 |

### 1.5 系统边界

本需求包含：

1. 营销应用入口新增「微信小额打款解决方案」。
2. 原型 demo 顶部支持总部/分店视角切换。
3. 打款配置单页表单、外部操作指引入口和字段获取说明。
4. 微信配置表单和字段获取说明。
5. 接口安全 IP 配置放入外部操作指引，不在系统表单中展示。
6. 一个商户号一个转账场景的配置限制。
7. 总部通过「适用门店」让分店使用总部配置。
8. 总部配置适用门店；适用门店继承总部配置，未适用门店可独立配置。
9. 三个功能页面：数据总览、授权列表、打款配置。
10. 数据总览中的门店维度打款统计、出款账户拆分和打款明细。
11. 授权列表中的授权用户列表、生成授权码和发起打款。

本需求不包含：

1. 真实微信支付接口联调。
2. 真实 API 证书解析、签名、验签、回调解密。
3. 自动读取微信运营账户余额。
4. 一个商户号配置多个微信转账场景。
5. 多商户号路由和跨商户号资金调度。
6. 真正发起资金出款。
7. 财务系统自动收款或扣款。

依赖系统 / 页面 / 接口：

| 依赖对象 | 用途 | 备注 |
| --- | --- | --- |
| 营销中心应用入口页 | 新增入口卡片 | 放在「支付方案」分组，「支付宝人资打款」前 |
| 微信支付商户平台 | 商户号、AppID 绑定、API 密钥、商户证书、商家转账开通、接口安全 IP | 原型只做指引 |
| 组织/门店体系 | 区分总部、分店、数据权限 | 原型用 Mock 数据 |
| 用户/会员体系 | 收款用户基础信息 | 原型用 Mock 数据 |
| 报表导出 | 数据总览中的打款统计和打款明细导出 | 原型可仅展示按钮 |

## 3. 业务结果说明

### 3.1 为什么做

平台原接入的微信打款能力即将下架/下线，现有依赖微信零钱打款的业务需要迁移到微信支付商家转账新能力，避免旧能力不可用后影响抽奖奖励、分销合伙人提现等业务发放。

同时，为连锁商家提供一个可配置、可分发、可统计的微信小额打款解决方案，把奖励、佣金、补贴等小额打款流程沉淀为系统能力，降低门店操作成本，并让总部能统一查看门店打款金额、形成线下收款依据。

### 3.2 希望改变的指标或结果

| 指标 / 结果 | 当前状态 | 目标状态 | 验证方式 |
| --- | --- | --- | --- |
| 旧能力迁移风险 | 原微信打款能力面临下架，存量业务依赖旧链路 | 依赖微信零钱打款的业务迁移到微信商家转账新能力 | 迁移场景走查 |
| 商家开通理解成本 | 需要自行理解微信文档 | 系统提供操作指引和字段级获取说明 | 业务方走查配置路径 |
| 总部门店打款可视化 | 总部无法统一看各门店小额打款 | 总部可看每店每月金额和明细 | 总览和报表验收 |
| 门店操作效率 | 门店人工沟通、人工打款 | 已授权用户可直接发起打款 | 门店路径走查 |
| 授权复用 | 每次打款都需要再次沟通 | 用户授权后可复用授权记录 | 授权列表验收 |
| 财务线下收款依据 | 缺少门店应收明细 | 按门店生成应收统计 | 数据总览验收 |

### 3.3 当前问题

1. 平台原接入的微信打款能力即将下架，存量微信零钱打款业务需要替换到新能力。
2. 微信支付商家转账配置项较多，商家难以知道每项在哪里获取。
3. 安全接口 IP 不是商家填写给系统，而是系统提供给商家去微信后台配置，容易误解。
4. 一个商户号当前实际可用的转账场景受微信商户平台开通结果约束，系统不能无限开放业务场景。
5. 总部统一出资和门店成本归属是两套口径，需要在报表上明确拆开。
6. 分店是否能编辑配置取决于是否被总部纳入「适用门店」；被纳入后配置只读，总部配置覆盖分店原配置。
7. 微信运营账户余额当前不适合设计为系统实时可读数据。

### 3.4 如何验证

| 验证类型 | 判断标准 |
| --- | --- |
| 业务验证 | 用户能从入口进入解决方案，理解配置步骤，切换总部/分店视角 |
| 用户验证 | 总部知道如何设置适用门店，门店知道自己是否被总部配置覆盖 |
| 数据验证 | 总部统计、门店明细、授权列表、打款明细口径清晰 |
| 流程验证 | 未配置到已启用、未授权到已授权、已授权到打款成功/失败路径完整 |

## 4. 需求分类判断

### 4.1 分类结论

- [x] 0-1 / 复杂交互需求
- [x] 有耦合需求
- [ ] 智能体需求
- [x] 合规 / 风险敏感需求

### 4.2 判断依据

1. 0-1 / 复杂交互：新增完整解决方案原型，涉及多角色、多状态、多路径。
2. 有耦合：影响营销应用入口、组织门店、支付配置、授权用户、报表。
3. 非智能体：不涉及 AI 自动决策。
4. 合规 / 风险敏感：涉及微信支付、商户密钥、用户授权、资金出款。
5. 数据 / 分析：涉及总部/门店打款金额、门店应收、导出统计。

## 5. AI-Native PRD 主体

### 5.1 需求目标

用户目标：

1. 商家可按系统引导完成微信小额打款能力开通。
2. 总部可统一配置并下发给分店。
3. 分店未被总部纳入适用门店时可独立配置；被纳入后自动使用总部配置。
4. 总部和分店均可在「授权列表」中管理授权收款用户。
5. 已授权用户可被操作发起打款。

业务目标：

1. 在平台旧打款能力下架前，完成微信零钱打款链路向微信商家转账新能力的迁移方案验证。
2. 支持奖励、佣金、补贴等线下小额打款场景的系统化管理。
3. 总部可统计各门店月度打款金额，用于线下向门店收款。
4. 分店可在授权列表页查看本店授权用户并发起打款，在数据总览查看本店统计和明细。

产品目标：

1. 新增应用入口「微信小额打款解决方案」。
2. 完成总部/分店视角切换的原型 demo。
3. 沿用现有系统后台风格：左侧模块导航、顶部面包屑、浅色卡片、表格筛选、蓝色主按钮。

技术目标：

1. 原型阶段使用 Mock 数据表达完整流程。
2. 页面结构可平滑迁移到真实接口。
3. 敏感配置字段支持脱敏展示和编辑权限区分。

### 5.2 范围边界

| 类型 | 内容 |
| --- | --- |
| 本期做 | 旧微信打款能力迁移到微信商家转账新能力的原型验证；入口卡片；总部/分店视角切换；操作指引；微信配置；一个商户号一个转账场景；适用门店；数据总览；授权列表；生成授权码；模拟打款；打款明细；门店应收统计 |
| 本期不做 | 真实微信支付接口；真实资金出款；自动获取运营账户余额；多个商户号；一个商户号多个转账场景；真实证书解析；真实回调 |
| 后续可能做 | 多商户号配置；多个微信转账场景；真实微信接口联调；自动查单；电子回单；真实导出；财务系统收款 |
| 明确排除 | 不把微信运营账户余额设计为实时可读；不允许未授权用户直接免确认打款；不允许业务场景绕过微信已开通转账场景限制 |

#### 5.3.1 当前业务场景与微信转账场景映射

| 系统业务场景 | 来源工具 / 来源模块 | 建议微信转账场景 | 用户收款感知 | 报备信息建议 | 备注 |

| 抽奖奖励 | 幸运转盘 | 现金营销 | 活动奖励 / 现金奖励 | 活动名称=幸运转盘活动名称；奖励说明=用户抽中奖励说明 | 当前主要奖励场景之一 |
| 抽奖奖励 | 零元抽奖 | 现金营销 | 活动奖励 / 现金奖励 | 活动名称=零元抽奖活动名称；奖励说明=用户抽中奖励说明 | 当前主要奖励场景之一 |
| 抽奖奖励 | 刮刮乐 | 现金营销 | 活动奖励 / 现金奖励 | 活动名称=刮刮乐活动名称；奖励说明=用户刮中奖励说明 | 当前主要奖励场景之一 |

说明：

1. 微信转账场景是微信商户平台开通的官方场景，本期一个商户号只配置一个。
2. 系统业务场景是业务侧统计和筛选口径，可在同一个微信转账场景下区分不同来源工具。
3. 若当前商户号开通的是「现金营销」，则优先开放抽奖奖励相关业务场景。
4. 若当前商户号开通的是「佣金报酬」，则优先开放分销合伙人提现中的「提现到微信零钱」业务场景。
5. 不允许将未匹配的业务场景强行绑定到不合规的微信转账场景。
6. 合伙人提现中的支付宝提现、空中分账等其他提现方式不受本方案影响，仍按对应原有方案处理。

### 5.7 权限规则

| 角色 | 可见内容 | 可操作内容 | 禁止内容 | 数据范围 |
| --- | --- | --- | --- | --- |
| 总部管理员 | 所有门店数据、总部配置、适用门店、授权用户、打款明细 | 配置微信参数、选择适用门店、生成授权码、打款、导出 | 无权限限制，除系统级安全策略 | 总部及全部分店 |
| 适用总部配置分店管理员 | 本店总览、本店明细、本店授权用户、总部下发配置只读 | 生成授权码、对本店授权用户打款 | 编辑微信密钥、修改总部配置、查看其他门店 | 本店 |
| 未适用总部配置分店管理员 | 本店配置、本店总览、本店授权用户、本店明细 | 编辑打款配置、生成授权码、打款、导出 | 查看总部和其他门店数据 | 本店 |
| 财务人员 | 总部汇总、门店应收、打款明细 | 查询、导出、标记线下收款状态 | 修改微信配置、发起打款 | 按组织权限 |
| 收款用户 | 授权页和授权结果 | 扫码授权 | 查看后台数据 | 本人授权链路 |

权限补充规则：

1. 被总部配置纳入「适用门店」的分店，配置页展示为只读，显示「配置来源：总部统一下发」。
2. 未被纳入「适用门店」的分店，配置页与总部一致，但数据范围仅本店。
3. 总部保存适用门店后，被选中分店可使用打款能力，实际出资账户为总部商户号。
4. 分店独立配置后，实际出资账户为分店自己的商户号，不进入总部垫付统计。
5. 若分店已存在独立配置，后续被总部加入适用门店，总部配置覆盖分店原配置；分店原配置进入「已被总部覆盖」状态，不再用于打款。

## 6. 按需模块

### 6.1 0-1 / 复杂交互需求

原型 / Demo 目标：

1. 可在页面内直接切换总部和分店视角。
2. 可展示总部、适用总部配置分店、独立配置分店三种不同体验。
3. 可演示总部在配置中设置适用门店，并覆盖门店原独立配置。
3. 可完整走通打款配置、授权列表、发起打款、查看数据总览。

提前验证流程：

1. 业务方确认入口位置和命名。
2. 财务确认总部统一出资、门店线下应收口径。
3. 运营确认操作指引和字段浮窗是否能指导商家完成微信后台操作。
4. 研发确认第一期数据模型是否支持后续真实接口扩展。

用户理解风险：

| 风险 | 表现 | 处理 |
| --- | --- | --- |
| 商家误以为安全 IP 要填写到系统 | 在配置页找不到输入框 | 在操作指引中明确写“到微信商户平台配置” |
| 商家误以为可以配置多个场景 | 想同时开奖励和佣金 | 页面提示本期仅支持一个场景 |
| 分店误以为自己出资 | 适用总部配置时不清楚资金来源 | 展示“出资账户：总部统一出资” |
| 分店不理解配置被覆盖 | 分店先独立配置，后续被总部纳入适用门店 | 在总部选择适用门店时提示覆盖；分店配置页显示覆盖来源和生效时间 |
| 总部误以为报表等于微信余额 | 想看实时余额 | 明确“不支持读取运营账户余额” |

业务反馈机制：

1. 原型评审收集总部、分店、财务三个角色反馈。
2. 记录是否需要第一期支持多个微信转账场景。
3. 记录是否需要真实导出、电子回单、线下收款状态。

不确定点：

| 不确定点 | 当前处理 | 后续确认 |
| --- | --- | --- |
| 微信商户平台是否允许同一商户号追加多个场景 | 本期按一个商户号一个场景处理 | 待商户后台实测 |
| 发起打款是否需要总部复核 | 原型先提供直接打款 | 待财务/风控确认 |

## 7. 测试与验收

### 7.1 测试覆盖点

主流程：

1. 从营销应用入口进入微信小额打款解决方案。
2. 总部完成打款配置并启用。
3. 总部在配置中设置适用门店。
4. 适用总部配置的分店查看总览和只读配置。
5. 独立分店完成自己的配置。
6. 生成授权码，生成二维码，模拟授权成功。
7. 对已授权用户发起打款。
8. 总部在数据总览查看门店汇总、出款账户拆分和线下应收。

分支流程：

1. 未配置只展示并进入打款配置。
2. 已配置进入数据总览。
3. 适用总部配置分店配置只读。
4. 独立配置分店可编辑配置。
5. 授权关闭后重新授权。
6. 打款失败后查看原因并重试。
7. 分店先独立配置，后续被总部加入适用门店，原配置被覆盖。

异常流程：

1. 必填配置缺失不可启用。
2. 安全 IP 不在系统表单中确认，配置异常通过校验失败或真实接口错误提示处理。
3. 未授权用户不可打款。
4. 一个商户号尝试配置多个场景时阻断。
5. 余额不可读时展示人工提示。

角色权限：

1. 总部可看全部门店。
2. 分店只看本店。
3. 适用总部配置分店不可编辑微信参数。
4. 独立配置分店不可查看总部和其他分店数据。

状态流转：

1. 能力状态：未开通 -> 待配置 -> 待验证 -> 待启用 -> 已启用。
2. 授权状态：待授权 -> 已授权 -> 已关闭/重新授权。
3. 打款状态：待处理 -> 打款中 -> 成功/失败。

数据口径：

1. 打款明细必须记录实际出款账户，区分总部账户和门店自有账户。
2. 适用总部配置且总部统一出资的门店，成功金额计入总部应收。
3. 门店自有账户出款不计入总部应收。
4. 同一门店如果既存在总部账户出款，又存在门店自有账户出款，统计表按「门店 + 出款账户」拆行展示。
5. 失败金额不计入成功打款和应收金额。

回归范围：

1. 营销应用入口分组展示。
2. 支付方案卡片排序。
3. 原有「支付宝人资打款」入口不受影响。
4. 左侧导航、顶部工具、表格筛选交互不破坏现有风格。

| 同门店双账户统计 | 门店先独立配置后被总部覆盖 | 同一门店存在历史门店自有账户打款和当前总部账户打款 | 数据总览按出款账户拆成两行 |

### 7.3 验收标准

| 验收项 | 标准 |
| --- | --- |
| 入口位置 | 「微信小额打款解决方案」位于「支付方案」分组，排在「支付宝人资打款」前 |
| 页面结构 | 未完成配置前仅展示「打款配置」；启用后展示「数据总览、授权列表、打款配置」三个页面 |
| 迁移目标 | 页面和流程明确表达本方案用于承接平台旧微信打款能力下架后的微信零钱打款链路 |
| 视角切换 | 可直接切换总部、适用总部配置分店、独立配置分店，页面数据和权限随之变化 |
| 打款配置 | 单页完成系统配置、适用门店选择和保存设置；外部操作通过飞书开通说明文档和字段浮窗说明 |
| 配置字段 | 包含 mchid、appid、API v3密钥、证书序列号、商户私钥、微信支付公钥ID |
| 安全IP | 不在打款配置页展示；通过「开通说明文档」说明商家需到微信商户后台配置 |
| 适用门店 | 总部配置页通过现有系统选择门店弹窗选择适用门店，支持门店标签筛选、多关键词搜索、复选列表和全部半选态；被选中的分店使用总部配置且不可编辑配置 |
| 覆盖规则 | 分店已有独立配置时，若被总部加入适用门店，总部配置覆盖分店原配置并有明确提示 |
| 授权列表 | 授权列表页支持查看授权列表和发起打款；分店视角在查询按钮后展示「生成授权码」，点击直接生成收款授权二维码；总部视角不展示该按钮 |
| 数据总览明细切换 | 数据总览支持在「打款统计 / 打款明细」之间切换，统计行和授权用户可跳转到打款明细 |
| 发起打款 | 仅已授权用户可打款，打款弹窗包含金额、原因、备注、报备信息 |
| 业务场景 | 打款明细可展示并筛选抽奖奖励、分销合伙人提现 |
| 打款明细字段 | 列表展示打款时间、单据号、门店、账户类型、出款账户、收款人姓名/手机号/openid、业务场景、金额、状态；操作列仅失败记录展示「重新打款」；不展示来源工具和来源业务单号 |
| 重新打款 | 失败状态的打款记录支持重新打款操作 |
| 出款账户 | 打款明细必须展示实际出款账户，区分总部账户和门店自有账户 |
| 打款处理规则 | 页面明确说明保存打款设置后按本方案承接微信零钱打款：抽奖类现金奖励发放到微信零钱；合伙人“提现到微信零钱”统一使用微信小额打款方案处理 |
| 总部总览 | 展示每个店每月打款金额；同一门店存在总部账户和门店自有账户时按两行展示 |
| 分店总览 | 只展示本店统计；本店明细在授权列表页查看 |
| 门店应收 | 适用总部配置的门店成功打款计入总部应收；门店自有账户打款不计入；同门店双账户按行区分 |
